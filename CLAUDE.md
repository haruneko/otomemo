# creative_manager — 開発ルール

実装は **SDD（スペック駆動）＋TDD** で進める。スペックが正準、コードは従う。

## スペックの所在（上ほど安定・権威）
- コンセプト＋要件: `docs/requirements.md`
- アーキテクチャ: `docs/architecture.md`
- 設計(SDD): `docs/design.md`
- タスク: Task機能（#1〜）
- テスト / コード

## 積みタスク（やりそびれ・保留）の在り処
- **`docs/backlog.md`** ＝「いつかやる／後回し」の置き場（undo/redo 版管理・死にコード撤去・systemd 自動起動・
  worker の claude_prompt 完全撤去・UX磨き 等）。着手したら Task 化して backlog からは消す。新しい保留もここへ。

## 研究・調査タスクの出力（ルール）
- **外部調査・理論研究・分析・実測の結果は必ず `docs/research/` 配下に `.md` で格納する**（会話に流して消さない）。
  例：`motif-extraction.md`・`melody-heuristics.md`・`self-check-log.md`。出典(URL)・実測値・設計含意を残す。
- **索引＝`docs/research/README.md`**（マスター目次・到達点サマリ＋全docのグループ別一覧）。**新規 research doc を作ったら README に1行追加**＝散逸防止。
- 外部研究は**出典付き**（Webなら引用URL）。**他者コーパス(POP909等)からは統計のみ抽出**＝リテラルな旋律/モチーフは保存しない（著作権）。自作ネタは可。

## レイヤーの正準順序
**コンセプト → アーキテクチャ → 要件 → 設計 → タスク → テスト → コード**
- 上ほど抽象・安定・権威。下ほど具体・流動。上位が下位の根拠。
- （要件↔アーキは入れ替え可。"何を"を先に見たい時は要件を上に置く）

## 修正時のルール（最重要）
既存を変えるときは、**上から下へ整合を確認してから**変える：
1. コンセプト/体験原則に反しないか
2. アーキテクチャに反しないか
3. 要件に反しないか
4. 設計(SDD)に反しないか
5. タスクに反映する
6. テストを先に書く（赤）
7. コードを変える（緑 → リファクタ）

鉄則：**変更は上位から下ろす。コードだけ変えて上位と乖離させない。** 乖離を見つけたら「上位を意識的に更新する」か「変更を捨てる」のどちらか。後追いでスペックを腐らせない。

## SDD
- 実装はスペックに従う。食い違ったら **スペックを先に直してからコード**。
- 設計変更は `design.md` に先に書く。

## TDD
- データ構造・契約（スキーマ / ジョブ / MIDI入出力 / トランスポーズ / 検索）は **テスト先行**。ここが命。
- 赤 → 緑 → リファクタ。

## strictness は段階的
- **序盤（探索・最初の縦スライス）**：緩くてよい。動かして学ぶ優先、テスト後追いも可。
- **コア・契約・修正フェーズ**：上の確認順序＋TDDを厳守。
- 実装は **縦スライス**（UI＋API＋ドメインを一緒に薄く）。横積みしない。

## 進め方
- 節目（一段落）ごとに残タスクを出す（忘れ防止）。勝手に「終わり」を宣言しない。
