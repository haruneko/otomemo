**この文書は「Chat（`claude -p` 薄いラッパー＋creative-manager MCP）で何ができる（べき）か」の発散→収束の成果**。作る系の使い方は `usecases-compose.md`（U1-U21）が持つ。ここは **調べる/解析/相談＋機材** の優先ユースケースを定義する。使いながら直す。

## 体験の芯（要件から不変）
機械は **候補まで・仕上げは人間**（#92/#151）。Chat は「何を・どんな雰囲気で」を汲む役＋Claudeの言語力。数値/理論の確実な部分は決定的な仕組みが担保し、**曖昧な所は候補として出し信頼度を添える**。

## 発散の地図（2026-07-05）
Chat 能力を 相談／調べる／作る／整える で洗い出し（[✓]既存 [＋]要拡張）。**優先4クラスタ**を収束（★＝オーナーの苦手/最優先に効く）。

- **相談**：方向性の壁打ち[✓]／詰まり打開＝選択肢[✓]★／情景・感情→作曲パラメータ翻訳[＋]／次の一手ナビ[＋]★

## ① アナリーゼ（音源解析）★★ 苦手直撃・feasibility済
- **目的**：好きな曲/流行曲を解析して、BPM・調・構成・音域・編成・コード進行・メロ特徴を**学ぶ**（オーナーはアナリーゼが苦手）。
- **トリガー**：Chat に曲を渡す。**入力＝ファイルアップロード主・URL補助**（yt-dlp は 2024以降 SABR/POトークンで難化＋ToSグレー＝一次はファイル、URLは必要になったら層で追加）。
- **Chatの動き**：分離(Demucs)→stem別に MIR（BPM/調/構成=allin1/音域=pyin/メロ輪郭=basic-pitch/三和音コード=BTC・Chordino）→**Claudeが数値と輪郭を「アナリーゼ文」に言語化**（メロ特徴の言語化はClaudeの独壇場）。1曲=数分CPU＝**非同期（投げて→裏で→トレイ）**。
- **出力**：**アナリーゼ知見ネタ**＝事実（BPM/調/構成/音域/編成=stem名/コード候補/メロ輪郭）＋Claudeの所見。**信頼度を分ける**（事実 vs 候補）。コード候補/メロ輪郭は otomemo のコード/メロネタに**"候補"で落として自分で弾き直せる**（学習の出口）。**音源は保存しない**（日本 著30-4「情報解析」＝派生的事実のみ残す）。
- **実現**：三和音/音域/調/BPM/構成は堅い。**7th拡張・混合音源の楽器名・多声ボーカル採譜は弱く候補止まり**。クラウド(Spotify API)は2024廃止で据えない＝**ローカルCPU自前**。詳細＝`research/audio-analysis-feasibility.md`。
- **要ビルド＝大**（新しい Python 音声。下記「実行基盤」参照）。

## ② 歌詞↔メロ相互変換 ★ メロ最優先×歌モノ
- **目的**：苦手なメロを「言葉から」引き出す／メロに歌詞の当て所を作る。
- **トリガー**：Chat で「この歌詞にメロ付けて」／「このメロに仮歌詞」。
- **Chatの動き**：歌詞→仮メロ（モーラに音を割当・日本語アクセント考慮・`gen_melody`＋歌詞制約）／メロ→仮歌詞（音数・リズムに合う母音/語感を Claude が）。複数案・ばらつき。
- **出力**：メロネタ（`Note.syllable` 付き）／歌詞ネタの候補。編集画面で弾き直し・流し込み。
- **実現**：既存資産（モーラ流し込み・`gen_melody`・Claude 言語）で成立。**要ビルド＝中**（歌詞制約付き生成の配線・相互変換のChat動線）。

## ③「次の一手」ナビ／詰まり打開
- **目的**：手が止まった時に前に進める。
- **トリガー**：「この曲、次どうする？」／「サビが決まらない」。
- **Chatの動き**：Claude が現状（構成・埋まり具合・欠け＝SongStatus/`analyze`）を読んで**具体的な次アクション＋選択肢**を出す。詰まりには複数方向を候補で。
- **出力**：次アクションの提案（＋そのまま作れるなら候補ネタ）。SongStatus の「段階/次の一手」を更新。
- **実現**：既存（SongStatus・analyze・streaming）＋**曲の状態を Claude に渡す文脈設計**。**要ビルド＝軽〜中**。

## ④ 機材相談＋知識ベース蓄積（新規・実需）
- **目的**：機材/プラグイン/音作り（ABILITY前提）の相談を、Claude知識＋**自分が貯めた知見**で。
- **トリガー**：「この音どう作る」「〇〇系のシンセ/プラグイン」「ABILITYで××するには」。
- **Chatの動き**：Claude が答え、**"ベースの知識を貯める"＝調べた機材知見を knowledge ネタとして保存**→後で引ける／Chat が参照（継続調査・知見化の骨格そのまま）。
- **出力**：機材知見ネタ（蓄積）。相談履歴。
- **スコープ注記**：otomemo の"作曲"からやや外縁（機材）へ踏み出す拡張。ただし **neta/知見化の器はそのまま機材にも効く**＝器を広げるだけ。意図的にやるなら良い拡張。深めるなら「機材」種別 or 知見の構造化を検討。
- **要ビルド＝軽**（Chat＋知見化は既存。まずは knowledge ネタの蓄積・参照で十分）。

## 実行基盤メモ（①アナリーゼ用・実装時に確定）
①は重い Python 音声ML（Demucs/basic-pitch/BTC）。今日 cm-worker を撤去したが**これは"脳"でなく決定的な音声道具**（gen_* を決定的TSにしたのと同じ思想の音声版）。素直な2択：
- **(A) api が Python音声CLIを spawn**：`demucs`/`chord-extractor` 等を子プロセスで叩き結果JSONを受ける。継続調査と同じ job consumer に `audio_analyze` intent を足す形。
- **(B) `cm-audio` 常駐サービス**（FastAPI・torch）新設＝cm-search と同型。api が HTTP で投げる（モデル常駐でウォーム）。
どちらも 1曲=数分CPU＝非同期＝**継続調査の「投げて→裏で→トレイ」骨格に乗る**。**stem は asset 化してキャッシュ**。
→ **決定：まず A**（新サービスを建てない・今日作った job consumer にそのまま乗る・「決定的な道具をCLIで叩く」＝gen_* を決定的TSにしたのと同じ思想。**B は"毎日多曲を回す"時代が来てモデルを温めたくなったら差し替え**＝後から可）。

## 決定事項（紙で決まるもの・ロック済 2026-07-05）
- **①実行基盤＝A**（api が Python音声CLIを spawn）。
- **④機材＝まず knowledge ネタで貯める**（専用種別は「メーカー/型番/用途で構造化して絞りたい」欲が出てから。器を先に増やさない）。
- **①入力＝ファイル主・URL補助**（yt-dlp は ToS/難化＝一次はファイル、URL は必要になったら層で追加）。
- **②歌詞制約の精度＝紙で決まらない＝②着手時に小さく実験**（gen_melody が音数/アクセントをどこまで守るかは試すだけ）。

## ①アナリーゼ 検証(POC)計画 ★ここを外すと無意味
**何を検証するのか**：「動くか/BPMが出るか」ではない（feasibility で既知＝出る）。**唯一の不確実性＝「既知の精度天井（三和音~85%/テンション~60%/多声ボーカル採譜~49%）のまま、Claude が統合した最終アナリーゼが、実際に“学びに使える”のか、誤りが多すぎて信用できないノイズか」**。これは**実曲を回して正解と突き合わせないと分からない**。堅い所（BPM/調/音域）だけ出す薄いPOCは何も検証しない＝禁止。

**方法：答えを知ってる曲で end-to-end ＋ 正解照合**
- 自分がコード/構成を把握している曲、または公開コード譜/アナリーゼ(U-FRET等)がある曲を **2-3曲**（バラード/疾走/打ち込み等ジャンル違い）。権利は**手元音源/CC**で確保。
- **フルパイプライン（分離→BPM/調/構成/音域/コード/メロ輪郭→Claude統合文）を通す**。難所（コード/メロ/統合）まで回す＝そこが不確実だから。
- 各項目を正解と突き合わせ「**使える誤り率か**」を判定（この照合が本体）：

| 項目 | 正解の取り方 | 見るもの |
|---|---|---|
| 構成 | 耳/公開構成 | セクション区切りが概ね合うか |
| **コード（核）** | 公開コード譜と照合 | **三和音の的中率／外れが"近い代理"で人が直せる範囲か／7th・テンションの信用度** |
| メロ | 実感・音域は既知と照合 | Claudeの特徴記述が実感に合うか（跳躍/順次/シンコペ/音域） |
| Claude統合文 | 主観 | **一般論でなく"その曲固有の学び"が出るか** |

**合格基準（先に決める＝POCの意味）**
- **GO（フル投資）**：三和音が体感8割当たり＆外れも近い代理で直せる／構成が概ね合う／**Claudeの文が"なるほど"を最低1つ生む**。
- **保留（②③④へ振る）**：コードが半分外れて信用できない／Claudeが当たり障りない一般論しか出さない／「自分で聴いた方が速い」と感じる。

**手順**：① 環境＝ffmpeg(apt)＋demucs/librosa/basic-pitch/BTC(or chord-extractor)（torch既存・数GB）② 手元/CC音源 2-3曲 ③ 分離→stem→各解析→JSON→Claude統合文（基盤A=CLI spawn の最小形）④ 出力を上表で正解照合＋合格基準で判定（**出力を一緒に見て決める**）。

**分かる/分からない**：分かる＝実精度天井でのUX（信用/ノイズ）・Claude統合の価値・A基盤の実装感。分からない＝多曲多ジャンルの安定性（代表2-3曲だけ）＝GO後に広げる。

軽い3つ（②③④）で Chat の"できること"を早く厚く／①は上の POC で GO/保留を判定してから本命投資。**この文書は定義まで＝コードは未着手**。
