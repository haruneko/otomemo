# 多エージェント・レビュー開発ループ（設計ドラフト）

スキル化の前提設計。**中心原則：実装者は自分の成果を「受け入れOK」と判定しない。** 受け入れは独立した別エージェントが、事前に定めた基準に対して合否を出す。回帰も別の素眼エージェントが見る。

今回の Otomemo（リポジトリ creative_manager）の自己監査（3巡で収束。各巡で前巡の"自分の回帰"を別エージェントが検出）で得た学びを反映する。

## 役割（分離が肝）
| 役割 | やること | 渡すもの | 渡さないもの |
|---|---|---|---|
| **Planner** | 変更をSPECに照らし定義。**受け入れ基準を先に書く**。タスク分解 | SPEC・現状 | — |
| **Implementer** | 計画通り実装＋テスト緑。**自己受け入れしない** | plan＋criteria | 受け入れ権限 |
| **Acceptor（独立）** | criteria の各項目を実コードで検証し**合否＋根拠**を返す | criteria＋diff／コード／テスト出力 | **実装者の「できた」主張**（コードと基準だけで判断） |
| **Regressor（独立・素眼）** | 変更が壊したもの/副作用を、できればdocsを見ずユーザー目線で探す | 触る領域の最小情報 | 変更の正当化文脈（確証バイアス回避） |

実装者＝判定者を**同一にしない**のが本質。判定者は基準と事実だけ見て、主張に引きずられない。

## フェーズと門（ゲート）
1. **計画** → 産物：plan＋**受け入れ基準（客観チェックリスト）**。基準は実装"前"に確定（後出し禁止＝採点表を動かさない）。
2. **実装** → 産物：diff＋テスト結果（事実のみ。完成主張はしない）。
3. **受け入れ判定**（Acceptor） → 各基準 pass/fail＋ファイル根拠。1つでもfail→実装へ差し戻し。実装者は覆せない。
4. **回帰チェック**（Regressor） → 新規バグ/退行を列挙（重大度付き）。
5. **収束ループ** → 受け入れ or 回帰で不合格なら **修正→Acceptor/Regressorを再実行**。**合格かつ無回帰になるまで繰り返す**（修正が新たな回帰を生むため＝今回の実証）。

## 洗練ポイント（"もう少し洗練"への回答）
- **基準は実装前に固定**：完成の定義を先に客観化し、Acceptorの主観・後付けを排す。
- **Acceptorに実装者の説明を渡さない**：コード＋基準＋テスト出力のみ。ナラティブで誤魔化せない。
- **Regressorは素眼**：変更内容を最小限しか教えず、ユーザー/別実装者の目で見る（docs非参照モードが有効だった）。
- **再実行を必須化**：fix→再チェックを収束まで。自己回帰（実装者の盲点）はこの再実行でしか捕まらない（round2/3の学び）。
- **read-only＋事実**：判定系エージェントは変更禁止、出力は合否＋根拠（ファイル/行）。指摘の重複は呼び側で排除。
- **コスト管理**：判定系は並列・バックグラウンド。基準が小さい変更はAcceptor 1体＋軽い回帰、広い変更は素眼回帰を厚く。

## スキル化（叩き台）
`/review-loop <変更の意図>` で：
1. Planner が criteria を生成（ユーザー確認可）。
2. 実装（人/実装エージェント）。
3. Acceptor＋Regressor をサブエージェントで spawn（criteria＋diff を渡す）。
4. 合否を集約 → fail があれば差し戻し、収束まで 3→4 を回す。
5. 全合格＋無回帰で「受け入れ完了」を宣言（実装者ではなく判定の集約として）。

## 決定（ユーザー確認済み）
- **Acceptor はテスト実行結果を信じてよい。ただし"網羅性"を監査する**：実行されたテストが受け入れ基準を網羅しているかを点検し、足りなければ Acceptor が不足分を自分で実施（実行/確認）する。
- **回帰チェックは 2 巡**（まずは）。
- **ゲートは重大度しきい値**（fail ゼロ必須ではなく、重大度で通す/止める）。
- **実装もサブエージェント化を試みる**。難しければメイン（人/メイン）で実装し、判定だけ自動化。

## 受け入れの層（今回の実走で判明＝重要）
UIは「コードが正しい」と「実機で正しく見える」が別レイヤー。実際 stage① で、コードAcceptorは2巡ACCEPTしたのにスマホ実機で横スクロール／2列カード／ヘッダ崩れが残った（原因の一つは `@media` のソース順バグ＝jsdomでは検出不能）。よって受け入れを3層にする：
1. **コードAcceptor（サブエージェント・別コンテキスト）**：spec/コード/テストで判定。網羅性も監査。**ただし実レンダリング/レスポンシブ/横スクロールは見えない（jsdom）。**
2. **ヘッドレス実機近似（Playwright）**：**Acceptor（受け入れエージェント）がこれを叩く**＝`pnpm --filter @cm/web test:e2e` を実行し、結果＋スクショ（`apps/web/e2e/__screenshots__/`）を確認して受け入れ判定に含める（日本語表示は `scripts/setup-fonts.sh` で NotoSansCJK 導入済み前提）。複数viewport（スマホ~390px / タブレット / PC）でアプリを起動し、機械判定：
   - **横スクロール無し**：`document.documentElement.scrollWidth <= window.innerWidth`（各viewportで）。
   - 主要フロー：カードtap→メインペーンに編集が出る／← 戻るで一覧へ／＋曲を組む／フィルタ。
   - **スクショ**を成果物に残す（人のレビュー用）。
   - これがコードAcceptorと人の間の層。今回の往復はこれがあれば消えた。
3. **人（実機）＝最終受け入れ**：手触り・見た目。これは外せない（実機固有のブラウザUI等もある）。

## 次アクション
1. **Playwright 実機近似チェックを足す**（上の層2）＝最優先（以後のUI段はこれで守る）。
2. `/review-loop` を実装：Planner(criteria) → Implementer(sub-agent優先) → Acceptor(網羅監査) → **Playwright実機近似** → Regressor×2 → 重大度ゲート → 人の最終受け入れ → 収束まで再実行。
