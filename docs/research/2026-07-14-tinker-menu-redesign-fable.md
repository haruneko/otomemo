## 0. 設計の芯（1段落）

**煩雑さの正体は「器」ではなく「1つの器に全パーツの設定が同居し、パーツが増えるたび同じ縦列に割り込む」こと。**
2026-07-10の器交換（ボトムシート化）は正しかったが、器の中の情報設計が「フラット縦一列」のままだったので、今日のWP群（ドラム/ベース型セレクタ4本=42選択肢）が生成ボタンの列に割り込んで壊れた（inventory §2.3・スクショ02-03）。
本設計は器の中を**二軸**で再構成する：**横軸＝パーツ**（メロ/ベース/ドラム/…＝これからも増える唯一の軸）、**奥行き＝深さ**（おまかせ→型/プリセット→ノブ）。
UIの形は**「棚（ハブ）＋引き出し（パーツ別サブシート）」**：いじるを開くと**スクロール不要の1画面のハブ**＝パーツのタイル棚。タイルをタップ＝そのパーツをおまかせ生成（現状と同じ2タップを死守）。タイル下端の状態チップ（「おまかせ」等）をタップ＝そのパーツの引き出しが開き、型・プリセット・ノブは**全部そこに沈む**。パーツが増えてもタイルが1枚増えるだけ＝ハブは壊れない。

## 2. 情報アーキテクチャ＝二軸原則と「ハブ契約」

### 2.1 二軸原則

- **パーツ軸（増える軸）**：メロ・ベース・ドラム・コード・骨格・リフ・管弦（今7）→対旋律・コード楽器・SE…（今後）。**新パーツ=新タイル1枚**。それ以外の場所には何も足さない。
- **深さ軸（こだわりの軸）**：`おまかせ（タイルtap即生成）→ 型/プリセット（引き出し前面・chip）→ ノブ（引き出しの「細かく」・群アコーディオン）`。オーナー思想「機械は候補まで・既定で良い音・こだわる人だけ潜る」の直訳。全ノブ既定OFF＝bit一致は**状態も送信も現行と完全同一**（UIの住所だけ変える）。

### 2.2 ハブ契約（固定サイズの不変条件）

ハブ＝いじるを開いた最初の1画面。**内容を固定契約にする**：

```
┌ いじる ────────────── ✕ ┐
│ 進行の色：[おまかせ][明るめ][土][哀愁][浮遊] │ ← 横断設定（旋法）＝chip 1行
│ ✨ おまかせで一式 ▸ ドラム→ベース→メロ…    │ ← ヒーロー1本（§4）
│ ┌メロ──┐┌ベース─┐┌ドラム─┐               │
│ │ tap=生成 ││        ││        │               │ ← パーツタイル棚（3列）
│ │[歌もの▾]││[おまかせ▾]││[ロック▾]│           │ ← 状態チップ＝引き出しの扉
│ └──────┘└──────┘└──────┘               │
│ ┌コード─┐┌骨格──┐┌リフ──┐┌管弦─┐…      │
│ 書き出し：[MIDI] [MIDI（分割）]              │ ← 固定フッター行
└──────────────────────────┘
```

- **不変条件**：ハブは 390×844（シート可視691px）で**スクロール0**。実測見積り＝ヘッダ64＋旋法44＋一式56＋タイル3列×3段252＋書き出し48＋余白≈500px。タイル12枚（4段）まで無スクロールで吸収＝**パーツが今の7→12に増えても契約は破れない**。13枚目からはタイル棚のみ縦スクロール（他の行は sticky）＝それでも壊れるのは棚だけ。
- **ハブに足してよいもの**：新パーツのタイルのみ。横断設定は旋法＋一式の2枠で打ち止め（3つ目が要る日は「共通▾」引き出しを新設して沈める）。**この契約をコードコメントに明記**するのが再発防止の本体（今日の4セレクタ割り込みは「どこに足すべきか」の規約が無かったのが原因）。
- タイルの見た目＝**ホーム画面の「＋メロ/＋骨格/…」3列タイル（スクショ00・`create-tiles .ct-parts` 3列化 `assets-bass.css:659`）と同じデザイン言語**＝新規学習ゼロ。色もレーン色（`--k-melody`等）を流用。
- song（編成）では現行どおり書き出し行だけ残る（`isSong` ガード温存）。

### 2.3 タイルの2ゾーン

- **上ゾーン（大・tap）＝生成**：現在の設定（既定=全部おまかせ）で即 `genPart`。シートを閉じ候補トレイへ＝**現行「いじる→メロ」の2タップ主動線を1タップも増やさない**。
- **下ゾーン（状態チップ・tap）＝引き出しを開く**：チップ文言＝そのパーツの設定サマリ（「おまかせ」「ロック・溜め4小節」「歌もの＋2ノブ」）。**設定の可視化と設定への扉を1つのUIで兼ねる**。設定を持たないパーツ（リフ/管弦/コード）はチップ無し＝タイルが軽いまま（引き出しは要るようになった日に生やす）。

## 4. 「おまかせで一式」（ハブのヒーロー）

- **何**：1タップで `ドラム→ベース→メロ（コード無ければ先頭にコード）` の候補を順に生成し、候補トレイに**パーツ別グループで積む**。置くかどうかは全部人間＝**「機械は候補まで」の範囲内**（自動配置はしない）。
- **なぜハブに**：オーナーの実フロー（track-wiring シリーズ＝ドラム→ベース→メロの結線）が既にパイプラインなのに、UIは1パーツずつ「いじる→ボタン」を往復させている。往復の除去はタップ数を半減させる（§6）。
- **実装含意**：`genPart` の直列呼び（既存APIのみ・新規APIゼロ）。候補トレイは現在「別kindが来たら置換」（`useMelodyGen.tsx:404-405 pushCand`）＝**kind別グループ保持への小改修が要る**（§7 T5）。ここだけが本設計唯一の挙動変更点。順序は固定でよい（設定は各引き出しの現在値を使う＝一式にも型が効く）。

## 5. 「増える」への耐性（ルールとして明文化）

1. **新パーツが増えた** → ハブにタイル+1（3列グリッドが折り返す）。設定が要るなら引き出しを1枚生やす。**他のどこも変えない**。例：対旋律が骨格ブロック経由からタイルへ昇格する日も、タイル+1で終わる。
2. **既存パーツのノブ/型が増えた** → そのパーツの引き出しの該当群へ。前面に出せるのは「chip6±1・seg1行」まで、超える分は細かくへ。**ハブには1pxも影響しない**。今日のWP-D1/B1がこの規約下なら、ドラム/ベース引き出しの中で完結していた。
3. **横断設定が増えた** → 原則禁止（旋法＋一式で2枠打ち止め）。3つ目は「共通」引き出し新設。
4. **50コントロール時代の姿**：ハブ=12のまま。メロ引き出し=前面8＋沈み20でも群アコーディオンで1画面。ドラム=前面8＋沈み15。**壁はどこにも再形成されない**＝縦スクロール総量はどの画面も1画面±α。

## 9. 残課題・オープン問題

- ジャンルchipの文言（「土っぽい」等の耳語ぞろえ）と、型直指定を「もっと…」で開く際の一覧デザイン＝【要実機・耳】オーナー較正。
- 一式の生成順・対象（リフ/管弦を含めるか）＝設定不要の固定から始め、実使用で判断。
- 候補トレイのkind別グループ化（T5）＝トレイ自体の再設計は本doのスコープ外（現トレイの上にグループ見出しを足す最小案まで）。
- PC幅（>640px）のハブ形＝ドロップダウン据え置きで開始、モバイルで検証後に統合。
- マイ設定の同期（localStorage→ネタ化するか）＝backlog 行き。
