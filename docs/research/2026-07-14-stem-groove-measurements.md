パイプライン（すべて既存資産の流用・**リポジトリ/DB 変更なし**）:
1. **分離**: `_audio_poc/.venv` の demucs 4.0.1（htdemucs・4stem）。⚠️ demucs CLI は保存時に `torchaudio.save`→torchcodec 依存で **ImportError**（torch 2.12+cpu 環境の既知穴）。回避＝demucs Python API で分離し **soundfile で保存**する薄いスクリプト（scratchpad）。既存 `analyze.py` は自前保存なので本番経路は無事のはず（未確認なら別途）。
2. **ドラム onset**: 本番 `_audio_poc/analyze.py` の `drum_onsets()` をそのまま import（帯域 kick 25–110 / snare 250–1500 / hihat 6k+・クロス帯域優勢ゲート・crash=高域長減衰）＝本番較正と bit 一致。
3. **拍子/downbeat**: 本番 `apps/api/src/audio-drums.ts` の `extractDrumPattern` を tsx で実行。
4. **BPM 精緻化（重要な学び）**: `librosa.beat_track` の BPM（0.1 刻み）は 5 分曲では**累積 0.5 拍以上ずれる**＝剛体グリッドが崩壊（kick の16分格子残差中央値 46ms ≒ 一様分布）。DTM=固定テンポ前提で **kick+snare onset の格子残差を最小化する ±1% スイープ**で精緻化 → 87.2→**87.001**・125.4→**125.014**、残差中央値 **8.8 / 9.5ms** に収束。自作曲は本当に整数 BPM だった＝固定テンポ仮定は正しい。
5. **ベース音高**: pyin（30–400Hz）→ RLE ノート化。⚠️ **エネルギーゲート必須**：ベースが休む区間（イントロ/アウトロ）は stem に残渣（上物ブリード）だけが残り、pyin が C4–G4 帯の幽霊ノートを量産する（LM で 548 中 217 ノートが残渣）。stem RMS の p90×10% 未満のノートを棄却 → G3 以上の幽霊は全滅。
6. **メロ onset**: vocals stem を pyin（65–1300Hz）→ RLE ノート開始時刻。

## 1. B1: kick×ベース onset 共有率（kickLock 較正）

許容窓を 30/50/70ms でスイープ（chance＝一様配置での期待共有率）:

| 指標（50ms窓） | LostMemory | DeepSea | chance | lift |
|---|---|---|---|---|
| **P(kick あり\|bass onset)** | **0.549** | **0.821** | 0.22 / 0.29 | 2.5 / 2.8 |
| **P(bass あり\|kick onset)** | **0.582** | **0.331** | 0.23 / 0.12 | 2.5 / 2.8 |
| P(bass\|snare)（参考） | 0.575 | 0.313 | – | – |

- 30ms→50ms でほぼ飽和（LM 0.452→0.582→0.598）＝同時打点の実体は **±30〜50ms** に収まる。グリッド生成なら「同一16分step」で数えてよい。
- **非対称が本体**：ベース側から見ると kick の上に乗る率が高い（0.55〜0.82）が、kick 側から見るとベースがいない kick が半分ある（P(bass|kick) 0.33〜0.58）。＝理論通り「**キックの onset 集合を骨格として共有しつつ、ベースは間引く/差分を作る**」。完全ユニゾン（share≈1.0）は自作曲に存在しない。
- 曲差＝ベース密度：LM は 8 分駆動（bass onset 636）で相互 share 均衡、DS は 6/8 で音価長め（onset 285）＝置く時はほぼ kick 上（0.82）だが疎。
- スネアとの share は kick と同水準＝「2・4 をベースが避ける」挙動は自作曲には**無い**（8分駆動が2・4を素通りするため）。snareGap（音価ゲート）は「置かない」でなく「切る」設計が正しい、の傍証。

**→ kickLock 既定の提案**：既定 0（bit一致）は不変。ノブを開けた時の**目標動作点＝「ベース onset の 55〜80% が kick と同 step」**（弱=0.6・強=0.8 のプリセット相場）。1.0 に張り付く実装（全onsetをkickへスナップ）は実曲から逸脱＝上限を 0.85 程度にクランプするのが安全。

## 2. B1: ベース実音域窓（duration 重み付き・残渣ゲート後）

| 統計 | LostMemory | DeepSea |
|---|---|---|
| min / max | E1 / F3 | C1 / F3 |
| **p5 / median / p95** | **F1 / G1 / A2** | **G1 / C#2 / A2** |
| p5–p95 幅 | 1.33 oct | 1.17 oct |
| オクターブ別 duration | oct1 85% / oct2 10% / oct3 5% | oct1 40% / oct2 58% / oct3 3% |

- 実測の実効窓 ≒ **G1..A2（MIDI 31..45）**、中心は G1〜C#2。**現行生成器の `36+pc`（C2..B2＝36..47）は実曲より約5半音高い**。
- 幅は約 1.2〜1.3 オクターブ＝「1オクターブ張り付き＋上下に半オクターブの逃げ」が実像。オクターブ跳躍の語彙を入れるなら跳び先クランプは E1..G3 でなく **G1..C3 程度が本人の作風**。

**→ 音域窓の提案**：既定窓を `31..45`（G1..A2）へ下げる、または少なくとも root 配置の基準を C2→**A1 付近**へ。bass-generation-upgrade の「33..55（A1..G3）へ拡張」案は上限が本人の実曲（実効上限 A2・絶対上限 F3）より広い＝**33..48（A1..C3）が実測適合**。synth 側の鳴り（現行 C2 基準）の耳確認は必須（backlog の較正と同時に）。

## 3. B1: アプローチノート頻度（小節最終拍→次小節頭）

定義：小節の最終拍に onset があり、次小節頭（先頭 0.5 拍以内）にも onset がある遷移について、音程差を分類。

| | LostMemory | DeepSea |
|---|---|---|
| 対象遷移数 | 35 | 24 |
| 同一 pitch class | 0 | 7 (29%) |
| **±2半音接近** | **22 (63%)** | **6 (25%)** |
| 　うち半音 / 全音 | 0 / 22 | 3 / 3 |
| 4度/5度 | 6 (17%) | 1 |

- **接近音は実在し、しかも主役は全音（ダイアトミック隣接）**。クロマチック半音接近は DS に少数あるのみ＝理論 doc の「半音接近は弱拍限定で耳が許す」以前に、本人の作風はほぼスケール内接近。
- LM（8分駆動）は「チェンジ際に動く」が 6 割超え＝アプローチはこの作風の**常用語彙**。DS（音価長め）はルート維持が3割で接近は控えめ。

**→ approach ノブの提案**：既定 0 は不変。プリセット相場＝**0.3（バラード系）〜0.6（8分駆動系）**。接近音の内訳は**全音:半音 = 3:1** 程度でスケール音優先（現理論案の chromatic 優先と逆）＝ scale-degree 隣接音を第一候補、半音は次点。

## 4. D2: hihat 重み 0.3 の実測較正

`onsetDensity[bar] = |kick| + |snare| + 0.3*|hihat|` の 0.3 の妥当性:

| 指標 | LostMemory | DeepSea |
|---|---|---|
| onset/小節: kick / snare / hihat | 6.4 / 6.3 / **9.2** | 8.3 / 7.9 / **9.9** |
| 小節別 count の CV: kick / snare / hihat | .32 / .38 / .41 | .28 / .27 / .27 |
| **corr(hihat, kick+snare)（小節別）** | **0.20** | **−0.02** |
| onset 帯域ピーク比 hihat/(kick,snare) | 1.17 | 1.19 |
| 帯域 RMS 比 hihat/(kick,snare) | 1.96 | 1.70 |

- hihat は個数で kick+snare の 6〜7 割を占め、エネルギーでも劣らない（比 1.2〜2.0）＝「小さいから軽い」は成り立たない。
- しかし**「busy-ness の信号」としてはほぼ無情報**：小節別 hihat 数と kick+snare 数の相関が 0.20 / −0.02 ＝ hihat は独立に刻み続け、グルーヴの詰み具合を反映しない（理論 doc の「ハット16分刻みは"密"でも体感は薄い」を相関ゼロという形で実証）。
- 重み 0.3 だと密度指標の約 18〜20% を hihat が占める。相関ゼロの項が2割はやや過大。

**→ 提案：0.3 は上限として妥当、**「密度＝メロ相補の駆動信号」用途なら **0.2 へ下げる**のがデータ寄り（kick+snare だけでも大勢は変わらない）。0.3→0.2 の差は小さいので、変更コストを掛ける優先度は低＝**現行 0.3 のままでも実測と矛盾しない**、が結論。

## 5. D2: メロ（vocal）×ドラム onset 共有率

vocal ノート開始 × 各ドラム（±50ms、lift＝chance 比）:

| | LostMemory | DeepSea |
|---|---|---|
| kick | 0.314（lift 1.43） | 0.453（lift 1.55） |
| **snare** | **0.359（lift 1.67）** | **0.485（lift 1.74）** |
| hihat | 0.439（lift 1.41） | 0.492（lift 1.42） |
| いずれか | 0.589（chance 0.75=飽和） | 0.631（chance 0.92=飽和） |

- メロ onset はドラムとチャンス以上に同期する（lift 1.4〜1.7）が、**7〜5割は外れている**＝理論 doc A の「全onsetをキックに揃えると平板」の実測面。drumLock 系ノブの目標動作点も「share 0.3〜0.5」程度が本人の実像。
- **最も同期するのは snare**（lift 1.7）＝バックビート位置にメロの音が居る。理論 doc B の「2・4 は velocity アクセントで呼応」の入力として整合的。
- 注：DB の project メロ MIDI との突合ではなく **同一曲の vocal stem** から取った（曲とメロの整合が保証されるため上位互換）。pyin RLE の断片化で share は下振れし得る（保守的な値）。

## 6. D1検証: クラッシュ間隔（フィル/セクション境界頻度）

crash onset（高域長減衰・1.5s 併合）→ さらに「2小節以内の連鎖」をクラスタ化（サビ内で毎小節クラッシュを刻む区間を1つの塊と数える）:

| | LostMemory | DeepSea |
|---|---|---|
| crash 数（併合後） | 8 | 21 |
| クラスタ数 | 7 | 7 |
| **クラスタ間隔（小節）** | 14, 11.5, 16.5, 4, 3.5, 38 → **median 12.7** | 6, 21, 9.3, 6, 9.9, 9.2 → **median 9.3** |
| 総小節数 | 99 | 84 |

- **D1 理論の相場「8〜16小節に1回」と一致**（median 12.7 / 9.3）。中間の 3.5〜6 小節の軽い節目も理論の「中間点 turnaround（4小節に1回まで）」の範囲。
- 曲による crash の使い方の差：LM は単発マーカー（境界にだけ置く）、DS はサビ内で 1 小節おきに刻む「テクスチャ」＋境界。**crash 間隔の生値はフィル頻度の代理にならない曲がある**＝クラスタ化してから数えるのが正しい（生 median は DS で 1.3 小節に化ける）。
- 生成既定への含意：**大フィル＝8〜16小節に1回・セクション境界のみ**を既定、中間 turnaround は 4 小節粒度で確率ノブ、が実測とも理論とも整合。

## 7. 制約・正直な注記

- **n=2**（LostMemory=4/4 87BPM ポップロック・DeepSea=6/8 バラード）。これは「一般統計」ではなく**本人の作風の点推定**＝既定値較正（本人が使う道具）には むしろ適するが、ジャンル外挿は不可。
- demucs 分離のブリード：bass stem の無音区間残渣は RMS ゲート（p90×10%）で除去したが、ゲート閾値依存の統計（音域の min/max 端）は ±数半音の誤差を見込む。p5–p95 は頑健。
- DeepSea の 6/8 は forceMeter＝ドラム型照合の自信度は低い（0.023）。downbeat/小節割りの誤りがアプローチ頻度（§3）と小節別統計（§4）に乗り得る。ただし格子残差 9.5ms・crash クラスタが小節整数倍に乗る事から、グリッド自体は概ね正しい。
- crash 検出は open hihat と混同し得る（DS のクラスタ内連打の一部はオープンハット疑い）。クラスタ化で頻度統計への影響は抑制済み。
- vocal/bass の pyin ノートは RLE 断片化・オクターブ誤りの残留があり得る＝共有率は下振れ方向（実際の同期はこれ以上）。
- 環境事実：`_audio_poc/.venv`（demucs 4.0.1 / librosa 0.11 / torch 2.12.1+cpu）。**demucs CLI の保存は torchcodec 不在で落ちる**＝API 経由＋soundfile 保存で回避（venv への追加インストール無し）。分析スクリプトは scratchpad のみ・リポジトリ/DB 無変更。
