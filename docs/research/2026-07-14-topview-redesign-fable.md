## 0. 設計の芯（1段落）

**情報過多の正体は「道具がコンテンツより先に・大面積で・二重に出る」こと。** 到達タップは主要タスク全部1〜2で浅い（inventory §2.3）＝深さは問題でなく、最初のネタに触れる前に47標的・可視9割のクロームが視界を占める**幅**が問題。しかも作成タイル14と種別フィルタ13は**同じKindIcon群の二重露出**で、kindが増えるたび両方に恒久1枠ずつ刺さる（今日のWP-X3で6→9/10→13）——いじるメニューと同型の病理。
本設計は主役を逆転する：**ファーストビューの主役＝実データ（つづき＋自分のネタ一覧）**。「作る」は**1ボタン→作成の棚**（姉妹のハブと同じ3列タイル・kindが増えても棚に+1するだけ）へ、「絞る」は**実在するkindだけのラベル付きチップ（件数順・≤6）＋引き出し**へ畳む。露出は登録レジストリでなく**実データ量から導出**する＝0件のリフ/管弦はトップに現れず、使い込んだ種別が前に出る。トップは**契約**（§2.2）で固定し、「増えるたびトップに割り込む」構造を断つ。

## 2. 情報アーキテクチャ＝主役の逆転と「トップ契約」

### 2.1 新しい序列（モバイル390px・上→下）

```
┌ Otomemo ──────────── 🎛 📥 ⚙ ┐ ← ヘッダ（現状のまま・4標的）
│ [すべて192][未仕分け179][研究13][＋]|[ライブラリ] │ ← 器チップ（現状のまま・非肥大）
│ [＋作る▾] [ 検索…            ] [絞る▾]        │ ← アクション行＝作る/探す/絞るの扉を1行に
│ [コード進行47][メロ42][セクション24][ベース19]…  │ ← 種別チップ＝実在kindだけ・件数順・≤6
│ つづきから ▸ 検収-Aメロ-20260714（きのう）▶    │ ← 作業の続き＝最終更新1件のピン
│ [▦カード|☰リスト]                 [既定順▾]    │ ← 一覧ヘッダ（既定=リスト提案）
│ ▍♪ JRock サンプル メロ         メロディ ▶     │
│ ▍≡ 検収-Aメロ コード(dorian)   コード進行 ▶   │ ← ★主役＝ネタ一覧（フォールドに8行前後）
│ ▍… （以下リスト。カード密度も選べる）          │
└──────────────────────［💬FAB］┘
```

- **アクション行**＝「作る」「探す」「絞る」の3つの扉を52pxの1行に統合。検索は現状どおり常時入力欄（1タップ死守）。
- **種別チップ**＝現スコープの実在kindを件数降順に最大6個、ラベル＋件数つき（無ラベルアイコンの当て推量を解消）。7位以下と0件は「絞る▾」の引き出しへ。現DBだと上位6＝コード進行47/メロ42/セクション24/ベース19/リズム18/コード楽器16＝**実務の主力6種がそのまま前に出る**（inventory §5）。
- **つづきから**＝現スコープで最終更新のネタ1件を一覧の上にピン（既定順=手動並びを壊さずに「作業の続き」へ1タップ）。
- 検索劣化バナー・E2Eジャンク対策は現状のまま/対象外（ジャンクは近く掃除される前提）。

### 2.2 トップ契約（固定サイズの不変条件）

姉妹設計のハブ契約と同文形式。**この契約をコードコメント（App.tsx の新構造の頭）に明記するのが再発防止の本体**：

1. ファーストビューの固定行は**6つだけ**：ヘッダ／器チップ／アクション行／種別チップ（≤1行）／つづき（≤1行）／一覧ヘッダ。残り全部が一覧＝実データ。
2. **新しいkindが増えた** → 作成の棚にタイル+1のみ。トップの種別チップには**そのkindのネタを実際に作った時だけ**、件数順の競争で現れる。**トップのDOMには1pxも足さない**。
3. **アクション行に足してよいものは無し**（作る/検索/絞るの3枠で打ち止め）。4つ目が要る日は「絞る」引き出しへ沈めるか、契約を意識的に更新する。
4. 種別チップは**最大6**・つづきは**最大1行**。溢れは引き出し/一覧へ。
5. 例外＝検索劣化バナー等の一時警告（状態依存・恒久でない）は行数に数えない。

### 2.3 「主役＝実データ」推定の検証（込み）

オーナー評は「情報過多」であって「実データを主役に」とは言っていない。推定の根拠と、外した時の逃げ道を設計に含める：

- **根拠1（実測）**：到達タップは全タスク1〜2で問題なし＝「作成1タップ」の維持は過多の主因でない。一方、実データはフォールド外＝「既存ネタを開く」だけが実質2タップ＋当て推量（inventory §2.3）。**浅くする余地が残っているのはコンテンツ側だけ**。
- **根拠2（思想）**：「機械は候補まで・仕上げは人間」＝人間が仕上げる対象は既にある自分のネタ。トップ＝仕上げ対象への玄関、が思想と整合。
- **根拠3（分布）**：project実データは主力6種に集中し日々更新される（inventory §5）＝「続き」動線に実需がある。
- **検証手段**：①本モックを姉妹モックと並べ、お題A（続きを開く）/B（新規作成＝**2タップ化のトレードを隠さず体感**）/C（絞る）で実測比較→オーナー判断。②実装後S2の段階で、＋作る棚の開回数 vs 一覧/つづきのオープン回数を localStorage カウント（送信なし・1週間）で確認。
- **外した時のフォールバック**：作成が想定より高頻度なら、アクション行の「＋作る」の右に**頻度上位2種のクイック作成チップ**（例：＋メロ／＋コード）を足す＝1タップ復活。枠は2個固定＝契約は壊れない。

## 3. 2枚の壁の畳み方＝比較と推し

### 3.1 作成タイル14の行き先

| 案 | 形 | 増える耐性 | 作成タップ | 二重露出 | 判定 |
|---|---|---|---|---|---|
| A 頻度順＋「もっと▾」 | 上位6タイル常時＋折り畳み | △ 頻度変動でタイルが並び替わり位置学習が壊れる。壁が半分残る | 1 | 残る | 却下 |
| **B 作成の棚** | ＋作る1ボタン→ボトムシート（3列タイル・現行と同じ絵/順/グループ） | **◎ 新kind=棚に+1のみ**。棚は姉妹ハブと同じ言語＝新規学習ゼロ | **2** | 解消 | **採用** |
| C 検索合流（コマンドパレット） | 検索欄に「メロ」→「＋メロを作る」行 | ◎ | 2（入力含む） | 解消 | 単体では発見可能性が弱い→**Bの補助（B-lite）として採用** |
| D 作成FAB | ＋FABを左下に常設→棚 | ◎ | 2 | 解消 | FAB2個は親指圏の奪い合い。Bの扉の置き場違いに過ぎない→アクション行に一本化 |

- 棚の中身＝**今のタイルをそのまま移設**（パーツ9／組み立て・文字4・グループと順序も現行どおり＝位置学習を保つ。頻度順に並べ替えない）。**取込（MIDI/楽譜/ハミング/音源/URL/歌詞）も棚の下段へ**＝「新しく増やす」系を1つの棚に集約。
- 発見可能性：新kindは棚に入れば必ず見える（棚は「作る」の全カタログ）。さらに新着タイルに小さな「new」ドットを初回のみ付す（姉妹の「初回チップ明滅」と同じ手当）。
- 2タップ化のトレードの根拠＝頻度：作成は192件/数ヶ月、開く/続きは毎日複数回。低頻度の1タップを高頻度の視界と交換する。§2.3のフォールバックで保険。

### 3.2 種別フィルタ13の行き先

| 案 | 形 | 増える耐性 | 絞りタップ | 可読性 | 判定 |
|---|---|---|---|---|---|
| A 頻度順に静的並べ替え＋もっと▾ | レジストリ順を件数順に | ✕ レジストリ由来のまま＝0件も居座る | 1 | アイコンのみのまま | 却下 |
| **B データ導出チップ** | **実在kindのみ・件数降順・≤6・ラベル+件数**＋「▾」で全種別引き出し | **◎ 露出∝実データ**。kind追加も0件掃除も自動追従（ジャンク掃除後は勝手に痩せる） | **1**（主力6種）/2（残り） | ラベル+件数で当て推量解消 | **採用** |
| C 全部を引き出しへ | トップは検索のみ | ◎ | 2 | — | 種別絞りは頻用（ジャンク下では必須動線だった）＝1タップを守る。却下 |
| D 種別タブ | 横タブ | ✕ 390pxで4枚が限度 | 1 | ○ | 却下（姉妹§3と同じ理由） |

- チップの導出＝**現スコープの未絞り込み一覧のkind件数**（クライアント集計・追加APIなし。§7 S3）。絞り込み中もチップ集合は直前のスナップショットで安定（チップが消える・並び替わるチラつきを防ぐ規約）。
- 「▾」引き出しの中＝全filterable種別（件数つき・件数順）＋**「まだ0件」ゾーン**（リフ/管弦…をグレーで明示＝「作ればチップに出る」の学習）＋mood入力。選択中はチップにリング（現行の`.on`流儀）。

### 3.3 小物の整理

- **mood入力**→絞る引き出しへ（英語ラベル・値の当てがつかない・頻度低）。
- **密度/並び**→一覧ヘッダに現状のまま残す（02-doc最有力の密度切替は主役の道具）。**既定をリスト密度へ**変更提案（localStorage未設定時のみ。カード派ならワンタップで戻る）＝一覧性の底上げ。オーナー耳/手確認つき（S4）。
- **取込トグル**→棚の中へ（§3.1）。**検索**→現状のまま常時1等地。**器チップ**→現状のまま（非肥大・inventory §0）。

## 4. 姉妹設計との設計言語の統一点

| 言語 | いじる（姉妹・承認済） | トップ（本設計） |
|---|---|---|
| 棚＋引き出し | ハブ＝パーツの棚→パーツ別引き出し | ＋作る→**作成の棚**／絞る▾→**フィルタの引き出し** |
| 契約（増えてもハブ不変） | ハブ契約＝スクロール0・新パーツ=タイル+1 | **トップ契約**＝固定6行・新kind=棚+1・チップはデータ実在時のみ |
| 3列タイル・レーン色・KindIcon | ハブのタイル棚 | 作成の棚（現行タイルの移設＝同じ絵・同じ順） |
| 状態の可視化＝扉 | タイル下端の状態チップ「ロック▾」 | 種別チップの選択リング／絞り中はチップがそのまま解除の扉 |
| 主動線タップ死守 | おまかせ生成2タップ不変 | 検索1・絞り1・続き1を死守（作成のみ1→2＝§3.1で根拠と保険） |
| B-lite（軽い合流） | — | 検索欄にkind名→「＋◯◯を作る」行（S5） |
| 増分の規約を明文化 | 新パーツ/新ノブ/横断設定の3ルール | §2.2の5ルール（コードコメントに明記） |

ホームの棚＝「ネタを増やす」、いじるの棚＝「パーツを生やす」。**どちらも「タップ＝実行、増えるものは棚の中」**＝アプリ全体が一つの思想。

## 5. 「増える」への耐性（ルールとして明文化）

1. **新kindが増えた** → `kinds.ts`（SSOT）に1行＝棚に自動で+1タイル・引き出しの全種別に+1行。**トップのDOMは不変**。ネタを作った日に件数順でチップ候補になる。
2. **新しい起票手段（取込の親戚）が増えた** → 棚の取込ゾーンへ。トップ不変。
3. **新しい絞り軸（mood/key/meter…）が増えた** → 絞る引き出しへ。トップ不変。
4. **20 kind時代の姿**：トップ＝固定6行のまま。棚＝タイル20（棚はスクロール可＝壊れるのは棚だけ、姉妹§2.2と同じ割り切り）。チップ＝相変わらず≤6（自分が使う6種だけが前に居る）。

| **S5 検索合流（B-lite）** | q が KIND_LABEL に前方一致したら一覧先頭に「＋『◯◯』を作る」アクション行（createBlank呼ぶだけ） | App.tsx or NetaList | 「メロ」入力で作成行が出る／tapで作成 |

- **S1-S3 だけで「2枚の壁の撤去」は完成**（S4/S5は主役の磨き）。各スライスは main 直・単独で出荷可能。
- リスク最小の根拠：データ取得・作成・検索の契約は不変＝壊せるのはJSXの配置だけで、webコンポーネントテスト（aria流儀）で先に赤を書ける。デスクトップは当面同構造が左レールに載るだけ（棚はモバイル=シート/PC=ポップオーバー。`@media`分岐は現行流儀）。
