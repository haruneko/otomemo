# 「いじる」機能のUX再設計提案（2026-07-10）

> 状態＝**設計提案（実装なし）**。対象＝`apps/web/src/components/SectionEditor.tsx`（生成/加工/書き出しの「いじる▾」メニュー）＋候補パネル（`reshape-bar`）。
> 起点＝オーナー（作曲家・修羅場P・**モバイル主体**）の明言「今のUXはかなり悪い」。
> 主動線＝**速く候補を見て・聴いて・選ぶ**。設計思想＝機械は候補/選択肢まで・仕上げは人間／改善は選択肢・ばらつき・足場に振る／モバイル一画面ずつ（[[project-design-philosophy-options-not-finished]]）。
> 前提の再考＝先行案 [2026-07-10-melody-menu-consolidation](2026-07-10-melody-menu-consolidation.md) は「メニュー内で二段化」だが、**本doは"メニュー(ドロップダウン)という器そのもの"を疑う**。

## 0. 所見の出どころ（正直に）

- **コード精読が主**（API起動が落ちていた＝`127.0.0.1:8787/health` 不通・Playwright実機は追わず、CLAUDE近辺の起動前例=[[reference-run-stack-and-e2e]] は温存）。
  痛点は**実コードの構造と、CSSに残る"応急処置コメントの地層"**から読み取った＝コメント自体が「器が限界」の自白になっている（後述 §1）。
- 実機で耳/指で確かめるべき点は各所に **【要実機】** と明示。既定値較正・触り心地の最終判断は音を聴ける時に。

## 1. 現状UXの解剖と痛点（file:line 実証）

### 1.1 器＝ドロップダウン `.tools-menu` が破綻している

「いじる▾」は `.assign-menu`（器ピッカー流用）の派生で、1枚のドロップダウンに **生成ボタン＋13ノブ＋メロ加工＋書き出し** を全部詰めている（`SectionEditor.tsx:560-713`）。CSS の履歴コメントが**器の限界**を証言している：

| CSS(`assets-bass.css`) | 何を後追いで塞いだか＝器が悲鳴を上げた証拠 |
|---|---|
| `:635 flex-wrap:nowrap` | 「縦カラム＋高さ上限で2カラム目へ折り返し、MIDI等が右へあふれる」→単一列固定 |
| `:640 max-height:min(66vh,520px); overflow-y:auto` | 「ノブが10個に増え縦に伸びた＝再生バー裏で下が見切れる」→**ドロップダウン内スクロール**を導入 |
| `:645 max-width:min(216px,…)` | 「right:0起点で240px左へ伸ばすと画面左に約12pxこぼれる（Playwright実測 x=-11.5）」 |
| `:648 z-index:7` | 「下端固定の再生バー(.transport z-index:6)より上に」＝**メニューが再生バーと殴り合っている** |

→ **幅216px・高さ66vh・内部スクロール・z-index戦争**。これは「ドロップダウン」の使い方ではない。オーナーの「そもそもドロップダウンに収まらなくない?」は**正しい**。ドロップダウンは3〜7項目の**選択**のための器で、13本の連続パラメータ＋候補ワークフローを載せる器ではない。

### 1.2 ノブ13個＝114行の壁（`SectionEditor.tsx:578-693`）

`gen-knobs` ブロックに13コントロール：

- **基本段（常時）**：細かさ`density` / 跳ね`swing` / 表情`expression` / 走句`runs`（:591-614）
- **詳細段（"▸詳細"折り畳み `detailsOpen`）**：食い`push` / 人間味`humanize` / 自由さ`foreground` / 入り遅れ`breathe` / 句割り`phrasing` / 形式`form` / 対位`counter` / 反復音`hook` / 発音`articulation`（:625-691）

問題は数だけでない：
- **全部が同格のスライダー/セレクト**で、視覚的な優先度も、値の意味（0=無効か中庸か）も伝わらない。`knob-val` は「疎/中/細」等の1文字（:594）で、**生成結果とノブの因果が見えない**＝回しても何が変わるか分からない＝回さなくなる。
- **説明はホバーtitle**が主で、スマホでホバーが無い→`？各つまみの説明`トグル（:582-590 `knobHelp`）を後付け＝**さらに縦に伸びる**。器の限界を器の中で塗り重ねている。
- 詳細段の `counter`（対位）・`hook`・`articulation` は**ベース/リズムレーンが在って初めて効く文脈依存**（:670 disabled、:428 hook>0時のみ送信）なのに、無関係な時も同じ場所にベタ置き＝ノイズ。

### 1.3 候補（`cand`）に"ネタとしての見た目"が無い ＝ 最大の痛点

生成結果 `cand`（`SectionEditor.tsx:79`）は `reshape-bar`（:727-747）に出るが、中身は**テキストラベル＋ボタン列だけ**：

```
[メロ候補（この進行に生成）]  [▶試聴] [別案] [メロレーンに置く] [閉じる]
```

- **視覚プレビューがゼロ**。`▶試聴`（:734 auditionCandidate）で鳴らすまで中身が見えない＝**耳を使うまで候補が"見えない"**。モバイルで「速く見て選ぶ」動線の逆。
- **`MiniRoll`（`MiniRoll.tsx:41`）は `notes` を直接受け取ってピアノロールを描ける**のに、候補には差さっていない。レーン上のブロック（:798）やカード（SectionMini）では既に使われている＝**資産が在るのに候補だけ盲目**。差せば即「ネタっぽい見た目」になる。
- **候補は常に1つ**（`cand` は単数 state）。`別案`（:737-739）は**現在の候補を上書き**＝前の候補は消える。並べて比較できない。
- `別案` は `lastPartRef`（:100,437）で「同じop・同じノブ・seedだけ変える」（:417 `seed:Math.random`）＝**サイコロの芽は既にある**が、UIとして「振る」概念にも「候補を貯める」概念にも育っていない。

### 1.4 サイコロ（ランダム探索）が無い

`seed` は毎回 `Math.floor(Math.random()*1e6)`（:417）で振られる＝**seedは既にランダム**。だが**ノブ自体をランダムに振る手段が無い**。オーナーの言う「サイコロ」は「density/swing/… を範囲内でランダムに設定して探索」＝**パラメータ空間のガチャ**。今は13本を手で回すしかない＝設計思想「ばらつき」の器が無い。

### 1.5 その他の不足（生成体験の定石に照らして）

| 定石機能 | 現状 | 根拠file:line |
|---|---|---|
| 複数候補の**同時比較** | ✕ `cand`単数 | :79 |
| **バリエーション一括**（1押しでN案） | ✕ 1押し1案 | :435-437 |
| **お気に入り/keep**（気に入った候補を退避） | ✕ 置くか捨てるか | :488,516 |
| 生成の**直前に戻る**（Undo） | ✕ 候補フローはUndo対象外 | design.md:352 (U2 scope) |
| **regenerate**（同条件で振り直し） | △「別案」＝上書きのみ | :737 |
| **keep-generating**（気に入るまで回し続ける） | ✕ | — |
| ノブ→結果の**因果の可視化** | ✕ 数値ラベルのみ | :594 |
| 候補の**長さ/密度メタ**表示 | ✕ | — |

## 2. 再設計：器（コンテナ）を替える

### 2.1 器の候補と比較

モバイル一画面ずつ・44pxタッチ標的・下端に再生バー常駐（`.transport` z-index:6）を制約に、器を4案で比較：

| 案 | 器 | 収まる量 | モバイル適合 | 再生バー衝突 | 実装量 | 既存資産再利用 |
|---|---|---|---|---|---|---|
| **A** 現状踏襲（二段ドロップダウン） | `.tools-menu` | △(内部スクロール) | ✕ 幅216/z戦争 | ✕ 殴り合い | 小 | そのまま |
| **B ボトムシート**（下から迫り上がる生成パネル） | 新 `.gen-sheet` | ◎ 全幅×可変高 | ◎ 親指圏・一画面 | ◯ シートが上に被る自然な階層 | 中 | ノブ/候補UIを移植 |
| **C 全画面「生成モード」** | 新ビュー(`mv-gen`) | ◎ 無制限 | ◎ mv-pane流儀と同型 | ◎ 別画面で衝突なし | 中〜大 | mv-home/mv-pane土台(:backlog 56) |
| **D インライン展開**（ツールバー直下に生成ゾーンを開く） | `.section-editor`内 | ◯ | △ タイムラインを押し下げる | ◯ | 中 | — |

**推奨＝B ボトムシート**（モバイル主体・親指到達圏・下端再生バーと素直に階層化・実装中規模）。**Cは将来の受け皿**（生成が「モード」に育つなら＝§6フェーズ3）。**Dは補欠**（PC幅ではBよりD＝タイムラインと並置して即置ける利点があるので、`useIsMobile`(:backlog、design.md 820境界)で**モバイル=B / PC=D** の二形も可）。

理由：ドロップダウンは「選択」の器、ボトムシートは「作業」の器。生成は**設定→生成→試聴→比較→選ぶ**の作業フロー＝シートが正しい器（NN/g のシート/パネル定石・モバイルの一次操作面）。

### 2.2 器の中の構造＝「速く選ぶ」を最上段に

ボトムシートを**3ゾーン**に分ける（上から重要度順・スクロールは候補ゾーンのみ）：

```
モバイル幅（〜400px）・ボトムシート ▲下から

┌───────────────────────────────┐
│  ✨ メロを生成         [🎲] [✕] │  ← ヘッダ：対象パート切替＋サイコロ＋閉じる
│  ◉メロ ○ベース ○ドラム          │  ← パートセグメント(GEN_PARTS :394)
├───────────────────────────────┤
│  プリセット                      │  ← ①プリセット＝主役（速い当たり）
│  [しっとり][跳ねる][走る][歌もの]│
│  [きらきら] [+マイ設定]          │
├───────────────────────────────┤
│  ▸ 細かく設定する                │  ← ②詳細＝折り畳み（13ノブはここ）
├───────────────────────────────┤
│  候補 3件            [🎲もっと]  │  ← ③候補トレイ＝横スクロール・視覚
│ ┌──────┐┌──────┐┌──────┐ ⟩   │
│ │▁▂▆▃▂▁││▁▅▂▆▂▇││▂▂▂▆▅▂│     │  ← MiniRoll(:notes)＝ネタの見た目
│ │2小節  ││2小節  ││4小節  │     │
│ │♡ ▶ ＋ ││♡ ▶ ＋ ││♡ ▶ ＋ │     │  ← keep(♡)/試聴(▶)/置く(＋)
│ └──────┘└──────┘└──────┘     │
└───────────────────────────────┘
        ▼ 下端に既存の再生バー（不動）
```

- **プリセット主役**（§4）＝オーナーは「まず当たりを引く」。13ノブは詳細に沈める。
- **候補は視覚**（§3）＝MiniRollで"ネタっぽく"見える。横スクロールで複数比較。
- **サイコロ**（§5）＝ヘッダ🎲＝ノブごと振る／`🎲もっと`＝候補を追加生成。

## 3. 候補の視覚プレビュー（最優先・最小実装で最大効果）

### 3.1 MiniRoll を候補に差す（1行の話）

`reshape-bar`（`SectionEditor.tsx:727`）に、`cand.content` から notes を起こして `MiniRoll` を差すだけで**候補が見える**：

- notes 化＝既存 `notesForContent(cand.kind, cand.content)`（:485 auditionCandidate が既に呼んでいる）。
- `MiniRoll` は `neta` 必須引数だが `notes` を渡せば content を無視して notes を描く（`MiniRoll.tsx:44` `given ?? notesForContent(...)`）＝**ダミー neta＋notes** で描ける。※将来は `MiniRoll` を `notes` 単独でも描けるよう props を緩める小リファクタが素直（`neta` を optional に）。
- 併せて**長さ（小節数）・音数**をメタ表示＝`Math.max(start+dur)/BPB` と `notes.length`。「ネタっぽさ」＝ピアノロール＋尺＋密度。

### 3.2 候補を単数→配列にする（比較の土台）

`cand: {kind,content}|null`（:79）を **`cands: Cand[]`** に拡張し、各候補カードに操作を持たせる：

```
候補カード（1枚）＝
  [MiniRoll(notes)]   ← 見る
  [2小節 · 24音]      ← メタ
  [♡keep] [▶試聴] [＋置く] [🗑]
```

- `▶試聴`＝現 auditionCandidate（:482）を候補IDで。**1枚再生中は他を止める**（`candPlay` ref を候補キーで管理）。
- `＋置く`＝現 placeCandidate（:488・置換ロジックはそのまま流用）。
- `🗑`＝そのカードだけ捨てる（配列から除く）。
- `♡keep`＝§4.3 お気に入り。

### 3.3 モバイルの並べ方

- **横スクロールの候補トレイ**（`overflow-x:auto`・カード幅≈150px）＝一画面に2.5枚見えて「まだ右にある」を示す。縦積みは再生バーとスクロールが競合する＝横が良い。
- 【要実機】候補カードのMiniRoll高さは30px（`MiniRoll.tsx:49 H=30`）だと小さい＝候補用は**60〜72px**に上げて輪郭を読ませる（カード用とは別サイズ引数）。

## 4. ノブ13個を「プリセット中心＋詳細折り畳み」に

### 4.1 プリセット＝第一級（速い当たり）

先行案（[2026-07-10-melody-menu-consolidation](2026-07-10-melody-menu-consolidation.md)）は「基本4＋詳細9」の二段。**本doはその上に"プリセット層"を主役として被せる**＝オーナーの主動線「速く当たりを引く」に直結し、13ノブは**触らなくても成立**させる。

プリセット＝web定数（state一括セット・API不変）。素案（【要実機】で命名/値は較正）：

| プリセット | density | swing | expression | runs | 効かせる詳細 |
|---|---|---|---|---|---|
| しっとり | 0.3 | 0 | 0.5 | 0 | breathe 0.4 |
| 跳ねる | 0.5 | 0.6 | 0.2 | 0.2 | push 0.3 |
| 走る | 0.8 | 0.1 | 0.2 | 0.7 | — |
| 歌もの | 0.45 | 0.15 | 0.3 | 0.1 | hook 0.5 / articulation 0.4 |
| きらきら | 0.7 | 0.2 | 0.4 | 0.5 | foreground 0.5 |

- 各プリセットは**「基準点」**＝そこからサイコロ（§5）で周辺を振る／詳細で微調整、が自然な流れ。
- `+マイ設定`＝今のノブ状態に名前を付けて保存（localStorage）＝オーナーの手癖を貯める（[[project-design-philosophy-options-not-finished]]「自分らしさの燃料」）。

### 4.2 詳細段＝文脈で出し分け

`▸細かく設定する` を開いた時だけ13ノブ。ただし**同格ベタ置きをやめ、効く文脈でグルーピング**：

- **表情**（常時）：細かさ/跳ね/表情/走句/食い/人間味/自由さ/入り遅れ
- **構造**：句割り/形式/反復音/発音
- **連動**（レーン依存・在る時だけ表示）：対位（要ベース :670）／【将来】ドラム連動 backbeat/drumLock（:433 は今ハードコード）
- 各ノブに**現在値のプレビュー**（数字でなく「これを回すと○○」の一言 or ミニ図）＝ knobHelp（:582）を常時1行の副題に格上げ。

### 4.3 お気に入り（keep）

候補カードの `♡` ＝その候補を**"keptトレイ"に退避**（配列で保持・再生成しても消えない）。置くか捨てるか迷う候補の一時避難＝生成体験の定石（気に入りを貯めて後で選ぶ）。実装＝web state（`kept: Cand[]`）＋シート下部に「キープ N件」小トレイ。永続化は任意（localStorage）。

## 5. サイコロ（ばらつきの器）

### 5.1 二種類の「振る」を分ける

- **🎲 ノブを振る**（ヘッダ）＝density/swing/…を**範囲内でランダム設定**して1案生成。「設定が思いつかない/探索したい」時の入口。
- **🎲もっと**（候補トレイ）＝**今の設定のまま seed だけ変えて**候補を1〜3枚**追加**（既存 :417 seed=random の素をUI化）。「この方向で別の当たり」。

### 5.2 サイコロの粒度＝「固定して振る」

設計思想「ばらつき」の肝＝**気に入った軸は固定し、残りだけ振る**。各ノブに**🔒ロック**を持たせる：

```
細かさ  ▬▬●▬▬  🔒   ← ロック＝サイコロで動かさない
跳ね    ▬●▬▬▬  🎲   ← 振る対象
表情    ▬▬▬●▬  🎲
```

- 🎲を押す＝ロックされていないノブだけを各レンジ内でランダム設定→生成。
- **プリセット＋一部ロック＋サイコロ**＝「歌もの の density は固定、他は振ってみる」＝探索と制御の両立＝**機械は候補まで/人は方向を握る**そのもの。
- seed 自体もロック可（同じ乱数で他ノブだけ変える＝A/B比較）＝【将来・任意】。

### 5.3 レンジの既定

各ノブのサイコロ範囲は**プリセット中心±幅**（例：density 0.3〜0.8）＝突飛な値で毎回ハズレを引かない。範囲は web 定数（較正は【要実機】）。

## 6. 推奨とフェーズ（効き順×実装量）

**原則**：API契約は一切触らない（`gen_melody` schema/opts/http 不変・[2026-07-10-melody-menu-consolidation](2026-07-10-melody-menu-consolidation.md) と同じ鉄則）。全部 `SectionEditor.tsx`＋CSS＋新小コンポーネントに閉じる。既定挙動＝ノブ未指定で従来 bit 一致（:420 body組み立ての条件送信を維持）。

| フェーズ | 内容 | 効き | 実装量 | 依存 |
|---|---|---|---|---|
| **P1（最優先・最小）候補の視覚化** | `reshape-bar` に MiniRoll 差し込み＋長さ/音数メタ（§3.1）。`cand` 単数のまま。 | ◎ 大（「見て選ぶ」が即成立） | 小 | MiniRoll props 微緩和 |
| **P2 候補を複数＋keep** | `cand→cands[]` 配列化・横スクロールトレイ・♡keep・🗑・per候補試聴（§3.2/4.3）。「別案」→「もっと」= seed違いで追加。 | ◎ 大（比較・keep-generating） | 中 | P1 |
| **P3 器をボトムシート化** | `.tools-menu` の生成/ノブ/候補ブロックを新 `GenSheet` へ移設（§2）。z戦争/幅216/内部スクロールを解消。書き出し・メロ加工は現メニュー据え置き可。 | ◎ 大（器の限界を撤去） | 中 | — |
| **P4 プリセット主役** | シート頭にプリセット5＋マイ設定（§4.1）。13ノブは`▸詳細`に沈める。 | ◯ 中（速い当たり） | 小〜中 | P3 |
| **P5 サイコロ＋ロック** | ヘッダ🎲（ノブ振り）＋ノブ毎🔒＋範囲定数（§5）。 | ◯ 中（ばらつき） | 中 | P4 |

**推奨着手順＝P1→P2→P3→P4→P5**。P1/P2 は器を替えずに**現 `reshape-bar` の中だけ**で効く＝低リスクで最大の痛点（候補が見えない）を先に潰す。器の作り替え（P3）はその後。P4/P5 は器が替わってから。
