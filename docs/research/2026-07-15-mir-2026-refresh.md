# MIR 2025–26 再サーベイ＝feasibility doc の鮮度更新（R3）

**目的**：`audio-analysis-feasibility.md`（2024–25時点の調査）以降、2025–2026に出た MIR モデルで現行スタックを更新すべき差分を洗う。現行＝**Demucs htdemucs**（分離）＋**librosa**（BPM/ビート）＋**BTC-ISMIR19**（コード・large_voca）＋**pyin**（メロ/ベースf0）。環境＝WSL2・**CPUのみ（GPU無し）**・torch有。原則＝**ローカルCPU自前・商用可ライセンス（GPL/AGPL回避）・pretrained必須（自前訓練しない）**。全主張に出典。法的助言ではない。

---

## TL;DR（正直な結論）

大枠は **feasibility doc の結論を覆す「乗り換え必須」級は無い**（コードの天井もボーカル採譜の弱さも据え置き）。ただし**周辺の"軽さ・導入しやすさ"が実質改善**しており、乗り換え価値のある発見トップ3は：

1. **ビート/ダウンビート＝`beat_this`（CPJKU・ISMIR2024・MIT・CPU可・pretrained）に乗り換え推奨**。madmom（NC・Py腐敗）を使わずに transformer で SOTA 級（GTZAN downbeat F≈75.5%）、DBN後処理不要＝madmom依存を切れる。C++/ONNX 版もありCPU軽い。→ **librosa の弱点（ルバート・ダウンビート・拍子）をMITで埋める初の実用駒**。
2. **f0（音域/ピッチ曲線）＝`PESTO`（Sony CSL・MIT・pip）に乗り換え検討**。CREPE比 **30kパラメータ＝約800分の1**・**CPUで実時間の12倍速**・精度はCREPE比 −1.7%以内。torchcrepe より圧倒的に軽く、音域抽出の常時実行に向く。
3. **コード認識の天井は動いていない**（ChordFormer/ISMIR2025 consonance系＝改善は限界的、しかも**コードとweights未公開でpretrained原則に反し採用不可**）＝**BTC 継続が正解**を再確認。これは「更新なし」を正直に確定できたこと自体が成果。

その他：**音源分離**は BS-RoFormer が `bs-roformer-infer`（MIT・pip・自動DL）で**導入は容易化**したが CPU では依然 htdemucs より重い＝**A/B実測して様子見**。**ボーカル採譜**は basic-pitch の後継(v2)は無いが、歌特化の **ROSVOT / STARS** が登場＝basic-pitch の vocals note-F≈49% を超える可能性、ただしGPU寄りで**実測要**。**キー推定**の新顔は無し（更新なし）。

## 分野別・所見

### 1. コード認識 — 天井は動いていない（BTC継続）
- **ChordFormer**（2025）はコードラベルを triad/bass/seventh に分解して class imbalance を緩和する新機軸だが、実数は **sevenths 72.3% / tetrad 65.3%** で **BTC の 71.8/65.5 とほぼ同値**＝feasibility が言う「maj-min ~82-87%が10年の天井／7th以上は候補止まり」を**そのまま追認**。しかも**コード・weights未公開・ライセンス記載なし**＝「pretrained必須」原則に反し**採用できない**。
- **ISMIR2025 の consonance-based ACE**（知覚協和度で誤差を重み付けする学習）も BTC 比の改善は「一部指標で限界的」と査読で指摘、再現物も未確認。
- **含意**：BTC-ISMIR19（MIT・pretrained・CPU可）を**据え置きが最適**。テンションは候補主義のまま。乗り換え価値＝**なし**。

### 2. 構成解析 — allin1 の代替は現れず、依存地獄も未解消
- allin1 の**後継/上位互換は見つからない**（勝者のまま）。だが **NATTEN の API が 0.21 系で変わり allin1 が旧APIを要求／madmom が Py3.10+ で `collections.MutableSequence` 撤去により壊れる**という**導入問題は2026時点でも継続**。「pip wheelで一発」には**なっていない**（librosa/MSAF フォールバックで逃げる運用が推奨されている）。
- **含意**：構成（Intro/Aメロ/サビ）を出すなら依然 allin1 だが、**インストールコストは高いまま**＝後回し継続が妥当。ただし**ビート/ダウンビートだけなら下記 beat_this で allin1 抜きに MIT で取れる**ので、構成ラベルが要らない局面では allin1 を回避できる。

### 3. ビート/ダウンビート — beat_this で madmom を切れる（今回の実質最大の駒）
- **beat_this**（CPJKU・ISMIR2024・**MIT**）：transformer で **DBN後処理なし**に SOTA 級。**pretrained weights も MIT**、**GPUが無ければCPUに自動フォールバック**、madmom は `--dbn` を使う時のみ任意。GTZAN downbeat F≈75.5%。**ONNX の C++移植**（beat_this_cpp）もあり CPU 実行が軽い。
- feasibility は「madmom＝2018停止・Py≤3.7・モデルNC＝避ける」「librosa は定常4/4なら十分・ルバート弱・ダウンビート/拍子は弱」としていた。**その穴を初めて MIT・CPU・pretrained で埋める駒**。
- **含意**：BPM/ビート/ダウンビートの権威を **librosa → beat_this** に置換検討。librosa は軽量フォールバックとして残す。

### 4. 音源分離 — 導入は容易化、CPU価値は実測待ち
- **bs-roformer-infer**（**MIT**・pip・チェックポイント自動DL・整合性検証）が登場し、**BS-RoFormer（≈12dB SDR）を"梱包済み"で試せる**ように（2024時点は UVR コミュニティ配布でライセンスが曖昧だった）。torch 実行なので**CPU可だが 700MB ckpt＝htdemucs より確実に重い**。
- **Moises-Light**（WASPAA2025）：BS-RoFormer 同等 SDR を **13×小（≈5M/stem）**＝CPU本命になり得るが、**公式weightsは未公開**（非公式実装のみ）。
- **含意**：feasibility の「真SOTA(BS-RoFormer)はGPU級で非現実／htdemucsがCPUの甘い所」は**まだ概ね正しい**。ただし**導入障壁が下がった**ので、**F1実測のついでに `bs-roformer-infer` の CPU 実時間と SDR を htdemucs と A/B**する価値はある（品質差 9→12dB が分離後のメロ/コード解析にどれだけ効くか）。乗り換えは**様子見**。

### 5. 採譜 — basic-pitch v2は無い／歌特化の新顔が実測候補
- **basic-pitch の後継(v2)は存在しない**（フォーク・WASM移植・DAWプラグイン化のみ）。多声・vocals の弱さ（note-F≈49%）は据え置き＝**既定維持**。
- **歌声特化の新顔**：**ROSVOT**（ACL2024・**MIT**・**ckpt配布**・**混合音でも転写可**＝分離前提でない）と **STARS**（ACL2025・音符+技法+スタイル統合）。いずれも **basic-pitch の vocals note-F を上回る可能性**があるが、**GPU寄り・研究コード寄り**でCPU実行と実精度は**実測要**。
- **含意**：ボーカルのメロ採譜を本気で上げたいなら **ROSVOT が第一候補**（MIT・pretrained・混合対応）。ただし CPU 実行可否は要検証＝F2（採譜比較）で **basic-pitch vs ROSVOT** を測るのが筋。

### 6. 周辺（f0・キー）
- **PESTO**（Sony CSL・**MIT**・pip・ISMIR2023、2025にジャーナル版）：**30kパラメータ（CREPEの約1/800）・CPUで実時間12倍速・精度はCREPE比 −1.7%以内**。音域抽出は「voicing/信頼度でゲート→パーセンタイル」の後処理が精度を決めるので、**f0エンジン自体は軽い PESTO で十分**＝torchcrepe 置換の有力候補。
- **RMVPE**（RVC で実績）：**混合音源から分離せず vocal f0 を直接**・全SNRで頑健・CREPE_full より軽い。**分離パスを一段省ける**可能性があるが、**ライセンス要確認**。
- **キー推定**：独立した pretrained 新顔は見当たらず。feasibility の「調はBTCコード頻度から導く」（POCで検出器83% vs コード整合96%）方針を**据え置き**。MERT/MuQ 等の音楽表現埋め込みは probing 前提で自前訓練が要る＝原則外。

## 後続の実測（F1/F2）への影響

- **F1（構成/allin1 実測）**：allin1 は**そのまま測る価値あり**（後継なし）。ただし**導入は依然 NATTEN/madmom 地獄**を覚悟。**同じ回で `beat_this`（MIT・CPU）を並走**させ、ビート/ダウンビートだけなら allin1 抜きで足りるかを判定すると、構成ラベル不要ケースの依存を軽くできる。テンポ倍取りフラグの検証も beat_this 側で。
- **F2（採譜比較）**：**basic-pitch vs ROSVOT（vocal stem 上）**を測るのが最重要更新。basic-pitch の vocals≈49% を ROSVOT がどれだけ上げるか、**CPU 実行時間**込みで。f0 曲線は **torchcrepe vs PESTO** を同時に測れば音域抽出の軽量化も同時決着。
- **分離 A/B（新規提案）**：F1 の入力生成のついでに **htdemucs vs bs-roformer-infer(BS-RoFormer)** の CPU 実時間 と、分離後のコード整合率(BTC)・メロ採譜精度の差を測ると「9→12dB がダウンストリームに効くか」が定量化できる。効かないなら htdemucs 据え置き確定。
