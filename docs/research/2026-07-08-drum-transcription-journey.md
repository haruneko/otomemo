# ドラム抽出 続き＝全曲書き起こし→区間分解（#S12改2/改3・試行錯誤の記録）

日付: 2026-07-08 ／ 前提: Fable版 `extractDrumPattern`（窓分割×型照合・1小節グルーヴ）を土台に、
オーナー指摘「1小節ループはゴールでない・全曲を音楽的区間に分解してネタ化が理想」を受けた続き。
※これはフィジビリ＝試行錯誤の journey をそのまま残す（何を試し何が壊れ何が効いたか）。

## ゴールの変遷（オーナーの是正）
1. （当初・私のバイアス）音源→**1小節ループ**のドラムネタ。
2. オーナー是正：**1小節ループは曲の中に存在しない・ビート抽出用の中間物**。それを種に横展するのは筋が悪い。
   **理想＝実ドラムを全曲書き起こし → 程よい音楽的区間に分解 → 区間ごとにネタ化**。

## 試行錯誤（順に）
### 試み1：全曲を1本の連続グリッドで per-bar 書き起こし（`transcribeFullSong`）
- 局所テンポ(窓)を台形積分して連続グリッドを作り、全オンセットを各小節へ量子化。畳まない。
- TDD＝小節0-3と4-7で叩きが違う曲→各小節を別々に書き起こす、は緑。**構造は正しい**（99小節・94ユニーク）。
- **しかし実データで2つの壁**：
  1. **perception ノイズがむき出し**：全オンセットを拾う＝分離ブリード/ゴースト/誤分類が全部ヒット。
     畳みループが綺麗に見えたのは**多数決の平均でノイズが隠れてただけ**（オーナーの「筋の悪い中間データ」が的中）。
  2. **連続グリッドがドリフト**：φ積分の周期誤差が99小節で累積→後半が前半とズレ→**集約が滲む**。
     Fableの畳みが綺麗だったのは**窓ごとに各窓のdownbeatへ回転してから足す**＝ドリフトを窓に閉じ込めてたから。
- 位相の詰め：スネア=2/4拍（半拍・拍上）＋**crash=小節頭で半小節の裏表を割る**（ロックの突っ込み＝
  crashが8分早いケースも許容）を実装。ただし上記ドリフト/ノイズで per-bar も集約も綺麗にならず。

### crash 検出（perception 4本目レーン・オーナー指摘）
- crashはセクション頭(小節1拍目)に入る＝(a)半小節の裏表 (b)区間境界。ハットと同じ高域だが**長い減衰**で区別。
- 実stem検証：LostMemory 11/DeepSea 17/SURFACE 6、いずれもセクション尺の間隔。小節内位相の集中度0.62
  （9/11が同一拍位置＝downbeatアンカーになる）。

### 試み2（採用）：**crashで区間を切り、区間ごとに `extractDrumPattern`**（`extractSectionPatterns`）
- **気づき**：連続グリッドは間違い（ドリフト）。区間内は1グルーヴ＝そこでの「1小節畳み」は**本物のグルーヴ**
  （曲全体の1ループ化とは別物）。＝Fableのドリフト無し機構を**区間化**すれば、ドリフトもノイズも区間平均で消える。
- 実装：crash近接統合→境界（短区間は貪欲に前結合＝最低minBars保証）→区間onsetsに extractDrumPattern
  （拍子は全曲meterをforceMeter）→連続で同一パターンの区間は結合（Aメロ×2等）。
- **LostMemory 実データ結果**：6区間に分かれ、区間ごとに違うグルーヴ＋**綺麗なバックビート**が出た：
  - `149-215s(24小節) conf0.34 rock+` → `S:.... x... .... x...`＝**スネアが step4,12＝2・4拍でクリーン**
    （全曲集約では出せなかった綺麗さが区間内では出る＝ドリフト回避の効果）
  - `265-299s(13小節) conf0.74` 高信頼／`215-265s 型four`＝四つ打ち区間、とセクションで違う。
  - 転換部(conf0.00)はゲート下＝ネタ化しない（グレースフル）。
- TDD：2区間の合成曲（前半8ビート/後半四つ打ち・区間頭にcrash）→2区間で別パターン(kick[0,8] vs [0,4,8,12])。
  crash無し→全曲1区間（後方互換）。api 579緑。

## 🔴 6/8検証で判明＝perception のオンセット過検出（Breathe -retrieval-, 修羅場P）
- **拍子は当たっている**：型に依らない畳みピーク性（オンセットを各 meter で畳み集中度を測る）で
  **meter6=1.63・3=1.57 >> 4=1.30・2=1.24**＝**明確に3連/6拍系＝この曲は本当に6/8**（LostMemoryは較正で meter4=1.97 が最大＝正しく4/4）。
  beat 安定性はむしろ良い（CV0.02）＝拍トラッキングは犯人でない。
- **なのに extractDrumPattern は conf0・全 meter score0・support0**＝どの正準型にも窓レベルで1つも合致せず、
  区間ネタ0枚（analysis は meter を chords へフォールバック）。
- **根本原因＝オンセット過検出**：kick **15.7発/小節**・snare 11.3発/小節（6/8＝1小節8分6枠なのに1枠2発超＝物理的にありえない）。
  対して LostMemory は kick 5.1/小節で妥当。**Breatheは約3倍過剰に鳴っている**＝畳みヒストグラムが飽和→全ステップが「叩いてる」に
  見え型照合が総崩れ。＝`analyze.py drum_onsets` のバンド別オンセット検出が、密/ラウドなミックスで over-trigger（ベース漏れ・ゴースト）。
- **含意**：6/8 の型は正常。詰めるべきは **perception のオンセット間引き**（バンド別に不応期/percentile 閾値/ピーク間引き）。
  ＝オーナーの「品質が低いなら音源分離から疑っていい」に一致。現状は busy 曲で **グレースフルに無出力**（安全）＝誤ネタは出さない。

## 6/8検証②＝米津玄師「orion」（別の失敗モードを露呈）
- **拍子**：型非依存の畳みピーク性で **meter4=2.35（snare）が最大**（6=1.89・3=1.68）＝この曲は**4/4に読める**
  （6/8でないか、6/8感が拍トラッキング層に出ない）。s4=0.524>s3=0.126＝ストレート。＝6/8のテスト曲としては不適だった。
- **密度は正常**（kick 6.9/小節・LostMemory並）＝Breatheのような過検出ではない。
- **なのに conf0・support0・0区間**。理由＝**snareが標準バックビート(2・4拍)に無い**（3.0/1.5/0.0に散る＝米津の重ねパーカス/
  クラップを snareバンドが拾う）。型(rock/four = snare on 1,3)は**バックビート前提**なので合致せず正窓ゼロ。
- **含意**：型ライブラリは backbeat groove 前提＝**非バックビート/装飾過多の曲は語彙外で無出力**（グレースフルだが hit率は上がらない）。
  ＝実曲2連続で別々に失敗（Breathe=過検出／orion=語彙外＋4/4読み）。**6/8の型流用は依然“クリーンに通る実曲”未発見で未検証**。
- **学び**：抽出の実曲 hit率は限定的（素直なバックビート曲＝LostMemory/DeepSeaは通る／密or装飾過多は落ちる）。
  失敗は**静かに無出力**＝安全側。6/8検証には「型に嵌まる素直な6/8（遅め・ドラム疎・生っぽい録り）」を選ぶ必要。

  Breathe の kick/小節を物理的に妥当な密度へ。TDD＝密度上限・既存 LostMemory/DeepSea が非退行。是正後に 6/8 区間ネタが出るか再検証。
- **3/4・6/8 の型流用**：perception 是正後に Breathe(6/8) で再検証（型自体は candidates=[4,3,6] で既に試行済）。
- **SURFACE(シャッフル)の区間分解**：sub=3 の区間畳みが綺麗に出るか未検証。

## 学び（要点）
- **全曲を1グリッドで書き起こす＝ドリフトで破綻**。区間に切って区間内で畳むのが正解（区間内はドリフト小・
  グルーヴ一定・ノイズは平均で消える）。＝オーナーの「区間分解」は理想であると同時に**質の問題への解**でもあった。
- **crashは構造の鍵**：半小節の裏表＋区間境界を1つの手がかりで両方決める（ただし突っ込みで8分早い＝要許容）。
