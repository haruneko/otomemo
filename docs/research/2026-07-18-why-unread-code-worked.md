# なぜ「コードを一行も読まない」のに高品質にできたのか — 開発プロセス自己分析（2026-07-18）

オーナーの問い：「10万行規模を一行もコードを読まずに作ってもらったのに、恐ろしいくらいいい感じにできている。原因はどこか？」

分析方法：①リポジトリ側（スペック文書・テスト構成・git履歴）と②セッションログ側（85セッション・801MB・オーナー実発話1,371件を全件集計＋168件精読）を別エージェントで独立分析し統合。数字はすべて実測。

## 前提の実測値

- 実装コード（テスト除くTS）**39,220行**＋テスト**26,463行/191ファイル**＝全TSの**約40%がテスト**。29日間（2026-06-20〜07-18）で**943コミット**、revertは**1件**のみ。
- コミット内訳：feat 286 / **docs 187（20%）** / fix 143（15%）/ refactor 27。
- research文書148本、design.md 2,068行。
- ログ側：型チェック実行**1,071回**・テスト実行**806回**・Playwright/e2e **597回**・curl実打635回・Agent起動**464回**。
- **オーナー発話でコード片・関数名・実装ファイル名に触れたものは0件**（コードフェンス含有0件。挙げるパスはdocsのみ）。「一行も読んでいない」はログ上裏付けられる。

## 結論：単一の原理＝「検証責任の再配置」

オーナーが手放したのは「コードを読むこと」だけで、「判断すること」は一切手放していない。全成果物が“判断が届く形式”（日本語の裁定・実測値・耳チェックリスト）に変換され続け、機械で保証できる範囲をbit一致と決定性で極限まで広げた。

| 層 | 担保者 | 手段 |
|---|---|---|
| 設計の正しさ | research実測＋オーナー裁定 | 148本の出典・実測つき文書 |
| コードとスペックの一致 | 双方向トレーサビリティ | design⇄コード参照263箇所、コミット「＝」形式419/943件 |
| コードの正しさ | 機械（テスト40%） | 契約SSOT(zod単一源)・プロパティ不変条件・bit一致28ファイル |
| 受け入れの公正さ | 独立Acceptor/素眼Regressor | review-loop.md（実装者は自己採点しない） |
| 機械で測れない質 | オーナーの耳のみ | ［耳/手］チェックリストとして明示配送 |

## 原因の分解（7要素）

### 1. 初日に自分でプロセスを敷いた（起点）
6/20のオーナー発話「実際に作るときは結構SDDとTDDを徹底しないと破綻しそう。実装時のルールって決められる？修正時にはコンセプト、アーキテクチャ、要件定義、設計、タスク、テスト、コードの順に確認せよみたいなの」→ CLAUDE.mdの「変更は上位から下ろす」ルールの起源。コードだけの場当たり修正が構造的に禁止され、全セッションに継承された。

### 2. 機械ゲートの高回転（コードレビューの代替）
コミット1回あたり型チェック1.2回・テスト0.9回。「全緑でないと報告できない」文化。さらに：
- **契約SSOT**：`apps/api/src/schemas.ts` の zod を HTTP検証・MCP inputSchema・型導出の3用途で共有＝乖離事故を構造的に排除。
- **bit一致鉄則**（design.mdに62回登場・28テストファイルが強制）：新ノブは「既定0＝既存出力とbit一致」で足す→機能追加＝回帰ゼロを機械証明。リファクタは逐語コピー＋deepEqual突合。
- **プロパティ/不変条件テスト**：拍子×mood×小節×seedの全走査で「非空・有限・妥当音域」を担保。

### 3. 機械で測れる/測れないの境界を正直に引いた（ドメイン設計思想）
「機械は候補まで・仕上げは人間」「Claudeは音符に触らない＝決定的ルールエンジンが音符を作る」。この2つが効いた理由：
- **目標の再定義**：「良い曲」は機械判定不能だが「理論的に破綻しない候補を複数」は不変条件で判定可能。品質バーを検証可能な位置に置いたこと自体が品質を可能にした。
- **非決定性の隔離**：LLMを音符生成から締め出し純関数＋seedにした→全生成がテスト可能・再現可能・bit一致検証可能。テスト26,000行はこの決定性の上に成立。
- design.mdは機械で受け入れ不能な項目を「［耳/手］未消化」として番号付き列挙し、オーナーの手番として明示配送。

### 4. オーナーは「受け入れ試験機」に専念（人間の時間の全振り先）
発話は中央値49字・41.5%が質問形。実装方法をほぼ指定せず（「DAGってなに？？わかんない」）、体験の言語で舵を切る。一方、実物フィードバックは異常に精密：
- 耳感想93件(6.6%)・UI感想172件(12.3%)。「16beatsスネアが二拍ずれてる x..x .x.. x...」「MIN2の和声的旋法のシャープがメロに反映されていない」＝**コードは読まないが出力は診断レベルで読んでいる**。
- ダメ出し188件(13.4%) vs 承認254件(18.2%)＝約1.3:1。「妥協ダメだよ、それは聞くまでもなくNG」「人間の確認が必要になるまでやって」が品質下限を規定。
- 検証コストの非対称性を突いた配置：コードレビューは重いが、音を聴く・画面を触るのは速くて確実。**音楽ドメインの専門性がなければこの受け入れ精度は成立しない**（同手法でも他分野なら品質は下がる可能性＝本分析の限界）。

### 5. 監査の多重化と自己採点の禁止
Agent起動464回（Fable=計画/設計33回、Opus/無指定=実装342回、他）。「機能とデザインの監査は混ぜない」（オーナー指示）。review-loop.mdでPlanner/Implementer/Acceptor/Regressorを分離し「実装者は自分の成果を受け入れOKと判定しない」「Regressorは素眼」。特徴的なのは**サブエージェントをユーザー役にして使わせ、その感想を監査として受ける**手法。夜間自律実行も実在（7/15深夜03-04時台にオーナー発話ゼロで監査133チェックが回った）。

### 6. 違和感駆動の反復と「完成宣言を人間が握る」
苦情→解消ペア23組を検出。パターンは「差し戻し→修正→実物で再確認→部分承認→次の違和感」の連鎖（「よくはなった、音長が切れちゃうのが悩ましい」）。1回で全面OKにしない。こじれたら「次に指示残してくれる？？したらclearするから」＝引き継ぎ書でコンテキストを仕切り直す回復パターン。

### 7. 失敗を都度「仕組み」に変換
- UI崩れ→「ヘッドレスブラウザで自己チェックできる仕組み作ったほうが」→Playwright監査体制
- 記憶揮発→「ステートレスな報告に直して」→報告様式のルール化
- 保留の散逸→backlog.md＋「着手ルール：なぜ今それが要るか問う」（AIの過剰実装ブレーキ）
- 判断の蒸し返し→オーナー裁定を文書に帰属記録（「#8b対応せず（オーナー裁定2026-07-13）」）
- 自走の暴走→confirm-list.md「止まる理由が無い限り進め、不明点はここに溜める」＝判断の非同期化

## 設計含意（今後へ）

1. この体制の再現条件は「①スペックの正準順序②機械ゲート高回転③ドメイン専門家による実物受け入れ」の3点セット。③が欠ける分野（オーナーが出力の良し悪しを診断できない分野）では同じ品質は出ない見込み。
2. revert 1/943は「上から下ろす」順序が手戻りを消していた定量証拠。プロセス変更を検討する際の基準線にする。
3. 生テキスト計数（「監査」5,005回等）は文脈重複を含むため実行回数でなく文化浸透の傍証として読むこと。
