## 0. 結論サマリ

- **推奨モデル＝「分家（ぶんけ）モデル」：参照が既定・変えたくなったら浅い分家・系譜を残す**。
  - 同一の繰り返し（サビ×2）＝**参照の複数配置**（現行 compose_edge がそのまま・実装済）。
  - 「同じだけど少し違う」（2番Aメロ・落ちサビ・転調ラスサビ）＝**浅い分家**：新 section ネタを作り**子ネタは参照で共有**、変えた子だけ fork（copy-on-write）。`variant_of` の relation_edge で系譜を残す。
  - スキーマ変更ゼロ（新テーブル・新カラム不要）。compose_edge の「子は親ごとの辺集合」という既存の粒がそのままオーバーレイ機構として機能する（§3.4 が核心）。
- **俯瞰＝セクションカードの列（フォームストリップ）**。小節精密グリッドは song では持たない（オーナーの直観を支持・§4 で吟味）。カードに必要な最低限＝役割・尺・調（曲基準と違う時だけバッジ）・分家マーク・レイヤ帯・×N。位置(position)は**カード列からの射影**（前置和）として保つ＝下のデータは今のまま拍の実数。
- **つなぎ**＝suggest_form/key_plan/energy_plan は「足場の実体化」で結ぶ：form候補→**空セクションの足場**を並べる／key候補→**セクションの key に落とす**（転調ラスサビは分家として）／energy候補→カードのΔチップ＋生成ノブへ。全部「提案→人が選んで適用」＝哲学と整合。
- **通し再生は既に正しい**（compositeNotes 再帰）。書き出しは **song の多トラックMIDIが1トラックに潰れるバグ級の穴**があり、part 別トラック化で直す。
- **層は今の3層で足りる**。song が form を持つ＝正しい（form は compose_edge の順序＋role タグの射影であり、新しい実体は不要）。
- スライス＝S1 フォームストリップ＋書き出し是正 → S2 分家 → S3 つなぎ結線＋遷移試聴 → S4 パーキング（テンポ変化/WAV/ミックス）。

## 1. 「曲にする」の再定義（作曲の実態から）

セクションが曲になるとは、次の5つの作業が回ることである。順序は行き来する（直列でない）。

1. **並べる（form）**：役割の列を決める（Intro→A→B→サビ→…）。ポップスの支配形式は verse–chorus 系で、セクションは8小節基準・16拡張（実測は [song-form-statistics](2026-07-14-song-form-statistics.md) §1）。
2. **繰り返す・戻す**：曲の記憶性は反復が担う。POP909 実測で「**近くでは変える・遠くでは戻す**」＝リテラル反復は隣接0.4–1.4%だがセクション回帰14–16%（[motif-transform-stats](2026-07-14-motif-transform-stats.md)）。つまり**サビは戻ってくるもの**であり、曲フォーム層の一級操作。
3. **変奏する**：「同じだけど少し違う」が常態。J-pop 伝統形＝1番(A–A'–B–サビ)→間奏→**2番(短縮)**→Cメロ→**落ちサビ**→**大サビ**→アウトロ（同 §0）。2番Aメロ（歌詞が違う＝メロのリズムが微差）・落ちサビ（レイヤを抜く）・転調ラスサビ（半音上げ/短3度上げ＝suggest_key_plan の二大頻出）。**変奏はコピーでも新作でもなく「同一性を保った差分」**——ここをモデルが表現できるかが本設計の主戦場。
4. **つなぐ**：境界の設計。転調準備（pivot/V7）・フィル・ブリッジ・エネルギーのΔ（知覚エネルギーは絶対値でなく前セクション比＝[energy-arc](2026-07-14-energy-arc-arrangement.md) §0）。縫い目は「前セクション末→次セクション頭」を耳で確認したい（design #20 S6 縫い目E裁定＝backlog済）。
5. **通して確認し、持ち出す**：頭から鳴らして「今どこ」が分かる。仕上げは ABILITY なので **MIDI で正しく持ち出せる**ことが出口（requirements「DAWとのやりとり」）。

**この道具の役割**（哲学との整合）：上記のうち機械が担うのは「候補と足場」＝form/key/energy の候補提示・変奏の複製作業・つなぎの準備和音候補。**決めるのは人・仕上げは ABILITY**。アレンジ層は「曲の設計図を素早く形にして俯瞰し、通しで聴ける足場」であって精密DAWミニではない（requirements L84「ゼロから生む→判断して方向づける、に変える」）。

### 2.2 穴（実測ベース・影響順）

1. **変奏の仕組みが無い＋参照の共有が不可視**：同じ section を2回置いて片方を編集すると**両方変わる**（子ネタは実体共有）。UI に「これは2箇所で使われている」表示がなく、fork の導線も無い。copy_neta は deep copy＝「同じ」の情報が消える。requirements L149「複製して最後だけ少し展開を変えて」に応える経路が無い。
2. **SONG_MAX_BARS=64 は一曲に足りない**：3:30/120BPM/4/4 ≈ **105小節**。standard 尺（suggest_form の言う≈3:30）が物理的に入らない。
3. **song の多トラックMIDI書き出しが1トラックに潰れる**：laneTracks()（SectionEditor L367-377）は**レーン単位**でトラックを作るが、song のレーンは [section] の1本＝全パートが "Section" 1トラックに混ざる（program も trackProgramOf の多数決で1つに潰れる）。compositeNotes は各ノートに `part`（melody/chord/bass/drums/counter）を付与済みなのに未使用。**ABILITY への出口が song で壊れている**＝オーナーの本番動線に対しバグ級。
4. **計画 verb が孤島**：suggest_form/key_plan/energy_plan は chat から呼べるが、結果を song に「落とす」対象物（役割列の足場・セクション key・エナジー表示）が UI に無い（design L1376 も「web は見送り＝セクション列UIが現状無い」と自認）。
5. **カード/タイムラインに曲情報が無い**：song タイムラインのブロックは MiniRoll 概形とタイトルだけ。役割・調・尺(秒)・エナジー・「どれが同じでどれが分家か」が見えない＝設計図として読めない。
6. **縫い目の耳が無い**：遷移試聴（前セクション末→次頭）は backlog 済み未実装（backlog L39）。
7. 小さい穴：song の feel は最初に見つかった子の content.feel（sectionFeel）＝section の子はレーン設定 content なので実質効かない／テンポ・拍子は曲全体で1つ（transport が単一クロック）。

## 3. モデル比較：セクションはどう曲になるか（問1）

### 3.0 変奏の実態＝要求仕様

作曲の頻度感（§1 と実測研究から）：
- **完全反復**（サビ2回目・Aメロのバッキング使い回し）＝最頻。曲の過半はこれ。
- **調だけ違う**（転調ラスサビ）＝定番2型の1つ（suggest_key_plan が「サビ短3度上げ/最終サビ半音上げ」を重み上位に持つ）。
- **レイヤだけ違う**（落ちサビ＝ドラム/ベースを抜く・2Aで管弦を足す）＝エナジー設計の主手段（energy-arc D3：知覚エネルギーはレイヤ抜き差しが担う）。
- **中身が少し違う**（2番Aメロのメロ微差・フェイク・「最後だけ変える」）＝メロ/コード子の一部差し替え。
- どの変奏も「**元と同じであること**」が意味を持つ：元のコードを直したら 2A にも効いてほしい（同じ進行なのだから）。一方「2A のメロだけ変えたのに 1A も変わった」は事故。**共有と分離の境界を作者が握れること**が要件。

### 3.1 方式A：純参照＋辺オーバーレイ

配置は今のまま id 参照。変奏は **compose_edge（または側テーブル）に per-配置の差分**（keyShift・ミュートレーン・ノブ）を持たせる。
- 利点：ネタ一覧が増えない。「同じものの見た目違い」が1実体に閉じる。転調ラスサビ・落ちサビは JSON 1個で表現でき、DAW の「クリップ＋再生時プロパティ」に近い。
- 欠点：**中身の変奏（2番Aメロ・フェイク）が表現できない**——notes の diff を辺 JSON に持つのは重装備（undo/比較/エディタ全経路が差分適用を知る必要）で、この土台の粒（content は自由形 JSON・エディタは content 直編集）と喧嘩する。結局「中身が違う」は別ネタに逃げるので**二重機構**になる。compositeNotes・エディタ・書き出し全部に overlay 適用層が入る＝改修範囲が広い。スキーマも増える。
- 判定：**過半の変奏（中身差分）を救えないのに機構コストが最大**。捨てる。

### 3.2 方式B：実体コピー（copy_neta 流儀＝クリップ複製）

繰り返しも変奏も deep copy。DAW のクリップ複製と同じメンタルモデル。
- 利点：**エイリアシング事故ゼロ**（編集は必ずその1箇所だけに効く）。実装ほぼゼロ（copy_neta 済）。分かりやすい。
- 欠点：(a) サビ×3 のコード直しが3回作業＝「同じ」が消えるので**曲を通した推敲コストが増える**（スケッチ道具はまさに推敲の道具）。(b) ネタ帳が複製で汚れる（同名 section が並ぶ・検索/連想の母集団も濁る）。(c) form の読み＝「この曲はサビが3回」が データから消える（3つの別ネタ）→ role 統計・key_plan 適用・俯瞰の×N表示すべてが弱る。(d) 哲学と不整合：この道具は「ネタがつながる網」（relation/compose の網）であって、コピー農場ではない。
- 判定：**「同じ」を失う代償が推敲・俯瞰・計画結線の全部に波及**する。既定にはしない。ただし「完全に切り離したい」ユースケース（別の曲に持って行って好きに崩す）には正解＝copy_neta は残す。

### 3.3 方式C：分家モデル（参照既定＋浅い fork＋系譜）＝推奨

- **繰り返し＝参照の複数配置**（現行のまま。×N は同一 child_id の連続辺の束として表示）。
- **変奏＝浅い分家（vary）**：新しい section ネタを作り、**compose_edge は元と同じ子を同じ position で参照**する（deep copy しない）。frame（key/mode/bars）と role タグと title（「サビ（ラスサビ+1）」等）は分家側で自由。元との間に `relation_edge(variant_of)` を張る。
  - 転調ラスサビ＝分家して key を +1 → compositeNotes L637-641 が**既に**子 section の key で再帰合成する＝**エンジン無改修**。
  - 落ちサビ＝分家してドラム/ベースの**辺を外す**（remove_child は分家の辺集合にしか効かない＝元は無傷）。
  - 2番Aメロ＝分家して、変えたいメロ子だけ **fork（copy-on-write）**＝メロを複製して差し替え。コード/ドラム/ベースは共有のまま＝**元の進行を直せば 1A にも 2A にも効く**。
- **copy-on-write の安全弁（必須）**：共有されている子（親が2以上 or 同親2配置以上）を編集しようとしたら「n箇所で使われています→ 全部に効かせる／この曲だけ変える（分家）」を選ばせる。共有バッジをブロック/カードに常時表示。
- 利点：(a) 変奏の全パターン（調・レイヤ・中身）を**単一の機構**で表現。(b) **スキーマ変更ゼロ**——compose_edge の「子は親ごとの辺集合」という既存の粒が、そのままセクション粒度のオーバーレイとして機能する（方式Aが新設しようとした差分表を、DAG が既に持っている）。(c) 「同じ」が保存される：共有子は1実体・分家は variant_of で系譜が辿れる→ 俯瞰の A/A′ 表示・role 統計・「サビを直せば全サビに効く」推敲が全部成立。(d) 実測とも整合：変奏の主要通貨は「ゆるい変奏60-73%・リズム保存・原型回帰」（motif-transform-stats）＝「大部分共有・一部差分」がデータの実態。
- 欠点と手当：(a) エイリアシング事故リスク → copy-on-write プロンプト＋共有バッジで潰す（これが無い分家モデルは出荷しない）。(b) ネタ一覧に分家が並ぶ → variant は一覧で親に畳む（neta_order/一覧の表示課題・S2で最小は「タイトル規約＋バッジ」で可）。(c) 「浅い fork」verb が新規 → core.copyNeta の再帰を depth=0 にした 20行級＋relation 1本＝小さい。
- **段階実装が自然**：S1（参照のまま俯瞰整備）→ S2（vary verb＋copy-on-write）→ 中身 fork は既存 copy_neta(子単体) の流用。

### 3.4 裁定

**方式C（分家モデル）を推奨**。決め手は2つ：
1. **作曲の実態**：変奏は「調・レイヤ・中身」の3種が全部頻出で、1機構で覆えるのは C だけ（A は中身を覆えず、B は「同じ」を覆えない）。
2. **この土台の粒**：compose_edge の親別辺集合＋ネタの自由形 content＋variant_of に使える relation_edge＋key-aware 再帰合成——**全部品が既製**。C は新機構でなく「既存の粒の使い方の宣言＋安全弁UI」に近い。
捨てたもの：A＝中身変奏に届かないのに機構コスト最大。B＝既定にはしないが**逃げ道として温存**（「完全に切り離す」ボタン＝copy_neta。分家との使い分けをUI文言で明示：「同じものとして育てる＝分家／別物にする＝複製」）。

## 4. 俯瞰の形と粒度（問2）

### 4.1 粗い設計図（カード列）で足りるか——吟味

オーナーの傾き＝「セクションカードを並べる・小節精密グリッドは持たない」。反証を試みる：曲にするのに小節精度が要る操作は何か。
- フィル・ブリッジの中身 → **section の中**の仕事（セクションエディタに既にある）。
- 弱起・食い（次セクションへの pickup）→ メロ側の負 start が compositeNotes の position オフセットでそのまま前セクション末尾に食い込む＝**データは既に表現できる**。曲画面で見えなくてよい（耳で確認＝遷移試聴）。
- 尺の半端（8+8+8+9 の着地引き延ばし等・form-statistics §1-B）→ section の bars を変える＝カードの尺表示で足りる。
- セクション間の隙間 → 音楽的意味が薄い（無音を挟むなら空 section を置くほうが正直）。
- **結論：反証不成立。曲レベルで小節グリッドが必要な操作は無い**。必要なのは「順序・繰り返し・差し替え・境界」であり全部カード列の操作。現行の song 小節グリッドは 105小節をモバイル幅に敷けない（1セル数px）ので、粗いカード列は妥協でなく**正しい粒度**。

### 4.2 フォームストリップ（推奨UI）

- **形**：横（モバイルは縦）1列のカード列。`Intro│A│B│サビ│A′│B│サビ│C│落ちサビ│サビ+1│Out` のような帯。position は廃止せず**カード順からの射影**（前置和で再計算して place/remove）＝データ契約は不変・グリッド版と同じ compose_edge を読む。
- **操作**：ドラッグ並べ替え／挿入（＋ボタン→ピッカー：既存section・新規・**form足場から**）／削除／×N（同一 child の連続反復は1カードに畳んで「×2」バッジ・タップで展開）／カードタップ=潜る（現行踏襲）。
- **カードの最低限情報**（"曲にする"に要る5点・§1 から導出）：
  1. **役割**（role タグ→色とラベル。無ければ無地）
  2. **尺**（小節数＋概算秒。曲ヘッダに合計）
  3. **調**（曲の key と違う時だけ「+1」「短3上」バッジ＝転調が見える）
  4. **同一性**（同じ子の反復＝同色系・分家＝「A′」バッジ＋元への線 or ツールチップ）
  5. **レイヤ帯**（SectionMini 流用の縮小帯＝どのパートが鳴るか→落ちサビが見た目で分かる）
- **エナジー**（あれば）：カード上端に Δ チップ（↑↑/↓等）。energy_plan 適用時のみ。
- 曲ヘッダ：合計尺（bars×BPB÷tempo）・key/mode・tempo・stage/next_action（SongStatus 統合）。
- 既存の song 小節グリッドは**フォームストリップで置換**（song kind のみ。section のグリッドは不変）。SONG_MAX_BARS は射影化に伴い実質撤廃（描画は尺に依存しない）。

## 5. つなぎ・通し再生・書き出し・層整合（問3-5）

### 5.1 つなぎ＝計画 verb の結線（提案→人が選ぶ）

3 verb とも「提案のみ」の思想で実装済み。結線は**候補を足場として実体化する適用ボタン**を足すだけ：
- **suggest_form → 足場**：候補（役割列＋小節数）を選ぶと、**空の section ネタ（role タグ＋bars 付き）をストリップに並べる**。空カードは「ここに何か要る」の見える化＝ピッカーで既存 section を差し替え/新規作成/生成へ。design L1163 の「役割列を place_child へ落とすのは人/上位の判断」の"人の側の受け皿"。
- **suggest_key_plan → セクション key**：ストリップの役割列を roles 引数へ（prev/next 導出と同じ position 順）。候補を選ぶと各セクションの key/mode を更新——**転調するセクションが共有子なら自動で分家を提案**（「ラスサビだけ+1＝分家を作ります」）。境界の準備和音は gen_chords の transition オプション（WP-C2 実装済）で**分家したコード子の末尾差し替え候補**として出す。
- **suggest_energy_plan → チップ＋ノブ**：候補を選ぶとカードにΔチップ表示＋各 section の生成時 frame.section.energy へ渡る既定値に。レイヤ操作の提案（「落ちサビ：ドラム抜き」）は分家＋辺外しのワンタップ提案として出せる。
- **遷移試聴（縫い目E・backlog昇格）**：カード境界タップ→前セクション末2小節＋次セクション頭2小節を連結再生。compositeNotes の部分窓で実装可（新機構不要）。フィル/ブリッジは「境界に interlude section を挿す」＋gen_drums のフィルで賄う（新 kind 不要）。

### 5.2 通し再生・書き出し（問4）

- **通し再生**：エンジンは compositeNotes 再帰で既に正しい（§2.1）。足すのは「今どこ」＝プレイヘッド位置→現在カードのハイライト＋カード内進捗。ストリップは position 射影を持つので変換は前置和の逆引きだけ。仮歌（メロ子の sing）も compositeNotes 経路なので song 通しで原理的に動く（音源レンダのキャッシュ粒度だけ確認）。
- **MIDI 書き出し（是正必須）**：song の多トラックは**レーン単位でなく part 単位**に：compositeNotes が各ノートへ付与済みの `part`（melody/chord/bass/drums/counter）でトラック分割し、program はノート単位のものを尊重。→ ABILITY に「曲全体・パート別」で渡る＝本番動線が通る。テンポ/拍子は曲単一（現行 transport と同じ制約＝正直にそのまま）。
- **WAV（将来）**：既存サンプラー経路のオフラインレンダで足りる見込み。本設計の必須条件にはしない（backlog）。

### 5.3 層整合（問5）

- **project=器 / song=曲 / section＝3層で足りる**。form という第4の実体は**作らない**：form は「song の compose_edge の順序＋各 section の role タグ」の射影であり、二重管理（form 表 vs 実配置の乖離）は SDD の腐り方そのもの。suggest_form の候補も適用した瞬間に足場 section＋辺になる＝正準は常に1つ。
- **song が form を持つべきか→持つ（今の形で既に持っている）**。song overlay（stage/next_action/loop）はそのまま。エナジープラン等を永続したくなったら song overlay に JSON 1列の余地がある（loop の前例）が、S3 までは不要（提案は揮発・確定はノブ/分家として実体に落ちる）。
- **song=sections-only（design L589）を維持**。「曲を通すパッド1枚」等の要求が出たら、通しパートを1本の長い section として置く逃げ道が現行でも成立（レーンは1本だが position は任意実数）＝再検討はそれが辛くなってから。
- 組曲＝project に song 複数（design L492）＝本設計は不変。
