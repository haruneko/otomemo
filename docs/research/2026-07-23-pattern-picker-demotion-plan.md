# パターン選択帯の「格下げ」計画（2026-07-23・オーナー方針・実装は次セッション・計画のみ）

**この doc の役割**＝オーナー指示「**『パターンを選ぶ』をもっと格下げして『ライブラリからインポート』みたいな二次アクションにする**」を、
文脈ゼロの担当が実行できる形に起こす計画書。正典の連なり＝本doc →（裁定後に）`design.md` へ下ろす → 実装。
**本 doc の時点では実装しない・design.md へも下ろさない**（オーナー確認待ち）。

## 0. オーナーの言葉（方針の原文・2026-07-23）
> 「パターンを選ぶ」をもっと格下げして「ライブラリからインポート」みたいな二次アクションにする。

読み解き：
- **作編の主役はグリッド**（右手/左手・ベース段・ドラム段を人が置く）。パターン流用は「主導線」ではなく「必要な時に呼ぶ」補助。
- 現状の「パターンを選ぶ ▸」帯が**設定の直上＝一等地**を恒久占有しているのは**格が重すぎる**。
- 中身は Task2/L3（design.md「Task2/L3」節・2026-07-23）で既に**ライブラリ検索**（`listNeta({scope:"library", tags:["genre:…"]})`）に差し替え済み＝
  「引っぱってくる」動作の実体は既に**インポート**。ならば**見せ方・配置もインポート相当の控えめな二次アクション**へ合わせる。

## 1. 現状（この計画の開始地点）
### 1-1. 器と配置
- 器＝`apps/web/src/components/PatternPickerBar.tsx`（折りたたみ帯。トグル→ジャンルchip→「候補を出す」→候補カード[▶試聴／適用]）。**契約 `PatternCand`（key/name/scene/audition/apply）は L3 で不変**。
- 3つの単体エディタが**エディタ最上部**（設定の直上）に常時レンダ：
  - `ChordPatternEditor.tsx:300`（`showPicker` true 時・chips=`COMP_GENRE_CHIPS`）。直下に BarsControl→長さ→両手グリッド。
  - `BassStepEditor.tsx:197`（非 compound meter 時・chips=`BASS_GENRE_CHIPS`）。直下に BarsControl→段グリッド。
  - `RhythmEditor.tsx:234`（chips=`DRUM_GENRE_CHIPS`）。直下に rhythm-toolbar→グリッド。
- Section 側は別UI＝`useMelodyGen.tsx`（`genPart`・「いじる▾」内のノブ＋候補トレイ）。コード楽器の候補は `COMP_GENRE_CHIPS`→variety でトレイに積む。**PatternPickerBar は使っていない**（別導線）。

### 1-2. 問題（格の不一致）
1. **一等地の恒久占有**：帯は既定閉でも「パターンを選ぶ ▸」の見出し行がグリッド設定より上に**常に**居座る＝視線の最初がグリッドでなく流用導線に向く。主役（人が置くグリッド）と補助（流用）の**上下が逆**。
2. **開閉の重さ**：折りたたみ帯は「開く→chip選ぶ→候補を出す→カード→適用」と段が深い。二次アクションにしては専有面積が大きい。
3. **文言のミスリード**：「パターンを選ぶ」は主動作のような能動命令形。実体は「ライブラリから1個借りて今のグリッドに流し込む」＝インポート。名が実体より重い。
4. **一貫性の穴**：単体エディタ3種は帯・Section 側はトレイと**導線が二系統**。格下げを機に「ライブラリから引く」の見え方を揃える好機。

## 2. 格下げ後の姿（案・ASCIIワイヤー）
共通の下敷き＝**器（PatternPickerBar のロジック：chip→候補取得→カード→▶試聴／適用）は残す**。変えるのは**入口の格（配置・初期の目立ち）と文言**だけ＝L3 の候補取得/適用は完全不変（bit 一致の担保／§5）。

### 案A：小リンク化＝「グリッド脇の二次アクション」（推奨）
帯の見出し行を、**設定行の右端に寄せた控えめなテキストリンク/小ボタン**へ降格。押すと**その場で下に候補パネルが開く**（＝現行 `open` の body をポップオーバー/インライン展開で流用）。閉時はリンク1個ぶんの高さだけ。

```
┌─ コード楽器エディタ ───────────────────────────┐
│  小節 [− 2 +]   長さ [♪▾]        ⤓ ライブラリから読み込む │  ← 設定行の右端に小リンク（二次）
│  ┌───────────────────────────────────────────┐  │
│  │ 右手  ■ ・ ■ ・ | ■ ・ ・ ・                 │  │  ← 主役＝グリッドが最上段の主
│  │────────────────────────────────────────── │  │
│  │ 左手  R ・ 5 ・ | R ・ ・ ・                 │  │
│  └───────────────────────────────────────────┘  │
└───────────────────────────────────────────────┘

押下時（インライン or ポップオーバー展開）:
      ⤓ ライブラリから読み込む  ▾   （いま：CP-POP1）
      [おまかせ][バラード][ロック][シティポップ][4つ打ち][フォーク]
      ┌ CP-POP1  verse  ▶  読み込む ┐  ┌ CP-ROCK2 …
```
- 長所：主役（グリッド）が最上段に来る／二次アクションらしい軽さ／器の中身を丸ごと再利用＝改修が浅い／「いま：<型>」も維持可。
- 短所：設定行が横に混む（モバイル幅で折返し要検討）／ポップオーバー化するとはみ出しクランプ（`5c7636d`/`18351d0` で既出のビューポート処理）を流用する必要。

### 案B：メニュー項目化＝「いじる/設定メニューへ収容」
帯を独立UIから外し、既存の副アクションメニュー（コード楽器なら管弦ゲート等の設定群、Section の「いじる▾」に相当）に**「ライブラリから読み込む…」1項目**として沈める。選ぶとモーダル/シートで候補パネル。

```
┌─ コード楽器エディタ ───────────────────────────┐
│  小節 [− 2 +]  長さ [♪▾]                    [ ⋯ ] │ ← ⋯メニューに沈める
│  （グリッド）                                     │
└───────────────────────────────────────────────┘
    [ ⋯ ] を開くと:  ┌───────────────────────┐
                     │ ⤓ ライブラリから読み込む… │
                     │ 楽器を替える…            │
                     └───────────────────────┘
```
- 長所：一等地から完全に消える＝最も強い格下げ／3エディタ＋Section で「メニュー内の1項目」に**構造を揃えやすい**（一貫性で最良）。
- 短所：発見性が最も低い（メニューを開くまで存在が見えない）／各エディタに共通の「⋯メニュー」器が現状無い＝**新規に器を1つ用意する改修が要る**（案Aより重い）／Section の「いじる▾」とコード/ドラム単体の設定群は今バラバラで、統一に別作業が要る。

### 案C：空グリッド時のみ出るゴーストCTA
グリッドが**空（未入力）のときだけ**、グリッド面の中に薄い誘導「ライブラリから読み込む / 自分で置く」を出す。1つでも置いたら消える。設定行には常設しない（or 案Aの小リンクと併用）。

```
空のとき:
  ┌───────────────────────────────────────────┐
  │  （右手・左手グリッド、まだ空）               │
  │        ⤓ ライブラリから読み込む               │  ← 空の時だけ出るゴースト
  │        またはタップして自分で置く              │
  └───────────────────────────────────────────┘
一つでも置くと → ゴースト消滅（設定行に小リンクが残る＝案Aと併用が自然）
```
- 長所：「最初の一歩」を最も自然に提示（白紙の不安を消す）＝新規ネタでの発見性が高い／入力済みでは完全に消える＝作業中は邪魔しない。
- 短所：単独では入力後に導線が消え「後から別パターンに差し替えたい」に応えられない＝**案Aとの併用前提**／空判定を3エディタそれぞれで持つ（コード=両手空・ベース=段空・ドラム=hit空）小配線が要る。

### 推奨＝**案A（設定行の二次リンク化）を土台に、案C（空グリッドのゴースト）を任意併用**
理由：
- 器（PatternPickerBar のロジック）を**丸ごと再利用**でき、L3 の候補取得/適用に一切触れずに「格」だけ落とせる＝最小侵襲・bit 一致が構造的に守れる。
- 案B（メニュー化）は格下げとしては最も強いが、3エディタ＋Section に共通の「⋯メニュー」器が無く**新規UI足場**が要る＝今回の主眼（格の是正）に対して過剰投資。将来の一貫性統合アーク（M5・トップ棚卸し系）で案Bへ寄せる余地は残す。
- 案Cは「白紙の一歩」を埋める効果が大きいが、単独では差し替え導線が消える。**A（常設の軽い入口）＋C（空だけの誘導）**が発見性と非侵襲を両立。
- Section 側（`useMelodyGen`）は既にメニュー内トレイ＝実質「二次」の位置にある。今回は**単体エディタ3種の格を Section 並みに落として揃える**のが本命（§3）。

## 3. 一貫性（3エディタ＋Section をどう揃えるか）
- **器は残す・呼び出し口だけ変える**：`PatternPickerBar` の内部（chip→候補→カード→▶／適用）はそのまま。変えるのは (i) トグル見出しの見た目（帯→小リンク）(ii) 配置（最上部→設定行の端 or ⋯メニュー）(iii) 文言（§4）。3エディタは同じ器を同じ格で呼ぶ＝一貫。
  - 実装案：`PatternPickerBar` に **`variant`（`"bar"`（現行）／`"link"`（案A小リンク）／`"ghost"`（案C））** props を1つ足し、見出しレンダだけ分岐。body（chip/候補カード）は共有。**既定 `variant` は呼び出し側で指定・現行 `"bar"` を渡せば bit 一致**（退避経路を残す）。
  - 3エディタの呼び出し（`ChordPatternEditor.tsx:300`／`BassStepEditor.tsx:197`／`RhythmEditor.tsx:234`）は `variant="link"` を渡すだけに揃える。位置は「設定行（BarsControl の行）」へ移動。
- **Section 側（`useMelodyGen`）**：既に「いじる▾」内トレイ＝二次。今回は**文言だけ揃える**（「パターン」→「ライブラリから読み込む」系／§4）＝格の移動は不要。将来 Section のコード楽器トレイと単体エディタのライブラリ検索を**同じ検索口**に寄せる案は L3 続き（pattern-library-arc L3/L5）で扱う＝本 doc の範囲外だが、文言を今そろえておけば合流が楽。
- **compound meter / 管弦ゲート**：現行の非表示条件（ベース 6/8 非表示・`showPicker=false` で帯ごと消す）は**格下げ後も同条件で入口を出さない**＝挙動不変。

## 4. 文言（造語を避け一般語・メモリ「造語でなく一般的な語で」）
- トグル/リンクの主文言：**「ライブラリから読み込む」**（推奨）。
  - 対抗案：「ライブラリから選ぶ」「パターンを読み込む」「型を読み込む」。オーナー原文の「インポート」に忠実にするなら「ライブラリからインポート」も可だが、**一般語＝「読み込む」**を第一候補（メモリ方針）。アイコンは `⤓`/取り込み系。
- 「候補を出す」ボタン：現行文言のままで可。ライブラリ検索であることを明示するなら「ライブラリを探す」。
- 「適用」ボタン：**「読み込む」**へ寄せると入口文言と一貫（＝この型を今のグリッドに読み込む）。現行「適用」でも意味は通る＝**軽微・オーナー裁定**。
- 「いま：<型>」表示：維持（現在型の可視化＝選び直しの助け）。文言変更なし。
- Section 側トレイの見出しも同語彙へ寄せる（存在すれば）。

## 5. スライスと TDD の当て所（bit 一致の担保）
**大原則＝候補取得（L3 のライブラリ検索）・適用（content 置換）・試聴（notesForContent→再生）のロジックは一切触らない。変えるのは見せ方/配置/文言のみ。** ゆえに「グリッド編集の出力 content が従来と bit 一致」は構造的に保たれる。

| # | スライス | 中身 | TDD の当て所 |
|---|---|---|---|
| S1 | `variant` 導入 | `PatternPickerBar` に `variant`（bar/link/ghost）props。見出しレンダのみ分岐・body 共有。既定は呼び出し側指定 | 単体：`variant="bar"` で現行 DOM/aria が不変（`pattern-picker-toggle` 等の aria-label 維持）。`variant="link"` で小リンク role/文言が出る |
| S2 | 3エディタの格下げ配置 | 3呼び出しを `variant="link"`＋設定行へ移動。文言差し替え | 各エディタの**適用フロー e2e/コンポーネントテスト**＝chip→候補→適用で書かれる content が**従来と同一**（適用ロジック不変の回帰確認）。配置移動でグリッド編集イベントが壊れない（hit 追加/消去の既存テスト緑） |
| S3 | 文言統一 | 「パターンを選ぶ」→「ライブラリから読み込む」・適用→読み込む（裁定次第）・Section 側語彙合わせ | 文言のスナップショット/aria-label 更新。**aria-label をテストが握っている箇所は同時更新**（下記地雷） |
| S4（任意） | 空グリッドのゴースト（案C） | 3エディタで空判定→ゴーストCTA。1件で消える | 空/非空でゴーストの出現/消滅。空判定（両手/段/hit）の単体 |
| 受入 | 実機＋耳 | モバイル幅で設定行が崩れない（Playwright 実機・`reference-playwright-live-ui-check`）。ポップオーバー化ならはみ出しクランプ流用確認。適用後の音が従来同一（耳） | M5（UI一貫性 Fable 監査）へ合流 |

**地雷（着手前に知る）**：
- `PatternPickerBar` の **aria-label（`pattern-picker` / `pattern-picker-toggle` / `pattern-fetch` / `pattern-card-N` / `pattern-apply-N` 等）をテスト・E2E が握っている**可能性大。文言/構造変更時は**同時に**テスト側を更新（変更は上位から：テスト先行で赤→緑）。
- ポップオーバー/インライン展開に変える場合、既出の**ビューポートはみ出しクランプ**（`18351d0`「ベース『その他』ポップオーバー…下端クランプ＋上フリップ」）を流用・重複実装しない。
- 設定行（BarsControl の行）へ小リンクを同居させるとき、**モバイル 390 幅で折返し/はみ出し**が出やすい＝Playwright 実機幅で確認（メモリ `reference-playwright-live-ui-check`）。
- 管弦（`showPicker=false`）・compound（ベース 6/8）の**非表示条件を格下げ後も維持**＝入口の格が変わっても「出さない場面」は不変。

## 6. 未決（オーナー裁定が要る点＝着手前に確認）
1. **推奨案でよいか**＝案A（設定行の二次リンク）＋案C（空グリッドのゴースト）併用。それとも案B（⋯メニュー沈め＝最も強い格下げだが新規メニュー器が要る）まで踏み込むか。
2. **主文言**＝「ライブラリから読み込む」で確定か（対抗：「ライブラリからインポート」原文忠実／「ライブラリから選ぶ」）。
3. **「適用」ボタンの文言**＝「読み込む」へ寄せるか、現行「適用」を残すか。
4. **展開の形**＝小リンク押下時に (i) その場でインライン下展開か (ii) ポップオーバー/ボトムシートか（モバイルの当たり判定と、はみ出しクランプ流用の要否が変わる）。
5. **案C を入れるか**＝空グリッドのゴースト誘導まで作るか（発見性↑だが 3エディタに空判定の小配線）。
6. **Section 側の扱い**＝今回は文言統一だけに留めるか（推奨）、L3 続きの「同一検索口への統合」までこのアークで踏むか。
7. **実施タイミング**＝pattern-library-arc（`2026-07-22-pattern-library-arc-plan.md`）の L4 量産・M5 監査との順序（UI が動くので M5 は本件の後が二度手間を避ける）。
