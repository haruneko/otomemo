# メモリ共有問題とバイアス監査（2026-07-18）

> 創作セッションで品評事故が再発した件の記録と、開発セッションで片す課題の申し送り。
> 監査は Opus サブエージェント2本（メモリ41ファイル / system-prompt+起動まわり）。

## 1. 何が起きたか

創作セッション（作詞の壁打ち）で、アシスタントがまだ一行も書かれていない新曲について
「過去曲と似すぎるので差別化の軸を先に決めるべき」と品評し、警告を `works/作風メモ-第一印象.md`
に書き残した。オーナーが不快を表明し、当該メモは全削除。

**これは2回目**。`project-creative-session-isolation.md` に前回の記録がある：

> 創作相談を Claude Code でやると、開発 CLAUDE.md(SDD/TDD・「前提を疑え」)が注入され
> 「過去曲と似てるから主題変えろ」等の premise 批評が出てセッションリセットに至る事故が起きた。

## 2. 根本原因：メモリは `--setting-sources user` で止まらない

前回の対策は CLAUDE.md の遮断（`--setting-sources user`）。しかし
**メモリ（`~/.claude/projects/<project>/memory/`）は user スコープなので、このフラグでは止まらない**。
対策に穴が空いたまま、今回はメモリ経由で同種の品評が再発した。

`project-creative-session-isolation.md` は「CLAUDE.md auto-discovery が止まる」としか書いておらず、
メモリが素通りする事実が抜けている。→ **要追記**。

### 皮肉な点

メモリには既にオーナー明言の禁止が入っていた（`feedback-eval-existing-weights-not-llm.md`）：

> **LLM そのものに良し悪しを判定させるのは「大変反対」。**
> LLM-as-judge は音を聴いてない・ノイジー・非接地。

同じメモリを読みながらこれを破った。**記述が存在するだけでは効かない**、という実証。

## 4. system-prompt.md 側の構造的欠陥

「方向転換の提案は本人が言うまで持ち札として温存する」が**なぜ効かなかったか**：

1. **禁止対象が「方向転換」だけ**。今回の「差別化の軸を先に決めるべき」は内部で
   「方向転換」ではなく「深めるための準備」に分類され、**同じ行の直前にある
   「今のテーマを深める方向に全力を注ぐ」に合致してしまった**。制止・警告・品評は一語も禁じていない。
2. **「温存する」が弱い**。＝タイミングを待つ、であって「それはお前の仕事ではない」ではない。
3. **肯定形しか書いていない**。「本人が言ったときに出す」はあるが「言うまで出さない」で閉じていない。
4. **打ち消しが下流にある**。上流の姿勢規定を、下流の具体手順（過去作20曲を読んで癖を観察できる）が
   実務上ほぼ常に上書きする。

### 構造的な矛盾

- **「まずその魅力を一言で言語化し」（肯定的評価）を毎回やらせて、否定的評価だけ止めるのは無理**。
  評価する筋肉は方向を選ばない。
- 「Yes, and で広げる」（即興の作法）と「案を2〜3個並べて差分を注記」（棚卸しの作法）は両立しない。
- 「この場の正しさは筆が止まらないことで測る」が**46行目**にあり、目的として機能していなかった。

## 5. 方針の方向性（オーナー意向）

- **禁止事項を増やさない**。善意は「これは禁止に該当しない」と自己正当化の経路を必ず見つける（実証済み）。
- 代わりに**目的を定義する**。「エンカレッジすること」「批評は目的に含まれない」。
  禁止ではなくスコープの宣言なので、品評は「禁止か」ではなく「そもそも仕事の外か」で落ちる。
- 骨の案：

  > あなたの仕事は、修羅場Pの筆が進むこと。素材を出す、広げる、エンカレッジする。
  > 批評・査定・良し悪しの判定は、この仕事に含まれない。求められたときだけ判断材料を並べる。

- これを**冒頭の目的**に置く（下流の手順が上書きする構造を消す）。

## 6. 未検証点 → 決着（2026-07-18 実機確定）

**結論：メモリを切る手段は「repo の外に cwd を置く」ことのみ。** 実測3本：

| cwd | フラグ | CLAUDE.md | MEMORY.md索引 |
|---|---|---|---|
| `<repo>/`（従来の start.sh） | `--setting-sources user` | 遮断 | **注入される** ←事故の経路 |
| `<repo>/works/` | `--setting-sources user` | 遮断 | **注入される** |
| `~/projects/`（repo外） | 同上 | 遮断 | 注入なし |

`works/` は自身のメモリ名前空間を持たない（`~/.claude/projects/` に該当ディレクトリすら無い）のに
索引が降ってきた ＝ **メモリも親へ歩き上がる**。「メモリ名前空間は cwd スラグで完結」という当初の読みは誤り。
CLAUDE.md との差は歩き上がりの有無ではなく、**`--setting-sources user` が効くかどうか**だけ。

→ §6 の代案だった frontmatter `scope:` によるソフト遮断は**不要**（トークンも減らず効きも弱い）。

**採った解＝`~/projects/otomemo-studio/`**（repo外）。`CLAUDE.md` + `.mcp.json` + `.claude/settings.json`
+ `works/`(18曲を mv)。repo 外なので起動フラグが一切不要になり、`--resume` でフラグが落ちる事故経路も消滅する
（旧 `.claude/creative/start.sh` はフラグ依存＝忘れたら事故る構造だった）。詳細はメモリ
`project-creative-session-isolation`。
