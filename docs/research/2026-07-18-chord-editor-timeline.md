**正典**：`docs/design.md` #26。本 doc は #26 の根拠（Q1/Q2 比較・ピアノロール判定・手数・セクション整合・折り返し採用）を残す。

## 実機で観測した問題（スクショ実測）
現行 ChordEditor は **1コード＝縦カード**（決定コード名の太字 `.chord-sym` ＋ root/三和音/拡張/オルタード/オンベースの `<select>` 行＋✕、下段に長さボタン）を縦積み。星空の唄 Cメロは10コード＝10枚。Playwright で 412px（実機相当・DPR2.6）と 900px を実測：
1. **決定コード名と隣の root select が見分けられない**：`.chord-sym`（枠なし太字 span・bg なし）と隣の root `<select>`（同 bg `#1b1e23`・細枠）が同色・同肉で一続きの塊に見える。さらに `.chord-sym` はモバイルで `width:40px` 固定＝「A#maj7」がはみ出し、隣の select に被って「A#ma」と切れる（`chat.css` の `@media(max-width:480px)`）。
2. **縦に長く冗長**：10枚×約110px＝スクロールが長い。中間挿入が事実上不能（「＋コード」は末尾のみ→間に入れるには以降を打ち直し）。

## Q1 概観：MiniRoll（ボイシングロール）は最適か → いいえ
現行の上部プレビュー MiniRoll は **鳴る音（ボイシング）を `<rect>` で描く**＝音域の輪郭は出るが**コード名・小節線・和声リズムが見えない**＝「進行の地図」でなく「音の影」。同一 Cメロ で4方式比較：

| 方式 | 見える | 隠れる/弱点 |
|---|---|---|
| a. 現状=ボイシングロール | 音域輪郭・和音の厚み | 名前/小節線/和声リズム全滅 |
| b. ロール＋名前/境界オーバーレイ | ボイシング＋名前 | 390pxで2拍コード幅22px＝「A#maj7」入らず・折返し不可 |
| c. 度数レーン（♭VII→I7→IIm7） | 機能・移調不変の骨格 | 実音名が消える＝主表示に遠い。**副表示なら強い** |
| **d. コード名ブロック・タイムライン**（幅∝拍・度数色・折返し） | 名前・長さ・小節線・和声リズム・付点の小節またぎ | ボイシング高低→下に薄ロール帯で補完 |

**結論＝d**。決め手：このツールはコードを名前（root+quality）で保存し思考も名前ベース。現 Cメロ の「Dm7 が付点で3→4小節目をまたぐ」事実は現 UI のどこにも出ないが、ブロックなら一目。度数色は機械的ハブ（hue=度数×30°）で機能分析の曖昧さに依存しない。ボイシング情報は薄帯として下段に残す（捨てない）。

## Q2 入力：縦積みカードの代替 → ブロック＝概観 兼 編集ハンドル
ブロックをタップ→画面下からシート（現行全コントロールそのまま）／端ドラッグ→長さ。縦積み消滅＝「直したいコードを探してスクロール」が「地図から直接つつく」に変わる。次点不採用＝コンパクト表（地図にならない）／チップ行（幅∝拍を捨て和声リズム喪失）。

## ピアノロール入力の判定（オーナーの発案を正直検証）→ 主入力として不採用
1. **操作数で負ける**：B♭maj7＝名前なら2–3タップ、描画は B♭ D F A の綴りを暗算して精密タップ4回。
2. **精度で負ける**：maj7 vs 7 は半音1行＝モバイル約20px、指接地約40px。select なら誤りようがない。
3. **スキーマと逆行**：保存は root+quality（C基準）でボイシングは再生時に自動生成＝描いたボイシングは捨てられるか要スキーマ拡張。音→名前の逆算は **C6=Am7 問題**で root 非一意＝下流（ベース生成/アナリーゼ）がブレる。
4. ロールの唯一の勝ち筋「ボイシングを自分で決めたい」は入力方式でなく**別機能**（ボイシング編集レイヤ）として切る。ロールは表示/確認用が正しい。

## セクションエディタとの整合（オーナー指摘＝同じ部品なら揃えるべき）
発見：**コード進行はすでに SectionEditor 最上段の「コード進行」レーンに紫ブロック「Cメロ」として存在**（実機 C メロ1＝34ブロックで確認）。つまりコード編集画面は**そのレーンをコード単位までズームインした姿**にすればよい。
- **一貫性の在り処＝「ブロックの文法」**：揃えるのは①ブロック外観（`.lane-block`＝ミニロール背景＋左下ラベル＋種別色 `--k-chord`）②挙動（鉛筆/消しゴム、タップ→編集、右端ドラッグ→長さ、＋シーム挿入、空き→追加）。**並べ方（横スクロール/折り返し）は葉の違い**。トークン SSOT＝`apps/web/src/styles/transport-cards.css` L257–475。
- **分岐は容器だけに閉じる**：ブロックの見た目＋文法をレイアウト非依存に factor し、容器を `WrapTimeline`（折返し・コード）／既存スクロール容器（Section）の2つに。後で Section を折返し対応にするのは容器差し替えのみ。

### 挿入・削除（モック検証で確定）
- **削除＝消しゴみモード**（Section と完全同型・新メンタルモデルゼロ）＋シート内✕（従）。選択ブロック常設✕は22px小ブロックで誤タップ源＝不採用。
- **挿入＝境界の「＋」シーム**（編集モードで常設・1タップで直前コード複製→reflow→シート即オープン。当たり判定は右端グリップと非重複の境界右14px）＋末尾空きセル（Section「空きをタップ→置く」の方言）。長押し分割は発見不能＋将来の並べ替えドラッグと衝突で不採用。

## 折り返し採用（オーナー最終判断 2026-07-18）
モックは Section 兄弟性を最大化するため**横スクロール**（`chord-progression-unified.html`）だったが、**コードに限り折り返し（N小節/段・リードシート的）を採る**＝入力の一貫性より**一覧性**を優先。意図的な分岐として #26 に確定。「折り返し可能なエディタは他エディタにもあってよい」は将来の一般化（backlog）。
