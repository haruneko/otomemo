# 研究→実装 方針（Opus委譲前提のワークパッケージ分割・2026-07-14）

**目的**：研究ファンアウト全31本（[2026-07-14-research-fanout-plan](2026-07-14-research-fanout-plan.md)）の成果を、**Opusサブエージェント1体で完結する実装ワークパッケージ（WP）**に分割し、優先順を付ける。オーナー承認後、WP単位で発射する。

## 実装の共通ルール（全WP共通・エージェント指示に焼き込む）

1. **SDD**：design.md の該当#へ先に契約を書く（新ノブ/スキーマ/レンズはdesign先行）
2. **TDD**：赤→緑→リファクタ。数値目標は研究docの実測値を受け入れ条件に
3. **既定=bit一致**：新ノブは全て既定OFF/0で従来出力と完全一致（例外は明示合意時のみ）
4. **思想**：審判にしない＝評価系は「候補の並べ替えレンズ」・生成系は「候補を増やす」。自動で完成させない
5. **耳較正は人間**：機械受け入れ（テスト・実測）とは別に［耳/手］項目を各WPの末尾に列挙、実装後にまとめてオーナーが確認
6. main直・論理単位コミット・全緑push

## 推奨実行順（オーナー判断待ち→第1弾・第2弾は承認済で消化）

```
第1弾: WP-0 → WP-1（基盤・小）＋ WP-M3（レンズ=独立・非破壊で効果が見やすい）
第2弾: WP-M1 → WP-M2（メロ本丸・順序依存）
第3弾: WP-C1 → WP-C3（和声の安い順）＋ WP-M4
以降: オーナーの制作ニーズ順（ゲーム曲書くならWP-X2先行等）
```

**根拠**：メロ最優先（オーナー苦手領域）・WP-M1がクリティカルパス（07-09レビュー結論）・レンズ系は既定bit一致で安全に先行可・打ち込み系（Tier3）は優先度低の既定方針。

## オーナー判断ポイント

1. 第1弾の構成はこれでよいか（WP-M3を先にやると候補選びの体験が先に良くなる）
2. WP-M5（歌詞↔メロ）を別トラックで並走させるか（チャット面の本命だが重い）
