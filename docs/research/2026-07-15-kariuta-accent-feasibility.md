## TL;DR

- **使える（強く推奨）**。pyopenjtalk 0.4.1 は `extract_fullcontext` のフルコンテキストラベルから**モーラ列・アクセント核・アクセント型・アクセント句境界**を一括で取れ、M3 §1.3 が要る「モーラごとの UP/DOWN/FLAT 遷移」へ純ルールで変換できた。追加の重い依存（PyTorch 等）はゼロ、辞書は自動 DL。
- **モーラ分割の的中＝20/20（完全一致）**。歌詞特有の崩れ（長音ー・促音っ・撥音ん・拗音・オノマトペ反復・カタカナ表記・英字 `Love`）を含む自作 20 文で、特殊拍を全て正しく数えた。**最初に外れたのは私の目視カウントの方**（名前=な・ま・え=4、とうきょう=4 等を数え落とした）で、ツールは全問正しかった。
- **アクセント（核位置）の肌感＝「語内の旋律照合に効く部分は高精度、語末の平板/尾高の別だけ弱い」**。孤立単語の型を NHK 引用形と厳密照合すると 6/12 だが、**外れの大半は平板⓪↔尾高②の取り違え＝語内の高低コンター（旋律が乗るモーラ列）は両者で同一**（差は後続助詞が下がるか否かだけ）。よって `analyzeLyricFit` の**旋律方向照合には実質無害**。語義を分ける**同綴異義（箸/橋・雨/飴）は文脈込みで正しく弁別**できた＝最重要ケースは通る。
- **統合は「都度 spawn」で十分**。新プロセス起動→抽出まで**約0.13〜0.23秒/回**（`analyze.py` の Demucs 系＝数十秒とは桁違いに軽い）。常駐や事前計算は不要。方式は `audio-analyze.ts` と同型の PythonCLI spawn を推奨。
- **SSOT 裁定案**：モーラ分割の権威は **pyopenjtalk 側へ寄せる**（アクセント付与に必ずモーラ分割が要るため二重実装を避ける）。ただし既存 `lyric.ts splitMora`（TS）は**かな読みが既知の高速パス**として残してよい＝pyopenjtalk の g2p-かな出力に splitMora を当てた比較で**モーラ数 20/20 一致**を確認済（食い違いゼロ）。**分割の正典は Python、TS はキャッシュ／即応フォールバック**という役割分担が安全。

## 1. 導入手順（再現可能）

システムの `python3 -m venv` は ensurepip 欠落で不完全だったため、**`uv`（既導入）で venv を作成**した。

```bash
# スクラッチパッド配下（_audio_poc は触らない）
uv venv --python 3.12 accent_poc
uv pip install --python accent_poc/bin/python pyopenjtalk   # 0.4.1
# 初回のみ辞書を自動DL（下記）
accent_poc/bin/python -c "import pyopenjtalk; pyopenjtalk.extract_fullcontext('あ')"
```

- **ビルド**: pyopenjtalk はソースビルド（Cython）＝初回 pip で約 **44 秒**。ランタイム依存は `numpy` と `tqdm` のみ（PyTorch 不要）。
- **辞書**: 初回 `extract_fullcontext` 呼び出し時に GitHub リリースから `open_jtalk_dic_utf_8-1.11.tar.gz`（**DL 約 23MB / 展開後 103MB**）を自動取得しキャッシュ。オフライン運用では事前同梱が要る。
- **ライセンス（商用可否）**:
  - pyopenjtalk 本体 = **MIT**。
  - 同梱辞書 `open_jtalk_dic`（NAIST-jdic 系）= **修正 BSD（3条項・NAIST 2009）**。**商用利用可**、要・著作権表示とディスクレーマの再配布同梱、NAIST 名の推奨表記への流用禁止。M3 の机上想定「修正 BSD 系・商用可」を**実物の COPYING で確認**。
- **参考**: 本機には VOICEVOX が同じ pyopenjtalk＋同辞書を既に同梱（`/mnt/c/Program Files/VOICEVOX/.../pyopenjtalk/open_jtalk_dic_utf_8-1.11`）＝実績ある枯れた組み合わせ。

### 抽出ロジック（フルコンテキスト → M3 形式）

`extract_fullcontext(text)` は音素ごとに HTS フルコンテキストラベルを返す。使うフィールド:

- **p3**（`…-P+…` の P）＝現在音素。母音 `a i u e o`（大文字=無声化）・`N`（撥音）・`cl`（促音）を**モーラ核**として子音を畳み、モーラ列を得る。
- **A:a1+a2+a3**：`a1`=アクセント核までの相対位置（**0 のモーラ＝核**）、`a2`=アクセント句内の前方位置（1 始まり）。
- **F:f1_f2#…**：`f1`=アクセント句のモーラ数、`f2`=**アクセント型**（核位置。**0=平板**）。
- 各モーラの高低（H/L）は東京式規則で `f2`（型）と `a2`（位置）から決定：平板=1モーラ目 L・以降 H／頭高①=1モーラ目 H・以降 L／中高・尾高=1 L、2〜核 H、核+1〜 L。隣接モーラの H/L 差を **UP / DOWN / FLAT** に量子化＝M3 §1.3 の入力そのもの。
- **アクセント句境界は `a2` が 1 に戻った所で判定**（`I:` フィールドの句番号は呼気段落単位で不安定＝実測で誤検出。`a2` リセット方式に切替えて解消）。

スクリプト実体（スクラッチパッド）: `accent.py`（抽出）・`run_eval.py`（20文評価）・`ssot_cmp.py`（TS比較）。

## 3. 実測表②：アクセント核の精度（孤立単語 × NHK 引用形）

型（`f2`）を NHK 系引用アクセントと厳密照合。**0=平板 1=頭高 2以上=中高/尾高**。

| 語 | 期待型 | 抽出型 | 判定 | 備考 |
|---|---|---|---|---|
| 猫 ねこ | 1 | 1 | ✓ | 頭高 |
| 心 こころ | 2 | 2 | ✓ | 中高 |
| 先生 せんせい | 3 | 3 | ✓ | 中高 |
| 箸 はし | 1 | 1 | ✓ | 頭高（食器） |
| 橋 はし | 2 | 2 | ✓ | 尾高（構造物） |
| 雨 あめ | 1 | 1 | ✓ | 頭高 |
| 電話 でんわ | 0 | 3 | ✗ | 平板↔尾高（※後述で無害） |
| 桜 さくら | 0 | 3 | ✗ | 平板↔尾高 |
| みかん | 0 | 1 | ✗ | 平板を頭高化 |
| 飴 あめ | 0 | 2 | ✗ | 平板↔尾高 |
| 花 はな | 0 | 2 | ✗ | 平板↔尾高 |
| 友達 ともだち | 0 | 4 | ✗ | 平板↔尾高 |

**厳密一致＝6/12。だが外れ 6 件中 5 件は「平板⓪↔尾高（末尾核）」の取り違え**で、これは**語内の高低コンターが両者で完全に同一**（LHHH…H）＝差は後続助詞が下がるか否かだけ。実測でも:

```
電話（孤立, 尾高判定）  : de(L) N(H) wa(H)      → UP·FLAT   ＝平板と同一コンター
電話が鳴る            : de(L) N(H) wa(H) ga(H) → が高いまま ＝この文脈では平板挙動
花が咲く              : ha(L) na(H) ga(L)      → はな内は UP（平板と同一）、差はが↓のみ
```

`analyzeLyricFit` が採点するのは**歌詞モーラ列に乗る旋律の方向**（A-01〜05）。助詞 1 音の下がりは A-06/09（語境界×リズム＝現状保留）側の話で、**語内コンター照合には平板/尾高の別は効かない**。よって**実効精度は 6/12 よりかなり高い**。

さらに歌詞で致命的な**同綴異義（語義誤解）は文脈込みで正しく弁別**:

```
箸を持つ : ha·shi·o·mo·tsu  型[1,…]（頭高＝箸 ✓）
橋を渡る : ha·shi·o·wa·ta·ru 型[2,…]（尾高＝橋 ✓）
雨が降る : a·me·ga·fu·ru     型[1,…]（頭高＝雨 ✓）
飴をなめる: a·me·o·na·me·ru   型[非頭高]（雨と別 ✓）
```

M3 §1.3 が「最重＝DOWN（下がり目）の裏切り＝箸/橋型の語義誤解」と置いた核心ケースは、パーサが漢字文脈から読みを確定するため**正しく通る**。

## 4. 限界（歌詞特有の崩れでの外れ方）

1. **辞書外の英字＝スペルアウトで崩壊**。`Love`→ラブ（辞書内借用語）は正しいが、`Yeah`→`ワ・イ・イ・イ・エ・エ・エ・イ・チ`（1文字ずつ音読み）と崩れる。**英語の間投詞・造語は要・かな上書き**（呼び側の手入力かユーザー辞書）。
2. **平板/尾高の別が不確実**（§3）。語内コンターには無害だが、将来 A-06/09（語末助詞×拍・語境界）を本実装する時は**助詞前の下がりを鵜呑みにしない**（ソフト警告に留める）。
3. **孤立単語は末尾核（尾高）へ寄る傾向**。実運用は文（助詞込み）を渡すのが正解＝助詞が付くとアクセント結合規則で自然化する。**単語単位でなくフレーズ単位で投入すべき**。
4. **固有名詞・新語・意図的な変則読み**は naist-jdic のカバレッジ外だと外す。M3 の想定どおり **UniDic / tdmelodic 参照は「穴埋めの補正層」**として将来必要（本 POC では未接続＝穴の実害はまだ低頻度と判断）。
5. **同綴異義の意味は文脈頼み**。周辺語が薄い断片（歌詞は断片が多い）では意図と逆の読みに倒れ得る＝`accents` 上書き口（design.md 既設）で人間が最終確定できる設計を維持すべき。

## 5. 統合設計の推奨

### 方式：都度 spawn（`audio-analyze.ts` と同型の PythonCLI）

- **コールドスタート実測（新プロセス起動→抽出）＝0.13〜0.23 秒/回**（5 回計測）。内訳は `import pyopenjtalk` が 0.127s 支配的・辞書ロードは実質 0s・抽出本体は 20 文で合計 3ms（0.2ms/文）。
- `analyze.py`（Demucs/librosa＝数十秒〜分）と違い**桁違いに軽い**＝**常駐（b）も事前計算（c）も不要**。歌詞 1 行ごとの都度 spawn でも体感遅延なし。1 曲全行をまとめて 1 プロセスに渡せば import コストを 1 回に償却でき更に軽い。
- 実装は `audio-analyze.ts` の `run()`（detached spawn＋timeout＋abort でプロセスグループ kill＋stdout から JSON 抽出）を**そのまま流用**できる。新スクリプト `accent.py` は stdin で複数行受け→JSON 配列を stdout、が既存作法に合う。`CM_ACCENT_PY` / `CM_ACCENT_SCRIPT` の env 差し替え口も踏襲。
- 環境：本番 `_audio_poc/.venv` に相乗りせず、**軽量専用 venv（numpy＋pyopenjtalk のみ）**を別に切るのが依存衝突回避に無難（PyTorch 系と混ぜない）。辞書はオフライン用に事前展開して同梱。

### SSOT 裁定案（TS/Python 二重実装の罠）

- **モーラ分割の正典＝pyopenjtalk（Python）へ寄せる**。理由：アクセント付与は必ずモーラ分割を内部で行う＝TS 側で別ロジックを持つと**二源泉が微妙にズレる恒常リスク**。核・句境界・モーラを**一括で 1 箇所から出す**のが堅い。
- ただし **`lyric.ts splitMora`（TS）は廃止不要**。実測で **pyopenjtalk の g2p-かな出力に splitMora を当てるとモーラ数が 20/20 一致**（長音ーの扱いまで一致＝g2p が長音を `ー` に正規化し、splitMora もーを 1 モーラに数えるため）。よって:
  - **かな読みが既に手元にある高速パス**（`set_lyric` でユーザーがかな入力済 等）＝**TS splitMora で即応**（Python 呼ばず 0ms）。
  - **漢字交じり原文からモーラ／アクセントが要る**（`analyzeLyricFit` 本番）＝**Python（pyopenjtalk）が正典**。splitMora は漢字を読めない＝そもそも守備範囲外。
  - 両者が同じ入力（かな）で走る場面では**カウント一致をゴールデンテストで固定**し、乖離が出たら Python を正とする。
- 具体接続：`analyzeLyricFit` の `accents`（語ごと核位置）供給を、現行「内蔵平板ヒューリスティック」→「pyopenjtalk 抽出（§1 の型→核）」へ差し替え。**フレーズ単位で投入**（§4-3）。英字間投詞・変則読みは `accents` 上書き口で人間が確定（design.md 既設の思想「機械は候補まで・仕上げは人間」を維持）。

## 6. 代替の軽い確認（1 段）

| 候補 | アクセント核 | 導入コスト | 判定 |
|---|---|---|---|
| **pyopenjtalk** | **あり**（fullcontext から核・句境界・モーラ一括） | 軽（辞書 103MB・PyTorch 不要） | **採用** |
| MeCab + UniDic | あり（`aType`/`aConType` フィールド） | 重（MeCab C 依存＋UniDic 数百MB〜） | 補正層候補（穴埋め）。単体採用は過剰 |
| SudachiPy | **なし**（形態素のみ・アクセント非対応） | 中 | 不適（別途アクセント辞書が要る） |
| kuromoji（Java） | **なし**（読みはあるが核なし） | 中（JVM） | 不適 |
| tdmelodic（NEologd にアクセント付与） | あり（新語・固有名詞向け） | 重 | pyopenjtalk の**カバレッジ穴埋め**として将来 |

結論：**アクセント核を「ほぼ無料で」返す唯一の軽量手＝pyopenjtalk**。M3 机上推奨は実測で追認。UniDic/tdmelodic は「pyopenjtalk が外す固有名詞・新語」の補正層として、実害が出てから足せばよい（v1 不要）。
