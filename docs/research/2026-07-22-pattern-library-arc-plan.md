# パターンライブラリ化アーク 計画書（2026-07-22・オーナー方針確定・実装は次セッション）

**この doc の役割**＝オーナーの方針転換「パターンはライブラリ（ネタ帳の一級市民）にする」を、次のセッション/担当が
文脈ゼロから実行できる形で残す引き継ぎ計画書。正典の連なり＝本doc →（設計確定時に）design.md へ下ろす → 実装。

## 0. オーナーの言葉（方針の原文・2026-07-22）
> 要はここがライブラリなんじゃない？？必ずしも生成とかサンプルによらなくていい。ベーシックなパターンをネタ帳に
> ライブラリ扱いで登録しといて、検索すれば良くない？？そうするとSectionの選択画面とパーツ共用できれいに行けそう。
> なんでデフォのパーツを、フォルダ可能なようにタグをしっかり整備して、たくさん準備する方向がいいはず。

読み解き：
1. **パターンの置き場＝ネタ帳**（コード内辞書ではなく）。工場出荷のベーシックパターンを「ライブラリ扱い」のネタとして登録。
2. **発見＝検索**（cm-search 意味検索＋タグ）。生成器で候補をひねり出す必然はない。
3. **Section の選択画面と単体エディタの帯は同じ棚を共用**。
4. **タグをフォルダ的に整備**して、デフォパーツを**たくさん**準備する。

## 1. なぜこうなったか（経緯＝統一原理の帰結）
- 2026-07-22 に H1/H2＋統一原理「**content は人が仕上げる単位**」を確定（`2026-07-22-performance-editing-architecture-audit.md`・
  `2026-07-22-melody-pattern-thought-experiment.md`・design「修理#1〜#3」）。背景パートはパターンが content。
- 実機を触ったオーナーの評定＝「**候補が貧弱**」「**左手いまいち**」。直接因＝**コード内辞書が薄い**（例：ベース相対の
  rock 型は2種→候補2件。E2E実測）。コードに焼いた辞書は「量を増やす＝コード変更」の歪みを持つ。
- パターンが content なら、**パターンの置き場もネタ（DB）が正しい**＝量産はコンテンツ作業になり、ユーザーの自作/採取
  パターンと工場出荷が同じ棚に並び、検索・タグ・試聴・採用の既存機構がそのまま使える。

## 3. 設計素案（design.md へ下ろす前の叩き＝次セッションで確定させる）
### 3-1. ライブラリネタの表現
- **新kindは作らない**（作成タイル/フィルタ肥大の病理＝`2026-07-14-topview-inventory.md`）。既存 kind
  （chord_pattern / rhythm / bass(相対 content) / 将来 section_inst）のネタに **`lib:` タグ名前空間**を付けて「ライブラリ扱い」。
- タグ設計案（フォルダ的階層＝オーナー要望の核・**ここはオーナーと相談して確定**）：
  - `lib:factory`（工場出荷）／`lib:user`（自作・採取。省略可＝lib: が付いてなければ普通のネタ）
  - `genre:rock` `genre:citypop` …（既存ジャンル語彙と統一）
  - `scene:verse` `scene:chorus` …（適用場面＝旧 roles）
  - `tempo:60-95`（域。検索絞りにも表示にも使う）
  - 楽器系統は kind＋program で表現済み＝タグ重複させない。
- パターン固有メタ（patternId・（改））は現行 content 契約のまま＝ライブラリネタの patternId は自分のネタ id でよい
  （辞書型ID の役目はタグへ移る。要設計判断）。
### 3-2. 工場出荷シード
- コード内辞書（chordLibrary/bassLibrary/drumLibrary）を**シードデータの源**に格下げ：辞書→ライブラリネタを一括生成する
  **seed スクリプト**（api 側・冪等＝同名 lib:factory 既存ならスキップ/更新）。実行はデプロイ手順 or 明示コマンド。
- 以後の量産は**ネタ登録＝コンテンツ作業**（研究docの未実装型＝ギター3連系2型・管弦8型・左手リズム型などが最初の弾）。
  真因調査の改善（feel既定・vel層・エンコード忠実度）を**シード生成時に反映**してから量産する（順序重要＝薄い上に平板、を防ぐ）。
- コード内辞書の去就＝当面残置（gen_* の style/pattern ノブの後方互換＋seed源）。ライブラリが安定したら縮退を別途判断。
### 3-3. ピッカーの共用＝ライブラリ検索化
- `PatternPickerBar` の候補取得を「生成器×seed」から**ライブラリ検索**へ差し替え：
  `kind＋lib:＋genre:タグ＋（テンポ域）` で絞り→候補カード（既存の▶試聴/採用UIは不変）。テキスト検索（cm-search）も同じ口で。
- Section のコード楽器/ベース/ドラム引き出しも同じ検索口を使う＝**共用**（オーナーの狙い）。
- 「おまかせ」＝絞りなしからの提示（+シャッフル）。生成器は「ライブラリに無い形が欲しい時」の第二経路として残す（位置づけ整理は L5）。
- 適用＝ライブラリネタの content を**コピー**して対象ネタへ（copy_neta の流儀・ライブラリ原本は不変）。
### 3-4. ネタ帳汚染への防御（最重要の設計点）
- 工場出荷100+件が一覧を埋める事故＝「すべて192件がE2Eジャンク」の前科と同型。対策案（**オーナー裁定必要**）：
  - 案A：一覧の既定で `lib:factory` を畳む（「ライブラリ」セクション/トグルで開く）
  - 案B：ライブラリ専用の棚ビュー（ネタ帳とタブ分離）
  - どちらもタグ整備が前提。検索・ピッカーからは常に見える。
