# 独立監査：ボーカル採譜レーンの天井判定 — 「採譜はこれ以上厳しい？」への回答

- 日付: 2026-07-15（夜間・独立監査）
- 監査対象: Demucs vocal stem → PESTO f0 → energyVAD → 半音RLE＋断片後処理（`_audio_poc/analyze.py`）→ corpus-Viterbi 復号（`apps/api/src/music/melody-decode.ts`・λ=0.5）
- 立場: 先入観のない読み・測り・判定のみ。**コード変更なし**。
- 使用データ: LostMemory（GT=`/tmp/piano2.mid` track8・offset 2.45s・mir_eval note-F onset±50ms/pitch±50cent）＝F2ハーネス（scratchpad `f2work/` の pesto.npz・prep.json・decoded.json・lm_full.json をそのまま再利用）。蜿蜒＝キャッシュ済み統計のみ（`out_enen.json` の before 458ノート＋f0輪郭50ms、DBネタ e7b9e7ad の after 426ノート）。**音源の再取得は不要だった**（リテラル採譜は本docに保存しない・統計のみ）。

## TL;DR（結論5行）

1. **現在地は8合目**。現行 note-F 0.749（クリーンレンダー）は、実ポップ混合音源でのGPU DNN SOTA（Mel-RoFormer COnP 0.798・Gu2023 0.700）と**同じ帯**に、CPUのみ・モデルサーバー無しで到達している。
2. **「あと一段」は実在する**。誤り分解の結果、残FN 206個の主犯は (a) **同音連続ノートのRLE構造的マージ**（62個・RLEは同ピッチの音節境界を原理的に切れない）と (b) **energyVADゲートの取り過ぎ**（61個・ゲートを外すと F 0.749→**0.801** を実測）。ピッチ系の誤りは49個(24%)まで減っており、**オクターブ誤りはゼロ**。
3. 上位レバー2本＝**①VADゲートの適応閾値化**（効き実測+0.05・数十行・低リスク）と**②同音区間の音節オンセット分割**（理想上限+0.05・中コスト）。両方が理想的に決まった場合の上限 F 0.849＝**PESTO生f0のフレーム天井（exact 0.862）にほぼ張り付く**。
4. **やらない判定**: ROSVOT CPU化（SOTA帯=レバー2本の着地点と同等・コスト大）、VADの学習モデル化（①で代替）、ビブラート中心線の追加投資（median130ms＋復号で中心線偏差>50centは残り4%＝ほぼ解決済み）。
5. 蜿蜒の残スケール外7.3%（31個）は**約1/3が和声的/旋律的短音階の正当音（G#・F#）**＝メトリクスの過大計上。真の残誤りは体感5%弱で、ノートの17%は**ノート内f0振れ幅>100cent＝単一半音ラベルが原理的に曖昧**な領域。ここから先は手法でなく歌唱の性質。

## 1. 誤りモード分解（LostMemory・定量）

最終出力（decode λ=0.5・481ノート）vs GT617ノート。matched 411（P 0.854 / R 0.666 / F 0.749）。

### FN（取り逃し206個）の内訳

| 由来 | 件数 | 割合 | 説明 |
|---|---|---|---|
| **onset_only（ピッチ正・オンセット>50ms）** | 94 | 46% | うち **62個＝同音マージ由来**（1本のestノートが同ピッチのGT複数個を吸収。est 59本がGT境界60個を潰していると実証）、37個＝純オンセットジッタ |
| **vad_or_post_removed（f0正・conf≥0.5なのに消えた）** | 61 | 30% | energyVADゲート（p42相対閾値）が実歌唱の弱音を除去。ゲートを外した対照実験で recall 0.666→0.768（+63個）＝ほぼ全数がゲート由来と確認 |
| semitone12（±1〜2半音の丸め違い） | 21 | 10% | 遷移/しゃくり区間の丸め |
| pitch_other / f0_wrong | 28 | 14% | f0自体が外れ（ポルタメント・子音区間） |
| conf_gate_removed | 2 | 1% | PESTO conf<0.5 |
| **octave** | **0** | **0%** | **PESTOでオクターブ誤りは根絶済み** |

### FP（偽ノート70個）の内訳

onset_only 36（51%・分割/境界ずれの相方）、pitch_other 22、semitone12 10、**ghost 2（3%）**。幽霊ノートはVADでほぼ根絶済み。

### 裏付け（寛容度スイープとフレーム天井）

- onset許容 50→100ms で F 0.749→**0.803**、200msで0.809＝**FNの半分はオンセット位置の問題でありピッチは正しい**。pitch許容を150centに広げても+0.003しか動かない＝ピッチはもう主戦場ではない。
- PESTO生f0（median130ms後）のGT有声フレーム一致率＝**exact 0.862 / octeq 0.862**（オクターブ誤り0）、カバレッジ1.000。**これがこのフロントエンドの知覚天井**。
- corpus-Viterbi復号の副作用＝ゼロ（λ=0.5でmatched数±0・クリーン曲では実害も実益も微少。効くのは揺れ歌手＝設計どおり）。

## 2. 蜿蜒（生歌・GT無し・統計のみ）の残誤り解剖

- before（analyze素通し458ノート）スケール外13.5% → after（DB本番・decode λ=0.5・426ノート）**7.3%（31個）**。
- **スケール外31個のpc内訳（Aマイナー・トニック相対）**: ♭2度(A#)=12・長3度(C#)=9・**長6度(F#)=5・導音(G#)=5**。F#/G#は旋律的・和声的短音階の**正当音**＝ナチュラルマイナー基準のメトリクスが過大計上している（評価の直しどころであって採譜の誤りではない）。真の疑義は♭2/長3の21個＝4.9%相当。
- **ノート内f0振れ幅**: 中央値33centだが **17%のノートが>100cent・10%が>200cent** ＝しゃくり/ポルタメントで**1音に1ラベルという出力形式自体が情報を落とす**帯域。スケール外ノートの振れ幅は中央33centで通常ノートと同じ＝「揺れたから外れた」のではなく歌唱がそこにいる。
- **丸めの一貫性**: ノート中心線とラベルの偏差>50centは**4%のみ**（中央値0cent）＝ビブラート中心線問題は median130ms＋復号で実質解決済み。

## 3. 天井の判定 — 外部相場との突き合わせ

同一メトリクス（COnP＝onset±50ms・pitch±50cent・offset無視）での相場：

| 系 | note-F/COnP | 条件 |
|---|---|---|
| Gu et al. 2023（GPU DNN） | 0.700 | MIR-ST500 実ポップ混合音源 |
| SpecTNT | 0.778 | 同上 |
| **Mel-RoFormer large（2024 SOTA）** | **0.798** | 同上（分離ファウンデーション＋Transformer・GPU） |
| ROSVOT（ACL'24・クリーン独唱） | COnPOff 0.774 | クリーン・GPU |
| **現行チェーン（CPUのみ）** | **0.749** | LostMemory クリーンレンダー |
| 〃 VADゲート無し（対照） | 0.801 | 同上 |
| 〃 ＋理想音節分割（上限見積り・GT境界使用） | 0.849 | 同上 |
| PESTO生f0のフレーム天井 | exact 0.862 | 同上 |

注意: 現行の0.749は**クリーンレンダー1曲**での値で、MIR-ST500系より易しい土俵（一方でSOTA側は500曲学習のGPUモデル）。絶対比較でなく「帯」の比較として読む。

**判定＝8合目**。pyin時代0.568→現行0.749。この手法クラス（f0追跡＋規則分節＋統計復号）の実用天井は**クリーンで≈0.85**（生f0フレーム天井0.862に規則分節で張り付く限界）、生歌はそこから−5〜10pt。上位レバー2本の現実的な着地は**0.80〜0.83＝9合目**。そこから先に残るのは、両レバー理想適用後の残FN 94個の解剖が示すとおり「f0誤り（ポルタメント/子音）30・遷移跨ぎ16・分節残り48」＝**GPU級の音響モデルに置き換えても数ptしか動かず、一部は正解が一意でない領域**（歌唱の連続的なピッチ運動を離散ノートに割る行為そのものの曖昧さ・MIR-ST500ですら注釈一貫性の問題が論文に明記される水準）。

## 5. オーナーへの正直な一言

**「あと一段ある。ただしそれが最後の一段」**。残っている誤りの76%はピッチではなく「区間の切り方」（同音マージ・VADの取り過ぎ・オンセットジッタ）で、これはレバー①②で+0.05〜0.08取れる見込みが実測ベースである。それを取り切ると生f0のフレーム天井（0.862）に張り付き、以降はGPU級DNNに総取り替えしても相場的に数ptの世界、しかも生歌の17%のノートは「1音1ラベル」という形式自体が情報を落とす原理的曖昧域に入る。**①②をやって、そこで打ち止めにするのが正しい**。なお蜿蜒の「スケール外6.6〜7.3%」は1/3が旋律的短音階の正当音なので、体感より数字が悪く見えている。

## 実装（2026-07-15）＝V1 ボーカルVADの適応閾値化＋recall犯人の是正

対象 `_audio_poc/analyze.py` のみ。venv-f2＋F2ハーネス（GT track8・offset2.45・PESTOキャッシュ再利用）で before/after 実測。

### 実装した2点
1. **VADゲートの適応閾値化**（`stem_energy_mask` を vocal/bass 共用の適応閾値へ）。pre-gate の voiced率で二段：
   休符あり曲（voiced率 ≤ **0.85**）＝従来どおり p42 で aggressive に滲み/幽霊を除去、常時歌唱/演奏曲
   （voiced率 > 0.85）＝RMS ヒストグラムの **Otsu 谷**だけを閾値にしてノイズ床のみ除去（固定 pXX が演奏
   レベルに食い込み実音を削る退行を防ぐ）。定数 `VAD_CONTINUOUS_VOICED=0.85`・谷は `_otsu_valley_rms`。
2. **postprocess の absorb を緩和**（`NOTE_ABSORB_NB_MIN` 0.30→**0.60**）。下記の犯人特定を受けた是正。

### 犯人の再特定（監査の帰属を実コードで検証・重要）
本監査§1は取り逃し61個を「energyVADゲート（p42）が実歌唱の弱音を除去」と帰属した。**実装時に計装して検証したところ帰属先が違った**：
- **p42 ゲートは LostMemory では実質ノーオプ**。PESTO conf≥0.5 が既に低RMSの滲みフレームを除外しており、
  ゲートで落ちるのは **18335→18305 frame＝1ノート**のみ（ゲート有無で pre-post note-F は 0.801 で不変）。
- 消えた61ノートの実体は **`postprocess_notes` の absorb 段**（短ノートを優勢な隣接持続音の"割れ"とみなし削除）。
  計装で GT マッチが **474→415（−59）** を確認＝これが取り逃しの主因。absorb 定数がボーカル向けに攻めすぎ、
  LostMemory の**速い実フレーズ**を dither と誤認していた（本監査がベース側で「盲移植で悪化」と観測したのと同じ機序）。
- 是正: `NOTE_ABSORB_NB_MIN` を 0.30→0.60 に上げ、吸収先を **≈0.6s以上の明確な held音**に限定。速い実フレーズは温存。
  → 適応ゲートは主に**汎化**（蜿蜒 vs SURFACE 型）と将来の常時歌唱曲での退行防止に効き、LostMemory の数値回復は absorb 緩和が担う。

|---|---|---|---|
| LostMemory（GT track8） | note-F | 0.755 | **0.792**（≥0.78 ✓） |
| | P / R / onF | 0.859 / 0.673 / 0.782 | 0.854 / **0.739** / **0.824** |
| | ノート数 | 483 | 534 |
| 蜿蜒（PESTO・GT無し・yt-dlp再取得→実験後削除） | ゲート後 voiced率 | 0.575 | **0.575**（不変＝幽霊防止は退化せず） |
| | 10秒ビン密度の二極（polar） | 4.4 | **4.1**（二極維持・SURFACE型と誤判定せず aggressive を選択） |
| | melody_notes（フル実走） | 426（DB旧） | 490（recall回復・voiced率0.575・polar4.1） |

- 蜿蜒は本実装で `yt-dlp --js-runtimes node:$(which node)`（android client で n-challenge 回避）→ demucs 分離 → PESTO →
  `analyze.py` フル実走（bpm_hint=235・exit0・JSON健全・total 112.7s）で検証。**音源/stem/リテラルf0は検証後に削除**（著30-4運用）。
- adaptive ゲートの mode 判定は LostMemory(voiced0.557)・蜿蜒(0.741) とも **aggressive**（=従来 p42 と同挙動）に落ちるため、
  計測した精度は「ゲート変更で悪化していない」ことも同時に担保。緩ゲートが発動するのは voiced率>0.85 の連続演奏曲のみ（下ベース監査参照）。
