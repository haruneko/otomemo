## 1. 手順ログ（何をどう操作したか）
1. ホーム「＋セクション」→エディタ。title入力→設定を開き 調=A/短調→小節 **＋を8→32まで（24タップ）**。📸01/01b
   - ⚠ 初回 `mode` selectOption が **strict違反で黙って失敗**（`getByLabel("mode")` が section-mode/mode-edit/mode-erase と多重一致）＝これは検収スクリプト側の教訓。実UIでは 短調 を明示選択で保存確認。
2. **コード進行**：UIに palette 入口が無いため `/music/gen_chords`（curl）で `palette=dorian` の8小節（Am–E7–Am–E7–F–**Dmaj(IV)**–E7–Am、♮6のD長がドリアン色）を生成→MCP `create_neta`+`place_child`×4（0/32/64/96拍）。
3. **骨格**：いじる▾→「構造=AABA」→「骨格」→候補トレイ「置く」。📸02/03/05/06。候補は **16小節・19音**（＝32小節でなく16で頭打ち・§3-H1）。
4. **骨格の机**：骨格ブロックをタップ→全画面の机。色付け=**少し**／かたち=**山** を設定→「機械に叩き台」で3案。📸17/18/19（採用はせず＝置いた骨格を保つ）。
5. **表面化(メロ)**：骨格ブロック「メロを作る▶」→候補3件（🎲もっと×2）→**並べ替え軸**を フック度/歌いやすさ/期待理論 に切替（📸07/08/09/10）→「置く」。📸11
6. **ベース**：いじる▾→ベース型=シティポップ→「ベース」→置く。📸12/13
7. **ドラム**：いじる▾→ビート型=ロック＋フィル=中→「ドラム」→置く。📸14/15/16
8. **32小節充填**：各16小節ユニットを 64拍(bar17)へ MCP `place_child` で再配置（同一ネタ再利用＝bars17-32は1-16の反復）。
9. **対旋律**：骨格「対旋律を作る▶」→候補→置く（bars1-16）。📸20
10. 全景：モバイル(📸21冒頭/📸22末尾)・デスクトップ1280(📸23)・ネタ帳カード(📸24)。メロ生成13ノブの壁は📸04w-00/01。

## 3. デザイン所見（重大度順・スクショ/該当箇所つき）

### 高（機能/シナリオを塞ぐ）
- **H1：全生成が16小節上限（黙って頭打ち）**。`normalizeFrame`＝`out.bars = Math.min(16, …)`（`apps/api/src/music/generate.ts:104`、`barsOf`も211で同）。骨格/メロ/ベース/ドラム/コードいずれも**1回の生成は最大16小節**。→ オーナー要求の「32小節Aメロ」は一発で作れず、16小節ユニット×2の配置（or 右端ドラッグでループ伸ばし）が必須。**UIに上限の明示が無く**、32小節セクションで「骨格」を押しても16小節候補が黙って出る（候補メタの"16小節"表示で辛うじて気づく）。📸05。→ 32小節では「16小節×2を置く」動線を案内 or 上限をセクション尺に追従させる検討。
- **H2：パレット（aeolian/dorian）にUIが無い**。`gen_chords` の palette は API/MCP のみ（`mcp.ts:549`・`http.ts:207`）。web の ChordEditor は手入力グリッド（root/triad/ext/alt/bass/len）で **palette 生成の入口が存在しない**（`GEN_PARTS` にコードは無い＝いじる▾でもコードは生成対象外）。→ 「コード進行生成＝palette を試す」は**web から実行不可能**。今回は curl/MCP で代替。→ コード生成（palette/cadence/borrow/secondaryDom）を web に露出する価値が高い。

### 中（毎回イラつく）
- **M1：骨格からの表面化(メロ/ベ/対旋律)が"置く"で必ず bar1 に着地**。`placeCandidate` の position 既定=0（`useMelodyGen` L408・SectionEditor.tsx:611 の呼びは引数無し）。32小節で骨格が bar17 にあっても「メロを作る▶→置く」は bar1 に落ち既存と重なる。正しい位置付けは**骨格の机**(skelPosition経由)が要る。→ セクション側トレイからは狙った位置に置けない＝32小節で顕在化。
- **M2：いじる▾で生成ボタンと修飾セレクタが近接していない**。並びは メロ/ベース/ドラム/リフ/管弦（生成）→ ビート型/フィル/ベース型/ベースフィル/構造/骨格（修飾）。ドラムの型(ビート型/フィル)が**ドラムボタンの3行下**＝「型を選んでから押す」動線が離れ、先に見えるボタンを押すと型未反映で生成。近接(proximity)/グルーピングが弱い。📸02/03。
- **M3：小節数変更が ±1 のみ＝8→32 に24タップ**。`section-bars` は −/＋ の1刻み（SectionEditor.tsx:628-633）。32小節要求では特に苦痛。→ 直接入力 or ±4/±8 ステップが欲しい。
- **M4：候補トレイの横スクロールで両端カードがクリップ**（メロ候補・骨格叩き台とも）。"16小節"が"6小節"に見える等、端カードの情報が切れて初見で誤読。📸08/19。

### 低（磨き）
- **L1：レーンラベル折返し**：「コード進行→コード進/行」「コード楽器1/2→コード楽/器1」。ラベル列が狭く2行折れ。📸01/21。
- **L2：メロノブ壁の右端ラガード**：seg/slider はロック🔓列があり、select 行(最小音符/句割り/展開/声種)はロック無しで全幅＝右端が不揃い。サブ説明の折返し（人間味/ベースをよける）でセグメントが縦にずれる。📸04w-00/01。
- **L3：骨格レーン(青)と対旋律レーン(青)が同系色**で近い（判別は可能）。📸21。
- **L4：セレクタ左端がラベル幅依存でラガード**（フィル/ベースフィル で select 開始xが不揃い）。📸03。

### 良かった点（相対劣化の逆＝維持すべき）
- **メロ生成13ノブの壁は"壁化"を回避**：プリセット8種＋🎲を主役に、詳細は「細かく設定する」で畳む(progressive disclosure)。開くと1756px(2.5画面)だが、● 歌い回し/フレーズの組み立て/他パートとの絡み/人間味 の**見出しでグループ化**され読める。📸04w。
- **骨格の机**：スナップ/入力先/色付け/かたち/機械に叩き台 がラベル群で整理・4ステージ(①ビート②コード③骨格④表面)・2小節レンズ窓・音程バッジ(5度/8度/4度)・対位ストリップ＝壁化していない。📸17。
- **候補レンズ**：軸切替で「フック度」等のバッジ表示＋並べ替えが効く。**弾かず並べ替えるだけ**の設計どおり。対位法警告(⚠並8×4 直5×1)も候補カードに出る。📸08。
- **新レーン(counter/riff/section_inst)**：ラベル・色・空セル(配置ターゲット)とも整合＝破綻/欠けは無い。📸21。
- **ネタ帳カード**：セクションカードは全レーンを積んだミニ概形＋「32小節」表示＝一覧で構成が見える。📸23/24。

## 4. 所要タップ/時間（実測概算・UIで作れる範囲）
- セクション作成の固定費：＋セクション(1)＋title入力＋調(1)＋短調(1)＋**小節8→32＝＋24タップ**（M3）。
- 16小節1パス（4トラック）：いじる/構造/骨格/置く/メロ▶/置く/いじる/ベース型/ベース/置く/いじる/ビート型/フィル/ドラム/置く ≈ **15タップ**。
- 32小節化：コードはUI不可(H2)＝curl/MCP、各パートの後半充填は"置く"がbar1固定(M1)のため**右端ドラッグでループ伸ばし** or 机経由が必要＝+数操作。
- 実測時間：各生成の api 往復 1〜3s（メロ候補「もっと」2.5s/回）。骨格→机→叩き台は即時。全体はネットワーク往復律速。
