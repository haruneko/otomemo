```markdown
# spec-anchored-dev — 検証責任の再配置プロトコル

この体制の原理：人間はコードを読まない。その代わり検証を5層（設計の正しさ/スペック一致/コードの正しさ/受け入れの公正さ/機械で測れない質）に分配し、全部埋める。人間が手放すのは「読むこと」だけで「判断すること」は手放さない。

発動場面を判定し、該当プロトコルに従う。

## P0. 前提チェック（毎回・10秒）

3点セットが揃っているか確認。欠けていたら作業前に指摘して止まる：
1. **スペック正準**：requirements → architecture → design の文書があり、今回の変更の根拠がどこかに書けるか
2. **機械ゲート**：型チェック・テスト・（UIなら）e2e がCIで強制されているか
3. **受け入れ責任者**：この機能の実物の良し悪しを診断レベルで差し戻せる人間がアサインされているか

**3が空欄の機能は実装しない**。「誰も受け入れられないものをAIに書かせない」が鉄則。粒度分解・サンプリング受け入れ・作らない、のいずれかの裁定を人間に求める。

## P1. scaffold（新プロジェクト/新領域の初回のみ）

以下を生成し、内容はプロジェクトに合わせて人間の裁定を得る：
- `docs/requirements.md`（体験の言語で。実装語彙を混ぜない）/ `docs/architecture.md` / `docs/design/`（領域別分割・節単位所有）
- `docs/backlog.md`（保留置き場＋着手ルール「なぜ今それが要るか」）
- `docs/research/README.md`（索引。research doc追加時に1行追記＝散逸防止）
- `docs/process/dri.md`（**層・領域ごとの裁定オーナー表**。空欄を残さない）
- `docs/process/review-loop.md`（Planner/Implementer/Acceptor/Regressor分離。実装者は自己採点しない・基準は実装前固定・Regressorは素眼）
- `docs/process/confirm-list.md`（自走時の判断待ち置き場。止まる明確な理由がない限り進め、不明点はここに溜める）
- CLAUDE.md に本規律の要約と「変更は上から下ろす」を焼き込む
- CIに型・テスト・e2e・（可能なら）ゴールデン/bit一致検証を配線
- **既存コードベースの場合**：スペック逆生成（AIが書き起こし→人間が裁定）→ゴールデンテストで現挙動を固定→以後の変更のみ本体制

## P2. 修正時プロトコル（実装・変更のたび）

変更は上から下ろす。順に確認してから書く：
1. コンセプト/体験原則に反しないか
2. アーキテクチャに反しないか
3. 要件に反しないか
4. 設計に反しないか（食い違うなら**スペックを先に直す**＝該当DRIの裁定を取る）
5. タスクに反映
6. テストを先に書く（赤）。契約・データ構造・外部IFは厳守、探索フェーズは緩くてよい（strictness段階制）
7. コード（緑→リファクタ）

実装規約：
- **契約SSOT**：スキーマは1箇所に書き、検証・型・外部IFをそこから導出。同じ契約を2箇所に手書きしない
- **既定bit一致**：新しいパラメータ/ノブは「既定値＝既存出力と完全一致」で足す。機能追加＝回帰ゼロを機械証明する
- **黙って劣化しない**：フォールバックで品質を落とすくらいなら明示エラー。未対応は未対応と言う
- **縦スライス**：UI＋API＋ドメインを薄く通す。横積みしない
- コード内コメントから根拠のspec節を参照する（トレーサビリティ）。コミットは「何を＝なぜ」形式

## P3. 報告プロトコル（人間への報告のたび）

- **ステートレスに書く**：読む人の記憶を仮定しない。文脈を毎回再掲し自己完結させる
- **全緑が報告の前提**：型・テスト・e2eの実行結果を数字で添える（「api 246緑」形式）。赤のまま「できました」と言わない。赤なら赤と、スキップしたならスキップしたと書く
- **実物確認依頼リストを明示**：機械で判定できなかった項目（見た目・使用感・ドメイン品質）を番号付きで列挙し、受け入れ責任者の手番として配送する
- **残タスクを提示**：一段落ごとに。**完成宣言は人間が握る**——自分から「終わり」を宣言しない
- 判断が要る点は confirm-list に溜め、止まる明確な理由がない限り進める

## P4. 受け入れプロトコル（検収のたび）

- **実装セッション/エージェントは自分の成果を受け入れOKと判定しない**。別セッション・別モデル・またはサブエージェントを「ユーザー役」にして実際に使わせ、その感想を監査として受ける
- 受け入れ基準は実装前に固定されたものを使う（後出し変更しない）
- リグレッションは素眼で（変更内容を知らせず触らせる＝確証バイアス回避）
- 人間の差し戻しは**体験の言語のまま**受け取り、実装語彙への翻訳はAIがやる。差し戻し→修正→実物で再確認→部分承認→次の違和感、の連鎖を「違和感が消えるまで」刻む

## P5. 裁定と知識の永続化（判断が発生するたび）

- 裁定は「誰が・いつ・何を・なぜ」で文書に帰属記録する。**記録なき裁定は存在しない**。記録済みの裁定を蒸し返さない
- 調査・実測・設計判断は `docs/research/` に出典＋実測値＋設計含意つきで格納し、索引に1行追記。**棄却した案も棄却理由つきで記録**（同じ検討を二度しない）
- 文書の陳腐化に気づいたら、直せないときでも**陳腐化を自分で宣言**する注記を入れる（黙って腐らせない）

## P6. 失敗→仕組み変換（同種の失敗が2回目に見えたとき）

失敗を直すだけで終えず、**再発を機械が検出する仕組み**を同時に提案する（例：UI崩れ→ヘッドレスブラウザ自己チェック、報告の行き違い→報告様式のルール化、セッションのこじれ→引き継ぎ書を書いてコンテキストを捨てる）。仕組み化の提案はP3の報告に含める。

## チーム運用の差分（ソロ原典との違い）

- ブランチ＋PR前提。PRの単位＝縦スライス。**PRの必須ゲート＝spec diff＋機械ゲート緑エビデンス＋受け入れ記録**。コード読解レビューは任意（読みたい人は読んでよい）であり必須ゲートにしない
- 上位docの変更は該当DRIの承認事項（dri.md参照）
- design文書は領域別ファイル・節単位所有で同時編集競合を避ける。research索引はappend-only
- モデル階層：計画・設計・高複雑度は上位モデル、実装・E2E実走・低複雑度修正は下位モデルへ委譲（コスト最適化）
```
