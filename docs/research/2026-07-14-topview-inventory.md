# トップ画面（ファーストビュー）完全棚卸し（2026-07-14）

オーナー評「トップのファーストビューが情報過多」。Fable エージェントが再設計するための一次資料。
「いじるメニュー」で使った手順（棚卸し→定量→過去提案突合→Fable設計）のトップ画面版。

- 対象コード：`apps/web/src/App.tsx`（830行・トップ全体）／`apps/web/src/components/NetaList.tsx`（カード＋密度切替）／
  `apps/web/src/kinds.ts`（種別レジストリ SSOT）。
- 実測：モバイル 390×844（DPR2）と デスクトップ 1280×900、`http://100.109.159.48:8787`、Playwright。
- スクショ：`scratchpad/topview-shots/`（14枚）。索引は §6。

## 0. 結論（先に）

- **モバイル初期表示（above the fold, 390×844）にタップ標的が 47 個**。うち **作成タイル14＋種別フィルタ13＝27個（57%）が「作る/絞る」の操作壁**で、
  **実データ（ネタのカード）はフォールドの一番下に1枚の頭だけ**しか見えない。可視文字数は約129字。
- 病理は **いじるメニューと同型**＝「**種別(kind)が増えるたびトップに1タイル＋1フィルタが恒久的に割り込む**」フラットな壁。
  今日の WP-X3（counter/対旋律・riff/リフ・section_inst/管弦）追加で **作成タイル 6→9（パーツ行）**・**種別フィルタ 10→13** に膨張。
  しかもこの3種は project スコープに **実在 0〜1件**（§5）＝「ほぼ使っていない種別のために毎回フォールドが伸びた」。
- ただし **プロジェクト・ピッカー（器チップ）は肥大していない**（実在する器は「研究」1個のみ・チップ5個で収まる）。トップの肥大は
  **2枚のフラット壁＝作成タイル(14) と 種別フィルタ(13)** に局在する。
- 到達タップは浅い（主要タスクは概ね1タップ）。**問題はタップ深さではなく、最初の意味ある操作に入る前に視界を占める操作数（認知負荷）と、コンテンツがフォールド外へ押し出されること**。

## 2. 情報過多の定量

### 2.1 フォールド内の情報量（モバイル 390×844）
- タップ標的：**47**
- 可視テキスト：**約129字**（フォールド内 DOM テキスト実測）

- notebook 列の全高：**16,818px**（すべて=192件を無限に縦積み＝仮想化なし。§5参照）。

### 2.2 「最初の意味ある操作」に入る前に視界へ入る操作の数
- 作成タイル14＋種別フィルタ13＋器チップ5＋検索1＋mood1＋密度3＝**27〜37個**の操作が、
  最初のネタ1枚に触れる前に視界を占める。**"作る/絞る" の道具がコンテンツより先に・かつ大面積で出る**。

### 2.3 主要タスクの到達タップ数（初期状態＝すべて/カード）

## 3. デスクトップ 1280 の対比

- 2ペイン（左=ネタ帳レール minmax(0,1fr)＋右=編集ペーン）。**同じ操作クローム（14タイル＋13フィルタ＋…）が左レール448px幅に収まり、先頭カードは y≈680px から見える**。
  右ペーンが「ネタを選ぶとここで編集」の空状態＋＋曲を組むCTAでバランスする＝**モバイルほど詰まって見えない**。
- 種別フィルタはデスクトップ幅だと **1行10個＋折返し3個**（`.filter-kinds` repeat(minmax(0,1fr))）。モバイルは 2行（10+3）。
- 示唆：**病理はモバイル単一列で顕在化**する（全部が縦1列に積むため）。再設計はモバイル基準で。

  → 実装後：**作成タイル パーツ行 6→9・種別フィルタ 10→13**。

### 4.2 判定：**いじるメニューと同型の病理＝YES**
- **1 kind 追加 = トップに恒久的に ＋1作成タイル ＆ ＋1フィルタチップ**が **フラット等価**で刺さる構造。
  頻度順・折り畳み・段階開示（progressive disclosure）が無く、**全種別が常時・同一重みで露出**。いじるメニューの「ボタン壁」と同じ。
- 差分：いじるは1メニュー内の縦壁、トップは **2枚の壁（作成14／フィルタ13）が別々にフォールドを食う**。
  さらに **同じアイコン群が"作る"と"絞る"で二重露出**（§1補足）＝重複感が情報過多を増幅。

### 4.3 過去 card-and-create doc（2026-07-02）の詰めしろが未回収
- 同 doc 結論＝「**作成側は定石でOK・詰めしろはカード一覧側**」＝①**表示密度の切替（リスト/コンパクト）**が最有力、②タイトル主役・id退避・アクション整理。
- 現状：①リスト密度は実装済だが **既定=カード**（背が高い）＝192件が16,818pxに膨張（§5）。リスト密度を見ると1行/ネタで一覧性が高い（shot 12）＝**既定の見直し余地**。

## 7. 病理の診断（Fable への申し送り）

1. **2枚のフラット壁がフォールドを占有**：作成タイル14 と 種別フィルタ13（＝同じアイコン群の二重露出）が、
   最初のネタに触れる前に大面積で常時露出。**段階開示・頻度順・折り畳みが無い**。
2. **kind 増加が芋づるでトップに恒久割り込み**（design.md L340 が明文化）＝いじるメニューと**同型**。
   実在0〜1件の riff/section_inst/counter が頻用種別と等価に並ぶ＝**露出が実利用に比例していない**。
3. **既定 scope"すべて"がジャンクで実用にならず**、フィルタ/検索が事実上必須＝壁が「毎回要る」構造を強化。
   既定 density=カード（背高）＋仮想化なしで 16,818px＝スクロール一覧性も低い。
4. **効きそうな方向（Fable が判断）**：
   - 作成タイルと種別フィルタの **アイコン二重露出を1系統に統合**（作る=フィルタと兼ねる/コマンド化）。
   - **頻度順＋段階開示**（上位6種を前・薄い種別は「もっと ▾」へ畳む）＝壁を平らに常時出さない。
   - **コンテンツ・ファースト**（ネタ一覧をフォールド上部へ、操作は検索1本＋最小限に退避）＝2026-07-02 doc の「作成はOK・一覧が主役」を今の kind 数で再適用。
   - 既定 density/仮想化・既定 scope（ジャンク回避）の見直しは別軸だが情報過多に直結。
- ピッカー（器）は現状 **非肥大**（器1個）＝今回の主対象ではない。壁は **作成＋フィルタ** に絞ってよい。
