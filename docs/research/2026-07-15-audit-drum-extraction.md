# 独立監査: ドラム抽出レーンの構造診断（蜿蜒 on and on and が落ちる理由）

- 日付: 2026-07-15
- 監査対象: Demucs drum stem → `_audio_poc/analyze.py drum_onsets`（多帯域独立オンセット検出）→ `apps/api/src/audio-drums.ts`（`extractDrumPattern` 窓分割×局所自己相関×正準型照合・`extractSectionPatterns` crash区間分解）→ `apps/api/src/reaper.ts`（信頼度≥0.3で rhythm ネタ化）
- 検体: DB analysis ネタ `e7b9e7ad-b50f-43dc-8842-d32869196c72`（畑亜貴「蜿蜒 on and on and」全自動・最終検収）。facts 実データ＝drum_onsets 2365個（kick 899 / snare 651 / hihat 804 / crash 11）・beat_times 1132拍（中央値 255.0ms＝235.3BPM・±25%外れ 0/1131）・311秒。
- 方法: **コード無変更**。tsx で実モジュール（`extractDrumPattern`/`extractSectionPatterns`/`_internals.matchTemplates`）に DB の実 facts を食わせ、段ごとに計測。診断スクリプトはスクラッチパッド（セッション限り）。
- 正典: design.md #S12改 / research/2026-07-08-drum-pattern-extraction.md（較正 conf: LostMemory 0.41・SURFACE 0.33・DeepSea 0.09）/ research/2026-07-15-allin1-beatthis-feasibility.md

## TL;DR（結論先出し）

**拍（格子の周期）は取れている。死んでいるのは型照合とその手前の格子解像度で、さらに「全体confidence<0.3で区間分解ごとスキップ」のゲート連鎖が、回収可能だった区間ネタ（実測1件 conf0.456）と bass/melody の区間ネタまで道連れにしている。**

- 本番同等呼び出しの実測: `conf=0.086`（<0.3）・template=half・bpm=232.5（beat列235.3と1.2%差＝**自己相関の周期推定は健全**）。
- 死因①（物理限界）: BPM235 の16分格子＝63.8ms に対し、オンセット時刻はSTFTフレーム 23.2ms（hop512/sr22050）粒度＝量子化許容 QUANT_TOL±30%（±19.1ms）が**センサー1フレーム分解能より狭い**。BPM≳194 で16分格子は構造的に破綻する。
- 死因②（型の語彙とブリード）: 実ドラムは高速ロック（キックが8分ギャロップ疾走・スネア変化多）＝正準型4種に居ない。加えて kick/snare 検出の同時率 28〜39%（±35ms）＝帯域ブリードで窓内 kick/snare ヒストの重なり ov 中央値 0.549 → ペナルティ −0.5×ov で raw相関 0.44 → 0.15 に沈み positive 閾 0.2 を割る。
- **beat_times を格子に直接使っても現行の照合は立たない**（実測: beat格子fold で support 0.10〜0.18・必要水準 ~0.5 に遠い）＝「格子を外から渡す」だけでは蜿蜒は直らない。直すのは (i) 区間分解を全体ゲートから独立させる（今すぐ・実測で1区間回収）、(ii) beat_this で拍＋**小節頭**を外部化し、confidence の根拠を「型に見えるか」から「格子に乗っているか」（実測 8分格子集中 0.52）へ再設計、の二段。

## 1. 一次診断: どの段で信頼度が死ぬか（段ごとの実測）

本番同等 `extractDrumPattern(beat_times, drum_onsets)` の debug 実測:

| 段 | 実測 | 判定 |
|---|---|---|
| オンセット検出 (analyze.py) | 2365個・時刻は 23.2ms フレーム粒度。8分格子からのズレ median 12ms / p90 43ms。kick↔snare 同時率 28〜39%。静かな中間部（134〜206s）は密度半減（snare 0.32個/拍） | **黄**: 使えるが、ブリードと取りこぼしあり。時刻精度±1フレーム≒±23ms |
| 局所自己相関（周期） | bpm 232.5 vs beat列 235.3（差1.2%）。窓 132枚生成 | **緑**: 周期は正しい |
| 位相集中ゲート（sub格子） | s4=0.263・s3=0.214 → sharp>0.3 を通る窓 50/132＝**証拠の6割を捨てる** | **赤①**: 16分unit 63.8ms に対し検出ジッタ±23ms。±1フレーム一様誤差だけで理論上限 R≈0.34＝ゲート0.3すれすれ。**BPM235では16分格子がセンサー精度より細かい** |
| 正準型照合 | meter=4: meanSc 0.301・support **0.160**・coherence 0.172 → score 0.043 → conf **0.086** | **赤②**: 窓解剖（felt4小節窓63枚）で raw相関 median 0.44・ov median 0.549 → ペナルティ後 0.154 < positive閾 0.2。positive 4/63 |
| 本番ゲート (reaper.ts:213/260/313/333) | conf 0.086 < 0.3 → meter は chords フォールバック（DB実物: `meter_detected: {confidence:0, source:"chords", template:"half"}`）・**`extractSectionPatterns` 自体が呼ばれない** → rhythmネタ 0件・**bass/melody の区間ネタも 0件**（区間に相乗りする設計のため） | **赤③**: ゲート連鎖の道連れ |

### 補足所見

- **template=half は誤りでなく「正しい別名」**: 体感 117.5 の 4/4 ロック（キック=表・スネア=バックビート）を 235 の速拍4拍小節に畳むと kick@0・snare@2＝ちょうど half 型に折り畳まれる。システムは正しいものを見ていたが、ブリードと語彙不足でスコアが立たなかった。
- **走行毎の型ブレ（rock+/six-d/rock）の機序**: `extractDrumPattern` は純関数＝同一 facts なら決定的（再走で再現確認済み）。ブレの源は解析再実行ごとの perception 側（Demucs/オンセット検出）の微差。meter スコアが近接タイ（m4=0.043 / m6=0.030、区間レベルでも rock+ 0.079 / half 0.12 等どれも弱い）なので、微小なオンセット差で勝者が反転する＝**スコア地形が平ら**なのが本態。
- **TEMPLATES に meter=8 が無い**: 235で追跡された速拍に対し「実小節=速8拍」は現行型ライブラリで表現不能（3/4/6のみ）。半テンポ（体感117.5）解釈を試すには周期を倍にするしかないが、hint は beat_times 中央値から取るため自発的には起きない。
- 区間分解をゲート無視で直接呼ぶと 10 区間が出て、うち **[250.9s–288.9s] が conf 0.456・rock で今日でも回収可能**（本番はここに到達する前に全体ゲートで死んでいる）。
- 静区間（134〜206s 付近）はドラム自体が薄い（ピアノロール目視でほぼ無音）＝conf 0 は**正しい諦め**。全区間が取れないのは曲の実態でもある。

## 2. 「正しい実ビートを外から渡す」効き＝drum_onsets × beat_times 突き合わせ定量

思考実験でなく実データで突き合わせた（拍内位相の円形集中度 R・0..1）:

| 対象 | 拍(sub1) | 8分(sub2) | 16分(sub4) | 3連(sub3) |
|---|---|---|---|---|
| 全オンセット (n=2342) | 0.067 | **0.524** | 0.202 | 0.090 |
| kick | 0.038 | 0.368 | 0.164 | 0.038 |
| snare | 0.077 | **0.721** | 0.327 | 0.131 |
| hihat | 0.155 | 0.606 | 0.263 | 0.129 |

- オンセットは **235 の8分格子（127.5ms unit）には明確に住んでいる**（表拍と裏8分に二峰分布のため sub1 は打ち消し合って低い）。16分格子への集中 0.202 は、ジッタ±23ms が unit 63.8ms に対して大きすぎることの直接の帰結。
- **beat_times を格子として型照合を回した実測**（beat格子fold→実コード `matchTemplates`）: meter4×sub4 で support 0.104、sub2 相当でも 0.181。自前自己相関格子（support 0.160）と同水準以下＝**格子の出所を替えても現行スコアは立たない**。周期推定は元々死因ではないから。
- 半テンポ解釈（beat_times 1/2 間引き→本番コード）: sharp は 0.263→**0.464** に跳ね（格子解像度問題の傍証）、しかし conf 0.033＝型照合が依然死ぬ（ov ペナルティと語彙不足は格子と無関係）。
- **結論（問い2）**: 「beat_times を格子に渡す」**単独では蜿蜒は立たない**。ただし外部拍＋**ダウンビート**（beat_this）があれば、型照合に downbeat 推定をさせる必要がなくなり、confidence の根拠を「正準型に見えるか」から「格子に乗っているか」へ替えられる。その物差しなら蜿蜒は 8分格子 R=0.52 で立つ（較正は要るが、型不問で「鳴っている通り」を区間ごとに量子化して出せる）。

## 3. アーキ格付け（効き×コスト×副作用）

| 案 | 効き（蜿蜒/一般） | コスト | 副作用 | 判定 |
|---|---|---|---|---|
| **(a0) 区間分解を全体confゲートから独立させる**（reaper.ts:260 の条件を外し、区間ごとの conf≥0.3 だけで採否） | 蜿蜒で実測 **1区間回収（conf0.456・rock）**。bass/melody 区間ネタの道連れも解消 | 極小（条件1行の設計判断） | 低信頼曲で区間計算コスト増（数百ms）。誤検出区間は既存の区間conf ゲートが止める | **最優先**。今日の実害の最短止血 |
| (a1) 高BPMの格子適応（BPM≳190 で sub格子を8分へ落とす／perception の hop を 256 に） | sharp 0.263→0.46 相当の救済＝ゲートで捨てていた証拠6割の回収。ただし型照合の死因②は残る | 小 | sub契約（1step=16分）との整合設計が要る | (a0)の次。単独では不十分 |
| (a2) beat_times を格子に直結（問い2の案そのもの） | **実測で効かず**（support 0.10〜0.18）。周期推定は元々健全 | 小 | librosa beat の位相癖（蜿蜒では拍上に乗っていない）を継承 | **棄却**（単独では） |
| **(b) beat_this 導入**（拍＋downbeat 外部化＋confidence 再設計） | 格子と小節頭が確定＝型照合は「命名」に縮退でき、格子集中度ベースのゲートで蜿蜒級（速い・型に居ない曲）も「鳴っている通り」をネタ化できる。F1実測済: 5分曲7〜10s/CPU・BPM±0.5・downbeat Δ0.045s | 中（torch2.6系 venv 分離が宿題・facts 契約に downbeats 追加・interpretation 層のゲート再設計） | 6/8 を 3 拍に割る癖（実測）。オンセット検出の質（ブリード・取りこぼし）は**直らない** | **本命（中期）** |
| (c) ADT学習モデル（ADTOF 等） | 死因②の半分＝**検出ブリード/取りこぼしの根治**はこれのみ。kick/snare 同時率28〜39%・静区間ドロップアウトを置換 | 大（ライセンス/CPU実測が未・モデル運用・facts契約変更） | 型語彙の不足はそれでも残る（照合層の再設計が別途要る） | (b)の後。beat_this で格子を確定させてから載せる方が安い |

**推奨順: (a0) → (a1) → (b) → (c)。** (a2) は棄却。

## 4. 結論

**Root cause 一言: 「拍は取れているのに、①BPM235では16分格子(63.8ms)がオンセット時刻精度(±1フレーム≒23ms)より細かくて滲み、②高速ロックの実ドラムが正準型4種に居ない上に kick/snare 検出ブリード(ov≈0.55)で型照合スコアが沈没(conf 0.086)、③その全体confゲートが区間分解を丸ごとスキップして、回収可能な区間ネタ(実測1件)と bass/melody 区間ネタまで道連れにした」。**

最短で直る道:
1. **今すぐ（1行の設計判断）**: reaper.ts:260 の「全体conf≥0.3 でのみ区間分解」を外し、区間ごとの conf≥0.3 で採否 → 蜿蜒で rhythm ネタ1件が即立つ（実測）。※SDD則: design.md #S12改3 の当該記述を先に直すこと。
2. **次**: 高BPM時の格子適応（sub格子を8分へ・または hop 256）＝捨てている証拠6割の回収。
3. **本命（中期）**: beat_this（宿題の venv 分離）で拍＋downbeat を外部化し、confidence を「型照合スコア」から「格子集中度」ベースへ再設計＝型に居ない曲も「鳴っている通り」を区間ネタ化。
4. ADT学習モデルは beat_this 後に検討（ライセンス/CPU 実測が先）。
