# 歌詞先行メロ生成＝Fable審査団の統合裁定（2026-07-15）

**問い（オーナー）**：「歌詞に対してメロディを作るのを設計して難易度を測って」。
**方法**：Fable 3体並列＝[設計A=既存V2拡張](2026-07-15-lyrics-first-melody-A.md)／[設計B=Orpheus型専用DP](2026-07-15-lyrics-first-melody-B.md)／[監査C=意地悪な実測](2026-07-15-lyrics-first-melody-C-audit.md)→本docで統合Fableが裁定。

## 裁定（TL;DR）
- **難易度の答え：使える版＝S+〜M（Opus 2〜3日級）。フルOrpheus型＝XLで「やらない」。**
- **推奨路線＝減量版（A最小版≒C減量版・同じ絵に独立収束）**：歌詞→モーラ数/句割りを既存V2へ注入（注入口3つは実在確認済み）＋複数seed生成＋`analyzeLyricFit`（実アクセント）で**再ランク＆赤警告**＝アクセントは**hard制約にせず「検査とランキング」に使う**。
- 根拠＝**Cの実測がゲームを決めた**：実メロのアクセント厳密一致31.7%・核下がり目でも36%違反・**オーナー自作曲にA-01赤2発**＝「硬い制約」は自作曲すら生成禁止にする誤前提。完全整合メロ＝節付き朗読（構築的実証）。fitスコアは0.85以上で品質弁別力を失う＝**目的関数でなくフィルタ/ソートに使うのが正しい高度**。

## 推奨実装（M-1スライス・発注時の仕様の芯）
1. **計画純関数**：歌詞→（pyopenjtalk accent.ts 実装済）→行/句のモーラ数列＋句割り→V2 の `opts.phrases`＋`rhythmParts.custom`（音数厳密一致の既存経路）への変換。同モーラ数の行はリズム型を再利用（反復性の回復）。
2. **best-of-N＋再ランク**：`genMelodyCandidates`（既存n=8〜12）→ flowLyric → `analyzeLyricFit`（accents自動注入済み）でスコア軸を追加し**ソート＆赤警告つき候補提示**（機械は候補まで・selectはしない）。
3. **出口**：chat verb（「この歌詞にメロ」＝②の本実装強化）＋候補トレイ。仮歌パイプ（sing）で即試聴＝耳ループが既に開通しているのが最大の追い風。
4. **受け入れ**：機械＝モーラ⇔音数100%・句頭A-01赤の候補内残存を表示（ゼロ強制はしない）・E-rule/既定bit一致維持。耳＝素V2との質差・「歌える譜割りか」。
5. **やらない**（正典化）：Orpheus型フルDP新設（XL・限界価値≈0）／アクセントのhard刈り（実データ違反31.7%）／1番2番の同時整合。

## 残る未知数（正直に）
- 質の天井は「歌詞→リズム配分ヒューリスティック」（A指摘）＝ここは耳較正約30%が残る。
- B遺産の「人間骨格＋言葉乗せ」分業は思想的に美しい＝M-1の結果が物足りなければ、骨格注入（seedMotif/skeleton既存口）で第2スライス候補。
