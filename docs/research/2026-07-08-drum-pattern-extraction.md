# ドラムパターン抽出＝窓分割×正準型照合（#S12改・実測駆動の作り替え）

日付: 2026-07-08 ／ 対象: `_audio_poc/analyze.py drum_onsets`（perception）＋ `apps/api/src/audio-drums.ts extractDrumPattern`（interpretation）
検証データ: LostMemory.mp3（自作・打ち込み4/4）／ DeepSea.mp3（自作・打ち込み6拍子メイン+5拍子スパイス）／ SURFACE「それじゃあバイバイ」（実曲・4/4）。
demucs(htdemucs) 分離済み drums stem から。正解は既知の拍子＋stepグリッド可視化の目視。

## ミッションと結論

**1曲のドラムから「その曲のグルーヴに合った1小節パターン」＋「正しい拍子・小節頭」を出す。変拍子は捏造せず低信頼で諦める。**

結論＝3曲とも意図どおり：

| 曲 | 正解 | 検出 | confidence | 摘要 |
|---|---|---|---|---|
| LostMemory | 4/4 | **4/4** (rock+型) | **0.41** | スネア2・4拍がクリーンに立つ。bpm 87.2 |
| SURFACE | 4/4（実は**シャッフル**） | **4/4 sub=3**（3連） | **0.33** | シャッフル自動検出＝今回の大きな発見 |
| DeepSea | 6+5変拍子 | (4/4) | **0.09** | **正しく低信頼**＝rhythmネタは出さない（gate 0.3） |

## ボトルネックの実測（v1 の3欠陥）

旧実装（librosa onset_detect→帯域比の排他分類＋全曲一発の剛体グリッド＋自己相似で拍子）を実factsで走らせた結果：
LostMemory は meter=4 だが offset ズレ＋パターン崩壊、SURFACE は **meter=3 誤判定**、fold はほぼ空。原因を切り分けると：

1. **排他分類が hihat を潰す**。1オンセット=1ラベルだと kick+hihat の同時発音が強い方に食われ、hihat が全曲最少
   （LostMemory 147個。実際は八分ハット曲＝ありえない）。
2. **全曲一発の剛体グリッドは実録音で破綻**。16分位相の集中度（円形平均の |z|）を測ると
   LostMemory（打ち込み）0.61 に対し **SURFACE 0.03**＝5分の曲では 0.3% のテンポ誤差でも位相が一周して塗り潰される。
   librosa beat_times も挿抜だらけ（線形回帰の残差 max 0.3〜2.1s）で土台にならない。
3. **16分格子の決め打ちはシャッフルで全滅**。SURFACE のオンセット自己相関のピークは 0.178s 間隔＝**拍/3（3連）**。
   拍/4 格子には永遠に乗らない＝「格子が立たない」のは検出ノイズではなく**サブディビジョンの仮定違い**だった。
   また自己相似ベースの拍子推定は、密な8分活動がどの周期でも相関し**変拍子でも高信頼を出す**（捏造側に倒れる）。

## 新 perception（analyze.py drum_onsets・多帯域独立検出）

- **帯域ごとに独立して** onset_strength→peak_pick（kick 25-110Hz／snare 250-1500Hz／hihat 6kHz+）。
  同時刻に複数 kind を許す＝同時発音が消えない。包絡は帯域内95%tileで正規化＝帯域間でしきいが揃う。
- **クロス帯域優勢ゲート**：ピークの正規化強度が「同時刻の最強帯域×係数(k0.5/s0.6/h0.4)」未満なら
  ブリードとして棄却（kick体→snare帯、snareクラック→hihat帯の漏れ対策）。
- 実測（LostMemory）: kick480/snare233/**hihat147** → kick585/snare585/**hihat864**（hihat最多＝妥当な姿に）。
- facts 契約 `[[t_sec, kind, strength]]` は不変＝TS側は差し替え不感。

## 新 interpretation（audio-drums.ts extractDrumPattern）

1. **窓分割**（16拍窓・hop8拍。短尺は8拍窓に自動縮小）。窓ごとに局所自己相関で拍周期
   （放物線補間＋2倍lagで精緻化）→円形平均で位相＝**局所グリッド**。自己相関は平行移動不変＝周期誤差が累積しない。
   実録音のドリフトは窓の中に閉じ込める。
2. **sub 検出**：`sub∈{4=16分, 3=シャッフル3連}` を全窓の位相集中度平均で判定（マージン1.15倍必須＝
   拍上のみのデータは両格子同点なので既定は16分）。SURFACE: s3=0.78 vs s4=0.10 で劇的に分かれる。
3. **正準パターン型ライブラリ照合（本作り替えの芯・オーナー方針）**：
   「スネア=バックビート／キック=頭」というよくある型が**拍子と downbeat を同時に決める**。
   型は拍単位で定義（4/4: rock/four/half/rock+・3/4: waltz・6拍子: 4型）し、窓foldに対して
   zero-mean 正規化相関×全位相で照合。スコアには4つの安全装置：
   - **kick/snare 両骨格必須**（片レーンだけの偶然一致を positives に数えない）
   - **重複ペナルティ**（kickヒストとsnareヒストが似ている＝ブリード/塗り潰しの兆候）
   - **小節出現率**（型の位置が「毎小節」鳴っているか。kick@0/snare@3拍の6拍ループを4/4に畳むと
     rock に見える**折り畳みエイリアス**があり、それは各stepが1/3の小節でしか鳴らない＝ここで弾く）
   - **隣接小節反復度**（この bar 長で畳んで小節が繰り返って見えるか。変拍子を誤った bar 長で
     畳むと隣接小節の中身がズレる）
4. **窓を自分の downbeat へ回転→スコア重み付き集約**＝セクション違い・フィルは票が割れて沈む。
   2パス目で集約パターン（その曲自身の型・ハット込み3レーン）へ再整列し位相を安定させる。
5. **meter 決定と confidence** ＝ 窓スコア平均 × 支持率(0.5飽和) × 窓間一致 × 小節線コヒーレンス
   （連続する正窓の downbeat が半小節格子±20%に乗るペア割合）。係数は実3曲＋合成5ケースの
   グリッドサーチで較正（positiveSc0.2 / presFloor0.15 / repeatFloor0.5 / confScale2.0）。
6. **出力**：`{meter, sub, confidence, bpm, downbeat(秒), template, rhythm}`。
   rhythm は既存契約（**1step=16分**）のまま＝sub=3 はスイング写像（3連{0,1,2}→16分{0,1,3}＝
   付点8分+16分の標準近似）。シャッフルの事実は rhythm ネタのタグ「シャッフル」と
   analysis meta.meter_detected.sub に残す。

## 3曲の最終結果（stepグリッド可視化・[ ]内は強度0-9）

### LostMemory（正解 4/4・打ち込み）→ meter=4 sub=4 conf=0.41 型=rock+ bpm=87.2
```
kick   x... x.x. x.x. ..x.   [9042 4060 8051 4050]
snare  .... x... .... x...   [3121 8022 2032 9023]
hihat  x.x. x.xx x.x. x.xx   [6352 9054 6263 9155]
```
スネア2・4拍が9/9で立つ＝downbeat の芯。キックは1・3拍＋シンコペ（2.5拍/3.5拍）＝忙しめのポップキック。
ハットは8分＋16分ピックアップ。**そのまま叩き直せる下書き**の質。

### SURFACE それじゃあバイバイ（正解 4/4・実曲）→ meter=4 **sub=3(シャッフル)** conf=0.33 型=half bpm=111.4
```
kick   x.. .xx .xx x..   [941 474 048 414]     ※1拍=3列（3連）
snare  ..x ... x.. .x.   [028 302 920 383]
hihat  .xx ... x.. .x.   [349 334 932 492]
```
シャッフル（3連ノリ）を自動検出＝**16分前提だった旧実装では原理的に不可能だった曲**。
解釈はハーフタイム寄り（スネアの主山=3拍目）＋シャッフルの裏（各拍3つ目）に kick/hihat。
16分契約へのスイング写像後: kick[0,5,7,9,11,12] snare[3,8,13] hihat[1,3,8,13]。実曲＋分離残渣なので
LostMemory ほどクリーンではない＝「候補」の質（機械は候補まで・仕上げは人間）。

### DeepSea（正解 6+5変拍子・打ち込み）→ conf=**0.09**＝正しく諦める
```
(参考・最良だった4/4 foldの中身＝ほぼ一様な8分の塗り潰し)
kick   x.x. x.x. x.x. x...   [9040 8040 8040 9040]
snare  x.x. x... x... x...   [5040 9030 6030 9030]
```
どの meter 候補でも支持率が立たず（m4=0.19・m6=0.03）、小節線コヒーレンスも 0.3 前後＝
**gate(0.3) 未満で rhythm ネタは出さない**。「6/8です」と言い張らない＝ミッションどおり。

## 効いたこと／効かなかったこと

**効いた**
- 帯域独立検出＋優勢ゲート（hihat が正しい姿になり、窓の時計としてもハットが効く）
- 局所グリッド（自己相関×窓）＝実録音対応の決め手
- sub検出（シャッフル）＝SURFACE の「格子が立たない」の真因を解消
- 型照合＝スネアのバックビートが downbeat を一意化（LostMemory のスネア 4,12 が9/9）
- 小節出現率＋隣接小節反復度＝「6拍ループが4/4rockに見える」折り畳みエイリアスと変拍子の偶然ロックを排除

**効かなかった／捨てたもの**
- 全曲一発の剛体グリッド（コムフィルタ+最小二乗）＝打ち込みですら1%の周期誤差で崩壊
- librosa beat_times への位相補正だけで済ます案＝挿抜が多すぎる（テンポの当たりにだけ使う）
- レーン別自己相似で拍子＝密な活動がどの周期でも相関し変拍子で高信頼を出す（今回の置換対象）
- 円形平均そのままの小節線コヒーレンス＝推定揺れ（3連1step≈17%）まで罰して実曲を殺す→±20%許容の割合に変更

## 残る限界（正直に）

- **confidence の絶対値は較正済みだが根拠は8ケース**（合成5＋実3）。gate 0.3 は
  LostMemory 0.41 / SURFACE 0.33 / DeepSea 0.09 を分ける位置。曲が増えたら再較正前提（ノブは `DrumTune` で露出済）。
- **SURFACE の細部は目視未確定**（MIDI正解なし）。ハーフタイム解釈 vs 通常バックビートの半小節両義は
  half型が選ばれた結果であり、真値と半小節ズレている可能性は残る（型が対称な曲の宿命）。
- **キックの副次ヒット**（LostMemory の 2.5/3.5拍）は本物のシンコペか分離残渣かを音源レベルでは検証していない。
  閾値（lane最大の45%）に乗った候補は残す方針＝過剰に削るより人が消す方が速い。
- **6拍子の実曲サンプルが未検証**（DeepSea は変拍子すぎて 6 の検証にならない）。6拍子テンプレ4型は
  合成でのみ検証。オーナーの6拍子曲が入ったら要一巡。
- **swing写像は近似**（3連→付点8分+16分）。再生系が3連グリッドを持てば raw sub=3 のまま渡すのが本筋
  （backlog: rhythm content に sub フィールドを足し web 再生を追随させる）。
- DeepSea は「主要部分だけ6/8として低confで出す」ことも技術的には可能だが、小節頭が漂う出力は
  誤誘導のリスクが上回ると判断＝出さない。
