# 監査：「鳴らし方」の決定と再編集はどこに住むべきか（インターフェース設計監査・2026-07-22）

**発端**＝オーナー疑義：「Section で楽器の鳴らし方を決めるのは違反では？ 作る（生成）のはいいけど、そこでしか決められないのは、編集場所も無いし、まともな設計ではない気がする」。
**性格**＝Fable による実装なしの設計監査。実コード・実機スクショで裏取り済み。この doc が次の設計の土台。

**検証仮説**：
- **H1**＝「生成の入口はどこにあってもよいが、『鳴らし方（演奏パターン・奏法）』の決定と再編集の家は、そのネタ自身のエディタに常在すべき」。
- **H2**（オーナー追加）＝「伴奏系ネタの正体は**相対パターン**（ドラムのリズムパターンと同格＝度数×グリッドの『演奏の仕方』）であるべき。パターンが文脈（進行・調）に当たって良きように鳴る。編集の家＝ネタエディタでパターンそのものを編集。生成ノブ（キックに合わせる/2・4で抜く/接近音等）は『パターンを作る道具』だから生成時だけでよく、ネタに残さなくてよい」。

---

## 0. 結論（先出し）

**H1＝正しい。H2＝正しい（共存条件付き）。しかも決定的な発見：H2 のアーキテクチャは既に半分実装されている。**

- H1：オーナーの疑義は「非対称の検知」として正確。奏法シリーズ（style/strumMs/左手/D-U）は正しくネタに常在させたのに、**同じ『鳴らし方』であるパターン型辞書だけが Section の いじる▾ に取り残された**。
- H2：伴奏系4kindのうち **3つ（chord_pattern / section_inst / rhythm）は既に相対パターンが content**。ベースだけが絶対 notes 焼き込みの歴史的例外で、しかも **web には相対ベースの contract・コード解決器・度数グリッドエディタが全部既存**（`RelativeBassContent`＋`resolveRelativeBass`＋`BassStepEditor`＝#bass S2）。**生成（genBass）だけが相対モードに結線されていない**。つまり H2 は新パラダイム導入ではなく「既存の相対モードに生成を繋ぎ、既定に昇格させる」追い工事。
- H1×H2 の補完：H1＝家はネタ／H2＝家に住む中身は相対パターン。H2 が立つと「ベースの生成ノブをネタに残すべきか」という問い（旧 monito 項目3b）は**大部分が消える**——パターン自体が編集対象になるから、ノブは産室の道具で良い。
- ただし H2 は**全面置換ではなく共存**（絶対 notes モードは残す）。反復パターンで表現できない実用ケース（本物のウォーキング・メロディックな動くライン・骨格明示・MIDI取込）が確かにあり、§6 に反証として明記。

違反の分解（H1/H2 込みの最終形）：
1. **入口の非対称**（演奏層）：パターン辞書の入口が Section にしかない。★本丸
2. **ベースの表現形式の例外**（H2）：伴奏系で唯一、演奏の仕方が絶対 notes に焼かれ、パターンとして編集できない。
3. **来歴の喪失**：採用後の content から「どの型で生まれたか」が消える（型IDすら残らない）。

（旧・違反②「ノブ振り直し不能」は H2 採用でベース/ドラムについては解消：パターンが手元にあれば振り直し要求の大半は「パターンを直す」に変わる。残りは候補トレイの「もっと」で足りる。）

## 1. 責務分割の原則（言語化）

パラメータを次の**3層**で分類すると、住所が機械的に決まる。

| 層 | 何を決めるか | 例 | 正準の家 | 根拠 |
|---|---|---|---|---|
| **作曲層**（音を決める） | どの音・どのフレーズか＝一回性の実体 | メロのノブ20本・骨格・リフの音列 | **焼き込み（notes）でよい**。生成の入口は文脈の濃い場所（Section）。来歴は残す | 「候補まで機械・仕上げは人間」＝**手編集が主権**。パラメータを正準にすると手編集が再生成で消える |
| **演奏層**（鳴らし方） | 反復可能な「演奏の仕方」＝文脈に当てて鳴る | **相対パターン**（chord_pattern の hits×voicing・ドラム lanes・ベース度数グリッド）・奏法 style/strumMs/左手/D-U・feel | **ネタに常在**（content＝パターンそのもの・単体エディタで常時再編集） | content が SSOT。レンダ時解決＝非破壊に何度でも変えられる＝再編集の家が必須。**H2＝この層の正体は相対パターンである** |
| **合奏層**（合わせ方） | 他パートとの関係 | kickLock/snareGap/approach（相手＝ドラム）・共通ノリ(section feel)・lanes_muted・配置移調 | **Section**（配置＋文脈の家）。生成時の道具は**非永続で正当**（H2） | 相手は Section にしか居ない。パターンを作る道具の結果は**パターンに刻まれる**＝ノブ値自体の保存は不要 |

**原則文**：
- **ネタ＝中身の家**。伴奏系の中身＝相対パターン（H2）。パターンを見る・決める・直す操作は単体エディタに常在する。
- **Section＝配置と文脈の家**。どこに置くか・進行/調/テンポ・合奏層・**文脈が要る生成の入口**。
- **生成の入口はどこにあってもよい**（オーナー「作るのはいい」）。鉄則＝**content に効果を残す決定は、後からその content の家で再編集できなければならない**（入口が家を兼ねてはならない）。
- 生成ノブの永続要否の判定基準（H2 由来）：**ノブの効果が content（パターン）に読める形で残るなら、ノブ自体は消えてよい**。kickLock の効果＝オンセット格子・snareGap の効果＝dur、どちらもパターンに刻まれる＝再編集はパターンを直せばよい。効果が notes の海に溶けて読めない（メロの20ノブ）場合だけ、来歴を残す価値が出る。

先行エンジン調査（`2026-07-22-accompaniment-style-engines.md`）とも一致：Band-in-a-Box/アレンジャーキーボードは**パターン＝度数データ／変換＝規則**の二層が本体で、伴奏の正体は常に相対パターン。Logic だけが「確定 MIDI 化して手編集」＝作曲層の流儀で、これはメロ/リフに対応する。

## 2. 実コード裏取り（事実）

親の把握はほぼ正確だった。確定事実：

### H1 系（決定場所と残留物）
- **genBass**（`apps/api/src/music/generate.ts:1061-1259`）：opts＝kickLock/snareGap/approach/skeleton/style(33型)/fill/slashBass/swing/humanize。**出力 content＝`{notes}`（＋swing/humanize>0 の時だけ `feel`）＝1258行**。style も kickLock も残らない。→疑い確定。
- **genDrums**（同 `:1482-1488`）：style/fill は生成時のみ。出力＝`rhythm{steps,bars,lanes}` のみ。ただし**フィルは複数小節の lanes に焼かれて content 内に残る**（`applyDrumFill`＝パターン自体が成果物の好例）。
- **genChordPattern**（同 `:932-1010`）：型辞書経路は mode/voicing/steps/hits(vel/dir)/lh を content に焼く＝採用後は単体エディタで全て手編集可。ただし**型ID（`GT-FOLK8` 等）は content に残らない**。
- **単体エディタの現状**（`apps/web/src/components/KindEditorBody.tsx`）：bass＝PianoRoll（絶対）/**BassStepEditor（相対）** のモード切替あり・rhythm＝RhythmEditor（グリッド）。**生成・型辞書の入口はゼロ**。メロには「いじる▾→崩す」＝単体エディタ内生成の先例あり。
- **ChordPatternEditor**＋**MetaPanel 奏法 select**：演奏層（奏法/ストロークの速さ/左手/D-U/トップ/広がり/高さ/arp 3ノブ/パワーコード）は**全てネタに常在・再編集可**。型辞書の入口だけ無い。
- **TinkerSheet**（`.tsx:329-497`）：ドラム/ベース/コード楽器の型・ノブの**唯一の置き場**。値は useState＝unmount で消える（保存されない）。
- **採用フロー**（`useMelodyGen.tsx:520-`）：placeCandidate は**常に createNeta＝新ネタ**。Section の生成が既存ネタを書き換えることは無い＝生成は「産室」。
- **CoW ガード**（`apps/web/src/useCowGuard.ts`）：親から潜った編集は placementCount>=2 で3択。トップから開いた場合はガード無効（意図的設計）。
- **feel の先例**：genBass/genChordPattern/genMelody は swing/humanize を `content.feel` に**パラメータとして保持**しレンダ時適用（`applyFeelEnsemble`）。section 共通ノリも section content.feel。＝演奏層パラメータ保持の成功例が既に2つ。

### H2 系（伴奏系の表現形式）
- **(a) chord_pattern / section_inst＝相対パターン ✅**：content＝`mode/voicing/steps/hits(/lh)`＝度数抽象＋グリッド。実音化は web `resolveChordPattern`（進行・調・テンポ・音色に当てる）。**rhythm＝パターンが content そのもの ✅**（音程レス＝文脈適応が不要なだけで同族）。伴奏系4kindのうち3つは既に H2 の形。
- **(b) ベースだけが例外 ✅**：genBass 出力＝絶対 notes。一方 **bassLibrary の DSL（`apps/api/src/music/bassLibrary.ts`）は表現力が十分**：`BassCell`＝`R/8/b2..#7`（クロマチック度数＝半音アプローチ #1/#4/b7 込み）＋`next`（R>/8>＝次コード先取り）＋`tie/rest/ghost`、16分格子。33型＋フィルは全てこの DSL で書かれ、**生成の最終段 `realizeBassGrid` で実音化して捨てている**（generate.ts:1264-1288）＝相対→絶対の変換を出力側でやっている。
- **(c) web に相対ベースの解決経路が既存 ✅**（#bass S2）：`RelativeBassContent{mode:"relative", steps, pattern:BassStep[], preview_chords?}`（`music.ts:470-480`）＋`resolveRelativeBass`（`music.ts:540-`）＝コードに当てて実音化。**分数コードの低音は R が既定で弾く**・裏拍→ダウンビート跨ぎのアンティシペーション解決あり。`notesForContent`/`compositeNotes` 両方に結線済＝**配置すればセクション進行で鳴る**。エディタ＝`BassStepEditor`＝度数レーン（8/7/5/3/R/→approach）×16分グリッド・長さツール・小節1-4＝**ドラムの RhythmEditor と同じ流儀が既に動いている**。
- **(b') ただし語彙が二重化している**：web `BassDegree`＝`R|3|5|7|8|approach` のみ（クロマチック度数・next・ghost/vel 無し）。api `BassCell` の方が広い。H2 採用は**この2つの相対表現の統一**が本体工事。
- **(b'') レジスタも二重化**：api genBass の窓＝33..48（A1..C3・2026-07-14 実測較正済）／web `band()`＝28..39（E1..D#2・較正前の古い窓）。相対経路を既定に昇格するなら要統一（較正値に寄せる）。
- **(d) 骨格ベースとの優先順位**：genBass 内の適用順＝style型格子 → kick/fig 経路 → approach → snareGap → **骨格明示区間（ピッチ差替え/休符抑制）** → fill（最後・格子正準）。骨格は**絶対ピッチの上書き**として生成時にマージされる。`resolveRelativeBass` に骨格入力は無い＝相対パターン化すると骨格明示は現状経路では効かない（→§5 工事 B'4）。

## 4. あるべき姿と推奨差分（工事順）

### (a) パターン辞書の入口を単体エディタへ ★最優先・小工事

**廃案（折りたたみ帯）は基本妥当。名前と2点だけ改良して復活させるべき。**

- ChordPatternEditor に折りたたみ行「**型から選ぶ ▸**」（「パターンから始める」だと初回専用に見える＝選び直しも同じ入口）。開くと：ジャンル chip（COMP_GENRE_CHIPS 共用）→候補4（cand-card＋▶試聴）→適用＝content 置換（既存 onChange 経路＝**Undo に自然に乗る**・CoW も NetaDialog の既存ガード経由）。CP行契約とは「畳みの中」なので矛盾しない。
- **試聴文脈**：単体では `content.preview_chords`（既存契約）or 既定 C 進行。Section から潜って開いた時に section 進行を渡せると理想（現状渡していない＝独立小工事）。
- **型IDを残す**（違反③）：`ChordPatternContent.patternId?: string`（additive・未指定＝既存全ネタ bit 一致）。適用時に書き、手編集したら消す（or「型：GT-FOLK8（改）」）。選び直し UI の現在型ハイライト・検索・来歴が立つ。
- **同じ帯をドラム単体（RhythmEditor）にも**：ドラムは content が既にパターンなので工事は入口だけ（drumLibrary 型→content 差替え・同 UI 流儀）。
- 規模感：**小（既存部品の再配線・新発明なし）**。

### (b) ベース＝相対パターンへの昇格（H2 本丸）

**推奨＝H2 採用。ただし絶対 notes モードとの共存（opt-in→実績を見て既定昇格）。** 旧論点「ノブをネタに残すか」は H2 で解消：kickLock/snareGap/approach は**パターンを作る道具＝生成時だけ・非永続で正当**（効果はパターンのオンセット/dur に刻まれ、直したければパターンを直す）。gen メタ（ノブ来歴）の保存はベースには**不要**になり、patternId 程度に縮小。

工事分解（B'）：
- **B'1 相対 contract の語彙統一**：web `RelativeBassContent` を bassLibrary DSL の語彙へ拡張＝`degree` にクロマチック度数（`DEGREE_SEMI` の b2..#7）、`next?: boolean`（次コード先取り）、`vel?`（ghost 用）。additive＝既存の相対ネタ（R/3/5/7/8/approach）はそのまま有効。**規模：小-中**（resolver の分岐追加＋型拡張）。
- **B'2 genBass が相対 content を吐く**：型経路は `BassCell`→`BassStep` の**直写像**（realizeBassGrid で実音化せずそのまま出す＝機械的）。fig/kick 経路も内部はもともと度数（root/5度/oct/approach）で組んでいる＝実音化（bassPcToWindow）を出力側からレンダ側へ移すだけ。**opt-in ノブ `relative?: true`**（未指定＝従来絶対 notes＝bit一致）→実機実績後に UI 既定を relative へ。**規模：中**。
- **B'3 BassStepEditor の語彙拡張**：レーンに b3/6/b7 等を足すか「細かい度数」を畳みに（前面は現行6レーン維持＝行契約の流儀）。**規模：小-中・後続でよい**（B'1 の content は表示だけなら現行レーン外を「その他」表示でも凌げる）。
- **B'4 骨格・フィルの扱い**：フィル＝ドラムの先例どおり**複数小節パターンに焼く**（content の steps が配置長ぶんあれば表現可能＝既にドラムがこの形）。骨格明示区間＝絶対ピッチの上書きなので、当面「**skeleton 指定時は従来どおり絶対 notes 出力**」の escape hatch を維持（相対×骨格のレンダ時マージは将来判断）。**規模：留保の明文化のみ**。
- **B'5 レジスタ統一**：web `band()` 28..39 を実測較正の 33..48 へ寄せる（意図的 bit 破壊＝相対モードのみ・要耳確認）。**規模：小**。
- **ベースにも「型から選ぶ ▸」帯**（(a) と同 UI）：相対化した後は型→パターン差替えが単体エディタで完結する。

### (c) Section 側の生成動線＝残す（オーナー「作るのはいい」を追認）

- Section は文脈（進行・テンポ・**相手のドラム**）が最も濃い＝生成の入口として正当。実装も「常に新ネタを作る産室」＝現状維持。
- kickLock/snareGap/approach は合奏層＝Section 住まいが正当で、H2 により**非永続も正当**。TinkerSheet の現 UI はこのまま。
- 将来ガード（再発防止の設計契約）：**「TinkerSheet に足すノブは、(i) 効果が content のパターンに読める形で刻まれる か (ii) 採用時に content キー（patternId 等）に残る もののみ」**をハブ契約コメントに追記。今回の非対称は「入口に家を兼ねさせた」構造が生んだ＝契約にしないと kind が増えるたび再生産される。
- メロ（作曲層）はスコープ外＝現状維持。来歴 `gen?` の付与は「あると嬉しい」止まり＝backlog 行き。

## 5. 境界ケース：共有ネタ×「鳴らし方がネタに住む」

- **原則的に矛盾しない**。鳴らし方（パターン）がネタに住む＝共有ネタの鳴らし方は全配置で共有される。それは SSOT の意味そのものであり、分けたい時は分家＝既存 CoW ガード（`useCowGuard.ts`）がそのまま受ける：親から潜った編集は placementCount>=2 で「全部に効かす/この曲だけ/やめる」＝(a)(b) の適用（content 置換）も NetaDialog 経由なら**既存機構に自動的に乗る**。新機構は不要。
- H2 はむしろ共有と相性が良い：相対パターンは調・進行に依存しないので、**同じベースパターンを A メロとサビ（進行違い）で共有しても各文脈で正しく鳴る**＝絶対 notes 共有より共有の価値が上がる。
- **穴が1つ**：トップ（ネタ帳）から開いた場合はガード無効（parentId 無し・意図的設計）。再編集入口が単体エディタに増えると、この経路で共有ネタの型・パターンを無警告で全曲一斉に変える事故面が広がる。→ 分家3択までは要らないが、**placementCount>=2 の軽い表示（「N箇所で使用中」バッジ）**を推奨（工事順#10・任意）。

## 6. H2 の反証（パターンで表現できないもの・どれだけ痛いか）

オーナーの好みに追従するだけでなく破綻点を探した。結論＝**破綻はしないが、絶対 notes モードを殺すと痛い**。共存（opt-in原則）が条件。

1. **本物のウォーキングベース**：小節ごとに違う経過音で次のコードへ歩く線は、1-4小節の反復パターンでは「同じ歩き方の反復」になり硬い。`approach`/`next` トークンで疑似ウォークは出るが、ジャズ的な一回性の線は作曲層＝絶対 notes（or 配置長ぶんの長い相対列＝反復しない「相対 notes」）の領分。**痛み＝小**（オーナーの主戦場 J-pop/ロック/EDM では反復パターンが正で、CP-WALK 型もこの近似で収録済み）。
2. **リフ的ベース（音列そのものが曲の顔）**：進行が変わった時に度数追従で形が崩れるのが**望ましくない**ケース（音列固定こそ意図）。→ これは riff kind か絶対モードの領分＝kind の使い分けで解決済みの問題。**痛み＝無**（使い分けの案内だけ）。
3. **骨格ベース明示区間**：絶対ピッチの上書き＝相対パターンと直交する概念。escape hatch（skeleton 時は絶対出力）で当面回避＝**痛み＝小・留保明記**（B'4）。
4. **MIDI 取込・過去資産**：外から来るベースは絶対 notes ＝相対化できない（度数の意図の逆推定は別問題）。**共存が必須である決定的理由**。
5. **表現の細部（ghost/vel/スライド）**：現 web 相対契約に vel が無い＝B'1 で additive に足せる。奏法（スライド等）は演奏層の別キーとして将来 chord_pattern の style と同じ流儀で足せる＝**構造的な障害は無い**。
6. **セクション内の小節別変化（フィル）**：ドラムが既に「複数小節パターンに焼く」で解いている（applyDrumFill）＝同じ形で表現可能。**痛み＝無**。

総合：H2 の適用範囲を「伴奏系＝進行に当てて鳴る kind」に限る限り反証は全て回避可能。**メロ・リフ・counter へ広げようとした瞬間に破綻する**（一回性の音列＝作曲層）ので、適用境界を §1 の表のとおり明文化しておくことが再発防止になる。

## 7. 総評（今回の機能群全体の筋）

- **筋の良い所**：二層設計（型＝純データ辞書／実音化＝web 純関数）・content SSOT・「既定OFF＝bit一致」の規律・CoW・候補トレイ（産室モデル）は一貫して堅い。奏法シリーズ A/B/D＋S2/S3 が演奏層をネタに寄せた方向も正しい。feel と相対ベース（#bass S2）という**目指すべき形の実例が既にコードベース内に2つある**のが最大の資産。
- **筋の悪かった所**：生成の入口（TinkerSheet）に設定の家を兼ねさせたまま、S2 型辞書とスライスDのノブを入口側に積んだこと。そして genBass が bassLibrary の度数 DSL（相対）で組んだものを出力直前に実音化して捨て、web の相対モードに繋がなかったこと——**相対の材料は上流に、相対の受け皿は下流に、両方既にあるのに中間で絶対化している**。オーナーの疑義（H1）と好み（H2）は、この2つの実在する構造欠陥の正確な検知だった。
- **修理は安い**：パラダイム転換は不要。(1)型辞書の入口をネタへ (2)genBass の実音化をレンダ側へ移す（既存 resolver に接続） (3)patternId キー1個、の追い工事で H1・H2 とも立つ。全て additive／opt-in＝既存ネタ bit 一致で進められる。
